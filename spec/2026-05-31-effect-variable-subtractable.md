# `S-subtract(α,#a)`と`non-subtract(T, #A)`について
`S-subtract(α,#a) =>`の意味は「`#a`の寿命内で、エフェクト変数`α`からエフェクト集合`S`を引いてよい」である。`non-subtract(T, #A)`は「この型では任意の`#a∈#A`に対して`S-subtract(α,#a)`をキャンセルする」という意味である。

# 型の不等号の重みについて
型の不等号について`T <: U`ではなくもっと一般に`#a`などのidの集合に対して`T #A <: #B U`として型の比較を行う。通常の意味で型の分解が行われ、例えば`T -> S #A <: #B U -> V`であれば`U #B <: #A T`かつ`S #A <: #B V`が成立する。

また、`non-subtract(T, #A) #B <: #C non-subtract(U, #D)`は`T #(A∪B) <: #(C∪D) U`に展開される。さらに`α #A <: #B T`であったとき、`α`の上界に`non-subtract(T, #B)`が登録され、さらに`α`の下界`S`に対して`S #A <: #B T`が追加される。また`T #A <: #B β`であるとき、`β`の下界に`non-subtract(T,#A)`が登録され、さらに`β`の上界`S`に対して`T #A <: #B S`が追加される。

`α #A <: #B β`の場合は`α`の下界`T`と`β`の上界`U`に対して`T #A <: #B U`が追加され、`α`の上界に`non-subtract(β, #B)`、`β`の下界に`non-subtract(α, #A)`が追加される。

実装上は、各型コンストラクタに重み付き境界を保存するわけではない。制約分解器が左右の重み`#A/#B`を引数として持ち回り、通常の極性に従って子制約へ渡す。共変位置では`T #A <: #B U`のまま、反変位置では`U #B <: #A T`に反転し、不変位置ではこの両方を生成する。ADTやレコード、タプル、行型、エフェクトatom引数も、それぞれの宣言された分散に従って同じ規則で分解する。重みが実際に保存されるのは、変数境界または`non-subtract`境界に到達した時だけである。

# `α #A <: #B [S; β]`が制約として要求されたときの正確な処理
1. `α`に対し、`T_n-subtract(α, #a_n)`を全て集める。
2. `U`を`S`と「`#a_n∉#A`であるような`T_n`」の共通部分と置く。
3. `U`が空でなければ`α`の上界に`[U; non-subtract(β,#B)]`を登録する。
4. `U`が空であれば`α`の上界に`non-subtract(β,#B)`を登録する。
5. `(T_n - S)-subtract(β, #a_n)`を`#a_n∉#A`であるようなすべての`n`に対して登録する。
6. **注意**: `α <: [handled2; β]`をそのまま登録しては**いけない**。絶対に止めること。登録の前にこれらを処理しておくこと

# `effect<α> #A <: #B [handled; β]`
- `effect`が`handled`に`effect<γ>`の形で含まれるなら`α+ #A <: #B γ-`かつ`γ+ #B <: #A α-`。そうでなければ`effect<α> #A <: #B β`。

# 新規導入変数について
1. 返りエフェクトが注釈されていない引数の`f`に対して`f x`を推論したときに出てくる`f: α [β] → [γ] δ`について: `γ`は「何も引いてはならない型変数」であるので、新しく`#a`を立て、`Empty-subtract(γ, #a)`を登録する。これの登場する関数の述部には新しく変数を立て`non-subtract([ε] ζ,#{#a})`とする。
2. `my f(x: [handled] α)`など注釈された変数について: `x: [β] α`と登録し、新しく`#b`を立て、`β`には`handled-subtract(β,#b)`を登録する。`handled`が`_`だった場合は`All-subtract(β,#b)`にする。これの登場する関数の述部には新しく変数を立て、`non-subtract([γ] δ, #{#b})`とする。
3. `my f(g: α [β] -> [handled] γ)`と注釈された関数について: `f: α [β] -> [δ] γ`として登録し、新しく`#c`を立て、`δ`には`handled-subtract(δ,#c)`として登録する。`β`に`io`などと書いてあった場合はそのまま登録してよく、`[io] <: β`とする。`α [β] -> [handled; δ] γ`と書いてあった場合は`η`を立て、`handled-subtract(η,#c)`、`f`の述部には新しく変数を立て、`non-subtract([ζ] η, #{#c})`とする。
4. `xs: list(α [A] → [B] β)`のように不変に注釈された変数について、変数`γ`と`δ`を定め、`γ`には`A-subtract(γ;#a)`かつ`A <: γ`、`δ`には`B-subtract(δ;#b)`かつ`B <: δ`を定め、`α [γ] → [δ] β`を登録。不変しか登録できないのでhandled相当か型変数しか受け付けない(混合したものはエラーにする)ように気をつける。
5. 実際に値の定まった型を引いてくる場合: subtractable classを参照。
6. 実際に値は定まっていないが、量化されていない値`f`に対して`f x`を推論したときに出てくる`f: α [β] → [γ] δ`について: なんの情報も付けない。
7. `\x -> y`について: どちらもなんの情報も付けない。

# 型注釈の上下分解
型注釈は、注釈に書かれた型そのものをそのまま上下界へ置くものではない。
注釈`T`は、型の極性に従って「下から抑える型」と「上から抑える型」に分解される。

`x: T`を接続するときは、注釈としてはexactである。
すなわち、`T`から作った下側制約と上側制約の両方を登録する。
ただし`x`が値であり、Simple-subの簡約によって片側だけが観測可能になる場合がある。
この場合でも、注釈自体が片側注釈になったわけではない。

エフェクト行注釈は、通常の型注釈よりも位置の影響を強く受ける。
ユーザが`[io]`と書いたからといって、全ての位置で「ちょうど`io`」を意味するわけではない。
具体的なエフェクト行は、現在の極性に応じて次のように特殊な型へ展開する。

- 共変位置では、書かれた具体エフェクトをそのまま境界に置く。ここではsubtractは作らない。
- 普通の反変位置では、書かれた具体エフェクトを直接境界に固定せず、新しいエフェクト変数を立て、その変数に`S-subtract`を登録する。
- 不変位置では、書けるのは具体的なeffectのみである。effect変数、wildcard、具体effectと変数の混合はここでは受け付けない。

例えば次の注釈を考える。
```yu
value: (a [b] -> [c] d) -> [e] list(f -> [g] h)
```

この注釈は、下からは次の型で抑えられる。
```yu
(a [b] -> [any] d) -> [e] list(f -> [g | 'eff1] h)
where 'eff1: g-subtract#1
```

また、上からは次の型で抑えられる。
```yu
(a [any] -> ['eff2] d) -> [any] list(f -> [g | 'eff1] h)
where 'eff1: g-subtract#1, 'eff2: c-subtract#2
```

ここで`[any]`は「不明だから逃がす」型ではなく、注釈分解によってその位置では具体エフェクトを固定しないことを表す特殊な上限側の行である。
未確定型を表す`Unknown`とは別物である。

型コンストラクタの不変引数など、不変位置で関数型が現れる場合は、上下の両側から同時に抑えるため、具体effectは具体rowとsubtract proxyのsandwichとして現れることがある。
上の例の`list(f -> [g] h)`では、`g`そのものと`g-subtract`を持つ`'eff1`を組み合わせて、`list(f -> [g | 'eff1] h)`として上下の両側に現れる。
ただし、不変位置に書けるeffectは具体effectだけであり、effect変数やwildcardをそのまま書くことはできない。

また、同じ関数型
```yu
a [b] -> [c] d
```
を関数定義の引数注釈として書いた場合は、外側の関数型に含まれる引数型部分として書いたときとは、上下から見る側が逆に見える。
下からは次のように抑えられる。
```yu
a [any] -> ['eff] d
where 'eff: c-subtract#1
```

上からは次のように抑えられる。
```yu
a [b] -> [any] d
```

このため、エフェクト注釈について「反変位置ではsubtractを作らない」「共変位置ではsubtractを作る」といった単純な読み替えは誤りである。
普通の反変位置ではsubtractを作り、共変位置ではsubtractを作らない。
関数型の内部では、関数引数・引数effectが反変、返りeffect・返り値が共変であるため、外側の極性と合成してからこの規則を適用する。

# subtractable class
量化の際に、全ての`S-subtract(α,#a)`は別テーブルに保存される。ただし、共変位置で`non-subtract(γ, #a)`でない`γ`が存在しない時`T-subtract(γ,#a)`は消去して良く、また全ての`S-subtract(_,#a)`が存在しない時、`non-subtract(_, #a)`もまた消去してよい。

instantiateのときは型変数と同じように`#a`もfreshにする。また、**`non-subtract`が存在しない時**、`non-subtract(_,#{#a})`を呼んだ関数の出力型と出力エフェクト型に付与する。

# subtractableの合成
共起分析で別の型と統合する場合や、既存の型にsubtract情報を付けるときは次に注意する:
1. 同じ `#a` の `S-subtract` 同士は共通部分を取る
2. 異なる `#a` は寿命が違うので宣言としては混ぜない。

`S-subtract` の共通部分は次の代数で計算する。
- `Empty ∩ X = Empty`
- `All ∩ X = X`
- `Set(A) ∩ Set(B) = Set(A ∩ B)`
- `Set(A) ∩ AllExcept(B) = Set(A - B)`
- `AllExcept(A) ∩ AllExcept(B) = AllExcept(A ∪ B)`

ここで `A` や `B` はeffect atom familyの集合である。`ref_update _` のような引数付きeffectは、少なくともsubtractabilityの合成では同じeffect familyとして扱う。より細かい引数制約はordinary type constraint側で扱い、subtractabilityの共通部分そのものではfamilyを分割しない。

以後、`T_n-subtract(α, #a_n)`(n=1,...,N)のことを単に`(T_1∩…∩T_n)-subtract(α)`、または`α`は`(T_1∩…∩T_n)`-subtractableであると呼ぶ。

# `catch`のscrutinee/`k`/catch全体effectの制約
1. scrutineeの型を`[α] β`と仮定する(これは新しい型を作るわけではなく、ただのassume)。
2. まず、catchの中で引かれているエフェクトを`handled`とし、`α <: [handled; γ]`を要請する。このとき新しく立てた`γ`のsubtractableはNoneである(要請時に登録されるため)
3. `k`のeffectは`α`とする。
4. 新しく型`δ`を立てる。次に、各枝のエフェクト型`ε_1,...,ε_n`に対して`ε_1 <: δ, ... ε_n <: δ`を要請する。
5. catchの中で引かれているエフェクトが完全であれば`γ <: δ`を、1つでも欠けているものがあれば`α <: δ`を要請する。

## 注意
完全なエフェクトハンドルと不完全なエフェクトハンドルを混ぜたときに、完全なエフェクトハンドルまでαとして漏れ出すのは**仕様**である。これは、エフェクトを分離してハンドルしたときに意味が変わってしまうため。言語上の限界でもある。

# shallowとrecursive/deepの期待型が分岐する理由
1. Shallowについて: All-subtractable scrutinee`[α] β`を引数としてとり、またエフェクトをそのまま起動するだけのshallow handlerがbodyである関数を考える。このとき、全体の型は`β [α] → [δ] β`である。ただし各枝はエフェクトを起動するので`α <: δ`である。また、`catch`の特性により`α <: [handled; γ]`かつ`γ <: δ`である。ここでcompact型に全て展開すると、`β [α&[handled;γ]] → [γ|α|δ] β`が成立する。ここで極性を判定すると`δ`が消え、共起分析では共変位置で常に共起する`α`と`γ`が統合で結ばれる。このとき、全体の型は`β [α&[handled;α]] → [α] β`と推論される。これがshallow handlerの推論手順である。
2. recursive/deepについて: ある関数`f`がall subtractable scrutinee`[α] β`を引数としてとり、継続の値をそのまま自分自身に投射する枝しか持たないrecursive handlerをbodyとして持つとする。このとき、全体の型は同じく`β [α] → [δ] β`である。ただし、`k`の出力を自分自身に代入したことにより`α <: α`のみが成立し、`δ`に`α`がかかることはない。よって展開は`β [α&[handled;γ]] → [γ|δ] β`である。このとき、極性を判定すると、`α`が反変位置にしかないので消え去り、同様に共変位置の`δ`も消え去る。よって`β [handled;γ] → [γ] β`が最終的に推論される。
3. 足りない枝がある場合: ある関数`f`がall subtractable scrutinee`[α] β`を引数としてとり、継続の値をそのまま自分自身に投射する枝しか持たないrecursive handlerをbodyとして持つとする。ただし、足りない枝があるとする。このとき、`catch`の特性により`α <: [handled; γ]`かつ`α <: δ`かつ`γ <: δ`である。よって`β [α&[handled;γ]] → [α|γ|δ] β`が展開され、shallow handlerと同じ状態になる。

# `outer/local/repeat` の衛生性で何を残して何を消すべきか
```yu
act outer:
    our redo: () -> never
    my act local:
        our break: () -> never
        our judge(x: [_] _) = catch x:
            break(), _ -> true
            _ -> false
        our sub(x: [_] _) = catch x:
            break(), _ -> ()
            _ -> ()
        my act repeat = local   // 内部のみ同じ別エフェクト
        our run(f: () -> [outer] _) = local::sub: loop true with:
            our loop b = if b:
                loop:repeat::judge:catch f():
                    redo(), _ -> repeat::break()
                    _ -> local::break()

pub r = outer::run
```
を仮定する。
1. 通常の推論手順により、`repeat::judge: ⊤ [repeat; ε] → [ε] bool`、`local::sub: ⊤ [local; ζ] → [ζ] ()`となる(また、`ε`と`ζ`はall subtractableである)ことに注意する。また、1回しか用いないため、単相化した状態の型がこれであると簡単のため仮定する。
2. まず`run`の中の`f`の型は始めに`() -> [α] β`と推論され、`α`のsubtractableは`outer`に確定する。
3. 次に`f()`の起動が`catch`され、`α <: [outer; γ]`が要請される。この場合は素直に`α <: [outer; γ]`が登録され、`catch`全体の型`δ`には`repeat <: δ`, `local <: δ`, `γ <: δ`が登録されることになる。`γ`は`outer`-subtractableである。
4. 次に`repeat, local, γ <: δ <: [repeat; ε]`が登録される。このとき、具体的な型については`repeat <: [repeat; ε] ⇒ [] <: ε`、`local <: [repeat; ε] ⇒ local <: ε`、`γ <: [repeat; ε] ⇒ γ <: ε`が成立する。これを合わせると、`local, γ <: ε`であり、`ε`は`outer`-subtractableである。
5. loopの型は`bool → [ε] ()`であり(これは`loop`が純粋関数であることによる)、展開すると`bool → [local, γ, ε]`である。これらの型はレベルが全て低くなっているので、`loop`の型は簡約されず、`bool → [local, γ, ε] ()`のままである。
6. これにtrueが代入され、単一化はやはり1回限りの展開であるし簡単のために省略するが、`local::sub`によってあたらしく`local, γ, ε <: [local; ζ]`が推論され、仮定は同じなので省くが、`γ, ε <: ζ`が結論され、さらに`ζ`は`outer`-subtractableである。
7. 最終的な型は`(() -> [α] β) → [ζ] ()`であり、これを展開すると、`(() -> [α&[outer; γ&ε&ζ]] β) → [ζ|γ|ε] ()`である。`α`と`β`は反変位置にしかなく、また共起分析で`γ=ε=ζ`なので、最終的に`(() -> [outer; γ] ⊤) → [γ] ()`が推論される。

# 補足: `m`/`j`は巡回を生まない
```yu
act io:
    our read: () -> int
my j(x: [_] _) = catch x:
    io::read(), k -> j(k 0)
my m(x: [io; e] _) = catch x:
    io::read(), k -> j(k 0)
```
を仮定する。
1. `j`は通常の解析によって`α [io; β] → [β] α with β: All-subtractable`であることに留意し、1度しか用いないため単一化もされているものとする。
2. 次に`m`の内部の推論を行う。`x: [γ] δ with γ: io-subtractable`である。次に`catch`の制約によって`γ <: [io; ε]`であり、`ε: io-subtractable`になる。枝の中では`k(0): [γ] δ`が`j`に適用されるため、`δ <: α`かつ`γ <: [io; β]`である。`γ`はio-subtractableであるのでこのまま登録され、`β`がio-subtractableに矯正される。ここで枝の型は`β`であることに注意。
3. `catch`全体の型は`β,ε <: ζ`となる。全体の型を推論しよう。
4. `δ [γ] → [ζ] α`
5. `δ & α [γ & [io; ε & ζ] & [io; β & ζ]] → [ζ | β | ε] α | δ`
5. `δ & α [γ & [io; ε & ζ & β]] → [ζ | β | ε] α | δ`
6. `α [io; β] → [β] α`
7. このように巡回が発生する余地はない。

# (All except `ref_update _`)-subtractableと変数表現
データ型を定義する際には、反変表現として保存する。これにより次の様なデータが作成できる:
```yu
struct ref 'e 'a {
    get: () -> ['e] 'a
    update_effect: () -> [ref_update 'a; 'e] ()
}
```
このとき、
```yu
ref: {
    get: () -> ['e] 'a
    update_effect: () -> [ref_update 'a; 'e] ()
} -> ref 'e 'a
```
が定義されるが、また同じように`update_effect: ref 'e 'a → () -> [ref_update 'a, 'e] ()`も定義される必要がある。このとき、`'e`は何か？　(All except `ref_update _`)-subtractableであるとするのが妥当であろう。この後、`catch`で`[ref_update 'a; 'e]`を取り去るのであるから当然であり、分離とはそのように成される。

## 実際に推論されるもの
```yu
pub act ref_update 'a:
    pub update: 'a -> 'a

pub type ref 'e 'a with:
    struct self:
        get: () -> ['e] 'a
        update_effect: () -> [ref_update 'a; 'e] ()
    pub r.update f =
        my loop(x: [_] _) = catch x:
            ref_update::update v, k -> loop:k:f v
        loop:r.update_effect()
```
に対して`ref::update`を推論してみよう。最初は引数の型から。便宜上型変数名を数字、レベルも数字で置こう。`r: ref '1(1) '2(1)`と`f: '3(1)`だ。

次にloopを見よう。
- `x: ['4(2)] '5(2)`だね。ついでに`'4(2): All-subtract#1`だ。
- `catch`があるから、`'4(2) <: [ref_update '6(2); '7(2)]`だ。これは案外簡単に解けて、`'4(2) <: [ref_update '6(2); '7(2)]`かつ`'7(2): ref_update-except-subtract#1`だった。
- ここからちょっと複雑。`v`の型を`'8(2)`、`k`の型を`'9(2)`と置く。次は`'6(2) <: '8(2)`かつ`'6(2) -> ['4(2)] '5(2) <: '9(2)`だ。ルールの3番だね。
- 次。`f v`が出てきたので、`'3(1) <: '8(2) -> ['10(2)] '11(2) where '10(2): Empty-subtract`だ。比較が発生したからextrude。`'8,'10,'11`のlevelは`1`になる。`'3(1) <: '8(1) -> '['10(1)] 11(1)`だ。
- `f v: ['10(1)] '11(1)`に注意して次を見る。
  - `k:f v`の型を`['12(2)] '13(2)`と置いて、`'6(2) -> ['4(2)] '5(2) <: '9(2) <: '11(1) ['10(1)] -> ['12(2)] '13(2)`だね。
  - これは`'11(1) <: '6(2)`と`'4(2) | '10(1) <: '12(2)`かつ`'5(2) <: '13(2)`。これは純粋関数が効果をスルーするからだね。
- `loop`の型を`'14(2)`、catchの型を`['15(2)#1] '16(2)#1`と置く。そうするとまず下から`'5(2) ['4(2)] -> ['15(2)#1] '16(2)#1 <: '14(2)`が言える。
- そして同時に`k:f v`の型を突っ込むから`'14(2) <: '13(2) ['12(2)] -> ['17(2)] '18(2)`が言える。

# 実際の表現について
compact collect型については、通常の共変型が`変数 + record + ...`であるように、こちらも共変部分は`変数 + 行`であることが望ましい。また、反変部分は`A = 変数 + [効果1, 効果2; A] + [効果3; A] + ...`であるのが望ましい。

表示型についてもほとんど同様にしてよい。
ただし `S-subtract(ρ, #a)` と `non-subtract(ρ, #a)` はordinary row boundではなく、role-like constraintとして別に表示する。
通常のhandlerで使い切られるものはcleanupで消えるので、既存の表示はほとんど変わらない。

# 変数展開について
1. `α [β] → [γ] δ`という型をcompact表現に展開するとする。ただし、`β`の上界は`ε, [undet; ζ]`、`ε`の上界は`[io; η]`、`ζ`の上界は`[flip; γ]`とする。`α`や`γ`、`δ`は今回上下界を持たないとする。
2. `α [β & ε & [undet; ζ]] → [γ] δ`と展開する。
3. 次に見るのは`ε`だが、このとき`β`から直接出てきた上界なので、さらに上は展開しない。代わりに次の`ζ`を見る。
4. `α [β & ε & [undet; ζ & [flip; γ]]] → [γ] δ`と展開する。
5. 極性消去によって`α [[undet; [flip; γ]]] → [γ] δ`と展開される。終わり

ここで注意すべきは**変数を展開して出てきた変数は、上界を展開しない**ことである。これによって、本来推移律の成立しない世界を上手くコンパクト化している。

`S-subtract(ρ, #a)` はこのordinary compact展開に混ぜない。
compact表示ではordinary row boundを従来どおり展開し、残ったsubtraction/non-subtractionだけをrole-like constraintとして表示する。
