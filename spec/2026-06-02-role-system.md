roleには通常の引数と関連型が存在している。例えば`Mul<α,β,out=γ>`の場合、`α`, `β`が通常の引数であり、`γ`が関連型である。宣言時は次の様に書かれる:
```yu
pub role Mul 'a 'b:
    type out
    pub a.mul(b: 'b): out
```
全ての引数は不変である。また、`mul`はメソッドとして公開され、`mul: Mul<α,β,out=γ> => 'a -> 'b -> γ`と型がつく。

# 注意
不変な変数とは共変部分と反変部分の2つ組のことである。`(int|α, α&float)`などと書くとタプルと被ってしまうため、若干読みにくい記法ではあるが「共有された変数(ここでは`α`)を1つ選んで`int|α&float`と書く」ことに決まっている。
主型に現れる方向を判定するときも、不変型引数はこの2つ組として読む。例えば`a|b|c&d&e`は`(a|b|c, c&d&e)`であり、`c`は両方向に現れる。
極性解析と共起分析でも同じ読みを使う。引数位置の境界は中心つきの入力区間として扱い、`α|β&γ` は `β&γ -> α|β` と同じように読む。

ただし区間には関数読みに無い含意がある。不変区間`(lo, up)`はどのinstantiationでも「lo ≤ 実際の引数型 ≤ up」を要求するので、loとupの**両方**に現れる型変数は実際の引数型と等しいことがその区間単独で確定する。したがって共起分析は、他の出現位置で同伴していなくても、両側に現れる変数同士(constructorの不変引数のように中心変数があるならその中心と)をこの区間だけを根拠に併合してよい。スキームのinstance集合は変わらない。
この規則で併合するのは変数同士に限る。変数と具体型の等値は変数の除去を意味するが、区間1箇所の根拠で除去すると具体型が同伴しない他の出現位置から変数だけが消えて型が狭まるため、除去は通常の共起分析(全出現での同伴)に委ねる。
具体型については別の形を使う。両側の具体成分がどちらも同一の単一具体型`T`である区間は、実際の引数型が`T`に釘付けされるので、区間そのものを`T`に畳んでよい。これは位置の書き換えであって変数の除去ではない。同伴していた変数の他の出現はそのまま残し、後続の極性消去・共起分析に任せる。この畳み込みはbottom-upの一回パスで行い、不動点反復には入れない。

簡約とレベル保護の関係も区間規則だけ特別になる。途中の簡約は世代化境界より浅い変数(他のdefと共有されている足場。implのrequirement接続などが作る)を併合の対象にしない。それらの変数はこのスキーム内の出現が全てではないため、「全出現での同伴」を根拠にする極性消去や通常の共起併合は正当化できないからである。しかし区間由来の等式はdemand自体が機械側で強制する等値であり、共有変数に対しても成立する。そこでdefの世代化が確定する直前に一度だけ、レベル保護を外した区間等式専用の併合パスを掛ける(代表は浅い側を選ぶ)。これも一回パスで、不動点反復には入れない。tupleやopt等のパラメトリックimplのmember本体が、requirement足場の変数を署名に残してしまう問題はこれで消える。

同じ確定直前の段階で、schemeに量化できない床変数のうちsurfaceに残す根拠を持たないものも落としてよい。片側極性だけに現れるなら通常の極性消去と同じ根拠で消える。両極性に現れていても、同じ具体型に挟まれているだけならsandwich flatteningと同じ根拠で消える。これは床変数を新しく併合する規則ではなく、出力schemeへ自由変数として漏れるだけの冗長成分を取り除く規則である。裸の床変数まで消すと未確定のrootを`never`相当に潰すため、片側極性の除去は、深いlevelで生まれて境界までextrudeされた変数か、具体型と同伴している変数に限る。

# roleとlevelについて
1. roleの引数部分に変数が入ったとき、それらの全てのレベルは共有される。例えば第1引数がextrudeされる場合、第2引数も一緒にextrudeされ、レベルが引き上がる(=数字が小さくなる)。変数のレベルはその変数が量化される位置と密接に関係しており、それらの変数と一緒に量化される必要があるため。
2. 所属するレベルの変数内で明示的な計算(関数リテラル外で行われた直接の関数適用)が行われた時、role内のレベルは強制的に1つ上がる(=数字が一つ小さくなる)。1つ上がった変数でも明示的な計算が行われた時もレベルが1つ上がる。最終的にリテラルなどの計算の無い場所で量化される。例えば`my a = 1 + 1`と書かれた場合、`Add` roleは`a`内ではなく親によって(またはグローバルに)解決される。
3. roleから完全に型変数が消え失せたときにはその場で解決されるかエラーになるため、レベルは必要ない。

# 簡約について
1. role内も通常の式と同じように全ての型変数が展開される。
2. ただし、roleの全ての通常引数部分に共通の型変数を持っている場合、例えば`Mul<α&β,γ,out=δ>`と`Mul<β&ε,γ,out=ζ>`が要求されていた場合(1番目が`β`、2番目が`γ`を共有している)、これらは等号で結ばれ、`Mul<β|α&β&ε,γ,out=ζ|δ&ζ>`のようになる(分かりづらいが、共変部分と反変部分がそれぞれ合わさるというだけ)
3. 極性解析と共起分析ではroleの引数も対象になる。roleの引数は不変であるため、中心の型(共有された変数)が消えることはない。
4. 簡約後、roleが決定する場合がある。後述。

impl候補も、照合時には宣言headから作られたrole制約として同じcompact簡約を受ける。
これは候補を新しく推論し直す規則ではなく、demand側とcandidate側を同じ等価変形済みの表現で比較するための正規化である。
candidate簡約で許されるのは、Simple-subの極性消去・共起分析・sandwich、注意節の区間規則(両側出現変数の併合と釘付け区間の畳み込み)、およびこの節のrole引数の併合だけである。
impl本体から集めたprerequisiteは、candidate本体の照合で得た代入を適用してから通常のrole制約として再び解決する。

# roleの引数と決定について
1. 簡約直後、全ての**通常の**引数に対し、どちらかの境界部分に、たった1つだけ具体型(型変数以外の型)があり、**主型に現れる型変数が同じ方向のみの**場合にのみ、roleが解決される。例えば`Mul<α,β,out=γ>`の場合には`Mul<int|α,int|β&float,out=γ> => () -> α|β`などが相当し、`Mul<int|str|α, ...>`の場合には解決しない。両方の境界部分にそれぞれ1つだけ型がある場合には、不変引数の共変位置(左側)が優先される(関数への入力であることが多いため)
2. 曖昧である場合はエラーになるわけではなく、そのまま残される。ただし、簡約直後にroleが浮いている場合、すなわち「共通で使っている型変数があるroleを線で結び、さらに主部と同じ型変数があるroleを主部と線で結んだとき、主部からたどれないroleが存在する」場合は解決手段がなくなるためエラーになる。
3. トップレベルが簡約されてもトップレベル計算(`say "hello"`みたいな直に書いた式)にbindされたRoleが残っている場合もエラーになる。

# roleの再帰的な解決について
型制約を集めきらないとroleは解決しないが、roleを解決したことで初めて得られる型制約があるため、roleの制約は次の様に解決される:
1. 関連型の存在するroleの型をcompactとして全て展開する
2. 解けるroleを解く
3. それによって新しく型制約が生まれる場合がある(関連型の決定)ので、roleの型をまた全てcompactとして新しく展開する(このとき、展開したものを再展開するのではなく、現状のroleを新たに展開すること)
4. 解けるroleを解く。関連型による変化があったときは3に戻る

これを繰り返す。また、methodが未だ解決していない変数にも最終手段(型クラス、レコード解釈)の前にこれを行わねばならない(解決する場合がある)。

解決済みのdemandは出力スキームのwhere節に残さない。ただし解決ループは簡約済みビューの表記でdemandを記録するため、最終出力の生の述語と表記が一致しないことがある。そこで残置述語を選ぶときは、ループ末尾と同じ「主型＋全role」を一括簡約したビューでもう一度解決判定を行い、適用済みのimpl候補へ一意に解決するdemand(と適用済みdemandに表記一致するもの)を除外する。この再判定をrole単独の簡約で行ってはならない。単独簡約は共起が減るぶん強く効きすぎて、ループでは適用されなかった解決をここで初めて作り、制約を注入しないまま述語だけを消す恐れがある。
