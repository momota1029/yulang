roleには通常の引数と関連型が存在している。例えば`Mul<α,β,out=γ>`の場合、`α`, `β`が通常の引数であり、`γ`が関連型である。宣言時は次の様に書かれる:
```yu
pub role Mul 'a 'b:
    type out
    pub a.mul(b: 'b): out
```
lowering直後のdemandとimpl head照合では、roleの引数は不変区間として扱う。また、`mul`はメソッドとして公開され、`mul: Mul<α,β,out=γ> => 'a -> 'b -> γ`と型がつく。

role宣言そのものは、通常引数ごとにvarianceを導出する。
導出に使うのは、role内に宣言されたmethod signatureへreceiverを第1通常引数として補った型である。
関数引数では極性を反転し、関数戻り値では同じ極性で読む。
型コンストラクタの引数、effect familyの引数、effect row tailに通常引数が現れる場合は、そのデータ構造自体のvarianceを推論せず、保守的に不変出現として記録する。
これにより、データ構造は従来どおり不変のまま、`role Ord 'a: our x.le: 'a -> bool` のような「method signature上ではただの関数引数としてしか使われないrole入力」だけを、role側のkind情報として扱える。

導出結果は `Unused | Covariant | Contravariant | Invariant` のいずれかで保持する。
複数methodで同じ入力が同じ向きにだけ現れるならその向き、両向きに現れるか不変位置に現れるなら `Invariant`、どこにも現れないなら `Unused` である。
このvarianceは宣言signature内での現れ方なので、residualを`where`束縛としてsurfaceへ浮かせるときは向きを反転して読む。
例えば `Ord` の `'a` は宣言signature内では反変にだけ現れるため、`where 'a: Ord` 側では共変の要求として扱える。

role解決ループでimpl候補とdemandを照合する段階では、通常引数は従来どおり不変区間のまま扱う。
一方、解決されずschemeの`where`節へ残るresidual roleを簡約するときは、反転後のvarianceでrole引数を読む。
`Covariant` なwhere引数は区間のlower境界だけを共変位置として読み、`Contravariant` なwhere引数はupper境界だけを反変位置として読む。
`Invariant` と `Unused` はこの段階では不変区間として読む。
これにより、`list α -> list α where Ord α` のように「実際に要求される型」だけをsurfaceへ残し、supertype比較のために途中で生じた足場変数は共起分析・極性消去で落とせる。

# 注意
不変な変数とは共変部分と反変部分の2つ組のことである。`(int|α, α&float)`などと書くとタプルと被ってしまうため、表示では両側に共通して現れる型変数(ここでは`α`)を中心として`int|α&float`と書く。
中心は表示上の略記であり、内部表現として別に保存しない。
つまり表示上の `int|α&float` は、内部では単に lower=`int|α`、upper=`α&float` の区間である。
表示時だけ、lower/upper に共通する変数 `α` の重複を隠す。
主型に現れる方向を判定するときも、不変型引数はこの2つ組として読む。例えば`a|b|c&d&e`は`(a|b|c, c&d&e)`であり、`c`は両方向に現れる。
極性解析と共起分析でも同じ読みを使う。引数位置の境界は中心を持たない入力区間として扱い、`α|β&γ` は `β&γ -> α|β` と同じように読む。

ただし区間には関数読みに無い含意がある。不変区間`(lo, up)`はどのinstantiationでも「lo ≤ 実際の引数型 ≤ up」を要求するので、loとupの**両方**に現れる型変数は実際の引数型と等しいことがその区間単独で確定する。したがって共起分析は、他の出現位置で同伴していなくても、両側に現れる変数同士をこの区間だけを根拠に併合してよい。スキームのinstance集合は変わらない。
この規則で併合するのは変数同士に限る。変数と具体型の等値は変数の除去を意味するが、区間1箇所の根拠で除去すると具体型が同伴しない他の出現位置から変数だけが消えて型が狭まるため、除去は通常の共起分析(全出現での同伴)に委ねる。
具体型については別の形を使う。両側の具体成分がどちらも同一の単一具体型`T`である区間は、実際の引数型が`T`に釘付けされるので、区間そのものを`T`に畳んでよい。これは位置の書き換えであって変数の除去ではない。同伴していた変数の他の出現はそのまま残し、後続の極性消去・共起分析に任せる。この畳み込みはbottom-upの一回パスで行い、不動点反復には入れない。

2つの不変区間を同じ実引数としてすり合わせる場合は、両区間の下界を lower 側へ、両区間の上界を upper 側へ単に足す。
中心変数は保存しないため、この段階でどちらの変数を代表にするかは選ばない。
必要な等式は、後続の区間等式併合で lower/upper の両側に現れる変数から導かれる。
ただし、区間を足すだけでは矛盾を検出しきれない。
`A = (loA, upA)` と `B = (loB, upB)` を同じ実引数としてすり合わせるときは、同時に `loA <: upB` と `loB <: upA` の制約を生成する。
この制約は「演算のおまけ」ではなく、同じ実引数が両方の区間を満たすための交差条件である。

区間内の足場変数にも、共起分析へ入る前に戻せる制約がある。
ある区間の片側で、代表になりうる変数`α`以外の同側変数`β, γ, ...`が、その区間自身を除いた場所では反対極性に現れない場合、`β, γ, ...`は実際に代入される型としてroleを解決する根拠を持たない。
このとき区間の実型を`α`と同一視し、反対側の境界からそれらの足場変数を除いた型へ片方向制約を生成してよい。
例えば lower=`α|β|γ`、upper=`γ&δ&ε`で、`β`と`γ`がこの区間以外の反対極性に現れないなら、`α <: δ&ε`を制約として追加する。
制約追加後は通常どおりcompactを再collectし、共起分析・区間等式・sandwichに渡す。
これは表示上の中心選択ではなく、存在しない(代入されない)型でroleが解決されることはない、という区間の実型条件をConstraintMachineへ戻す処理である。

簡約とレベル保護の関係も区間規則だけ特別になる。途中の簡約は世代化境界より浅い変数(他のdefと共有されている足場。implのrequirement接続などが作る)を併合の対象にしない。それらの変数はこのスキーム内の出現が全てではないため、「全出現での同伴」を根拠にする極性消去や通常の共起併合は正当化できないからである。しかし区間由来の等式はdemand自体が機械側で強制する等値であり、共有変数に対しても成立する。そこでdefの世代化が確定する直前に一度だけ、レベル保護を外した区間等式専用の併合パスを掛ける(代表は浅い側を選ぶ)。これも一回パスで、不動点反復には入れない。tupleやopt等のパラメトリックimplのmember本体が、requirement足場の変数を署名に残してしまう問題はこれで消える。

同じ確定直前の段階で、schemeに量化できない床変数のうちsurfaceに残す根拠を持たないものも落としてよい。片側極性だけに現れるなら通常の極性消去と同じ根拠で消える。両極性に現れていても、同じ具体型や同じ変数に挟まれているだけならSimple-sub 4.3.1のvariable sandwich flatteningと同じ根拠で消える。変数に挟まれた床変数は、挟んでいる変数へ併合する。これはmachineの共有変数を破壊する規則ではなく、出力schemeへ自由変数として漏れるだけの冗長成分を取り除く規則である。裸の床変数まで消すと未確定のrootを`never`相当に潰すため、片側極性の除去は、深いlevelで生まれて境界までextrudeされた変数か、具体型と同伴している変数に限る。

roleの通常引数に2つだけ型変数が残り、その一方が通常引数内で出る方向と反対側の主型極性に現れない場合も、同じvariable sandwichの読みでその変数を消し、もう一方の変数をlower/upper両側へ広げてよい。例えば主型の反変位置にだけ現れるreceiver足場変数が`Size<α|β>`のβとして残るなら、βは主型の共変位置に現れないため、αへ併合して`Size<α>`としてよい。これはconcrete implを選ぶための特別規則ではなく、role解決のmain-polarity分析を変数同士の境界にも適用した簡約である。

# roleとlevelについて
1. roleの引数部分に変数が入ったとき、それらの全てのレベルは共有される。例えば第1引数がextrudeされる場合、第2引数も一緒にextrudeされ、レベルが引き上がる(=数字が小さくなる)。変数のレベルはその変数が量化される位置と密接に関係しており、それらの変数と一緒に量化される必要があるため。
2. 所属するレベルの変数内で明示的な計算(関数リテラル外で行われた直接の関数適用)が行われた時、role内のレベルは強制的に1つ上がる(=数字が一つ小さくなる)。1つ上がった変数でも明示的な計算が行われた時もレベルが1つ上がる。最終的にリテラルなどの計算の無い場所で量化される。例えば`my a = 1 + 1`と書かれた場合、`Add` roleは`a`内ではなく親によって(またはグローバルに)解決される。
3. roleから完全に型変数が消え失せたときにはその場で解決されるかエラーになるため、レベルは必要ない。

# 簡約について
1. role内も通常の式と同じように全ての型変数が展開される。
2. ただし、roleの全ての通常引数部分に共通の型変数を持っている場合、例えば`Mul<α&β,γ,out=δ>`と`Mul<β&ε,γ,out=ζ>`が要求されていた場合(1番目が`β`、2番目が`γ`を共有している)、これらは等号で結ばれ、`Mul<β|α&β&ε,γ,out=ζ|δ&ζ>`のようになる(分かりづらいが、共変部分と反変部分がそれぞれ合わさるというだけ)
3. 極性解析と共起分析ではroleの引数も対象になる。roleの引数は不変であるため、lower/upper の両方に現れる変数は正負両方の出現を持つ。
   ここでいう対象には、主型には現れずrole述語の中にだけ現れる変数も含まれる。role引数の不変区間の両側に同じ変数が残っている限り、その変数は正負両方に現れるものとして数えるため、極性消去の候補にはならない。
4. 簡約後、roleが決定する場合がある。後述。

lowering直後の生のrole制約は、構文由来の下界・上界だけを保持する。
実装上は`RoleConstraintArg { lower, upper }`の形になる。
schemeとしてfreezeされたboundsを再びrole demandへ戻す場合も、`Neu::Bounds(lower, upper)`から同じ形へ戻す。
中心をscheme側へ保存すると、instantiate後に余計な出現が混ざった時点でその変数が実引数として固定され、supertypeで解決できるprincipalなinstanceを失うためである。
しかしcompact表現に入った後のrole引数は、通常引数も関連型引数も`CompactBounds`そのものである。
つまりrole引数専用の「lower/upperを持つ別区間」は作らず、constructor引数・effect family引数・tuple payloadと同じ不変区間表現へ畳む。
両側が同一の具体構造へ畳める場合はlift済みの`CompactBounds::{Con,Tuple,Fun,...}`として持つ。
そうでなければ中心なしの`CompactBounds::Interval`として持つ。
finalize後の`Neu::Bounds(lower, upper)`も中心を持たない。表示時にだけ、簡約後のlower/upperから共通変数を選んで`lo|α&up`の形へ整える。

同じrole制約が共通変数を通じて同一componentへ併合されるときも、同じ場所に来たrole引数同士は`CompactBounds`のmerge規則で併合する。
したがって、同じ通常引数位置や同じ関連型名の値に複数のboundsが来た場合は、constructor引数やeffect family引数と同じく`loA <: upB`と`loB <: upA`の交差条件を生成し、制約が増えた場合は再collectを要求する。
これはrole解決の見かけの正規化ではなく、同じ実引数を要求するrole demand同士が同時に成立するための制約である。

impl候補も、照合時には宣言headから作られたrole制約として同じcompact簡約を受ける。
これは候補を新しく推論し直す規則ではなく、demand側とcandidate側を同じ等価変形済みの表現で比較するための正規化である。
candidate簡約で許されるのは、Simple-subの極性消去・共起分析・sandwich、注意節の区間規則(両側出現変数の併合と釘付け区間の畳み込み)、およびこの節のrole引数の併合だけである。
impl本体から集めたprerequisiteは、candidate本体の照合で得た代入を適用してから通常のrole制約として再び解決する。
impl本体に`type value = T`のような関連型等式がある場合、その等式はcandidateの関連型引数として、impl headと同じannotation変数scopeで読む。
candidateの通常引数と関連型引数は、impl member bodyの制約変数を直接共有しない。member body側のrequirement接続はrole宣言由来の関連型変数を使い、bodyの局所制約がcandidate headを汚さないようにする。
関連型等式が明示されていない場合は、従来どおりmember bodyのrequirement接続から関連型変数がclampされる。

# roleの引数と決定について
1. 簡約直後、全ての**通常の**引数に対し、どちらかの境界部分に、たった1つだけ具体型(型変数以外の型)があり、**主型に現れる型変数が同じ方向のみの**場合にのみ、roleが解決される。例えば`Mul<α,β,out=γ>`の場合には`Mul<int|α,int|β&float,out=γ> => () -> α|β`などが相当し、`Mul<int|str|α, ...>`の場合には解決しない。両方の境界部分にそれぞれ1つだけ型がある場合には、不変引数の共変位置(左側)が優先される(関数への入力であることが多いため)
   この「主型に現れる型変数」の判定は、roleを含めた簡約が済んだ後の主型側だけを見る。role述語の中にだけ現れる区間両側共通変数は、簡約では保持されるが、この解決可否判定では主型に現れない変数として扱う。
2. 曖昧である場合はエラーになるわけではなく、そのまま残される。ただし、簡約直後にroleが浮いている場合、すなわち「共通で使っている型変数があるroleを線で結び、さらに主部と同じ型変数があるroleを主部と線で結んだとき、主部からたどれないroleが存在する」場合は解決手段がなくなるためエラーになる。
3. トップレベルが簡約されてもトップレベル計算(`say "hello"`みたいな直に書いた式)にbindされたRoleが残っている場合もエラーになる。

# roleの再帰的な解決について
型制約を集めきらないとroleは解決しないが、roleを解決したことで初めて得られる型制約があるため、roleの制約は次の様に解決される:
1. 関連型の存在するroleの型をcompactとして全て展開する
2. 解けるroleを解く
3. それによって新しく型制約が生まれる場合がある(関連型の決定)ので、roleの型をまた全てcompactとして新しく展開する(このとき、展開したものを再展開するのではなく、現状のroleを新たに展開すること)
4. 解けるroleを解く。関連型による変化があったときは3に戻る

これを繰り返す。また、methodが未だ解決していない変数にも最終手段(型クラス、レコード解釈)の前にこれを行わねばならない(解決する場合がある)。

解決済みのdemandは出力スキームのwhere節に残さない。ただし解決ループは簡約済みビューの表記でdemandを記録するため、最終出力の生の述語と表記が一致しないことがある。そこで残置述語を選ぶときは、ループ末尾と同じ「主型＋全role」を一括簡約したビューでもう一度解決判定を行い、適用済みのimpl候補へ一意に解決するdemand(と適用済みdemandに表記一致するもの)を除外する。この再判定をrole単独の簡約で行ってはならない。単独簡約は共起が減るぶん強く効きすぎて、ループでは適用されなかった解決をここで初めて作り、制約を注入しないまま述語だけを消す恐れがある。
