うん、ここは **「10ms前後で止まってるEvidence VMを、危ない近道なしで次の構造へ進める計画」** に落とすのがよさそうだねぇ。

# Yulang Evidence VM 高速化計画

## 目標

### 短期目標

`bench/nondet_20_discard.yu` の release runtime を、warm cache 条件で **安定して一桁ms** に寄せる。

現在の最新代表値は `9.3〜10.3ms` くらいで、もう一桁には片足が入ってる。`showcase` は `20.5〜22.1ms` くらい。

### 中期目標

Evidence VM を、今の

```text
Then / MarkerFrame / ProviderEnv / CatchBody が混ざった continuation tree
```

から、

```text
eval segment + scope/evidence delta
```

へ少しずつ移す。

これは「scopeを雑にflattenする」話ではなく、**意味論上必要な境界は保ったまま、毎回continuation treeを歩く形を減らす**方向だねぇ。既存の設計メモでも、次の本筋は `CatchBody-aware ResumePlan / ScopeDelta rewrite` とされている。

---

# 方針

## やらないこと

まず、危ない高速化は封印する。

* ProviderEnv miss を invisibility 扱いしない
* `any ProviderEnv` を permission evidence として使わない
* `Then` / `MarkerFrame` / `ProviderEnv` を scope-preserving invariant なしでflattenしない
* native close branch を shadow stats と専用certなしで広げない
* path名・bench名・test名に依存する特殊分岐を入れない
* generic request fallback を弱めない

この制約は既存メモの制約と一致している。

## 優先順位

| 優先 | 作業                                  | 目的                       |
| -: | ----------------------------------- | ------------------------ |
|  1 | ResumePlan shadow builder           | 構造変更の当たり判定を取る            |
|  2 | CatchBody-aware narrow execution    | 安全な一部経路だけ実行を置き換える        |
|  3 | MarkerDelta / ProviderDelta sidecar | marker/provider再入コストを減らす |
|  4 | stats-off fast API                  | 効果が見えた後に軽く検討             |
|  5 | bytecode / compiled block           | ResumePlanがIRとして固まった後    |

Control VM trick の移植は優先しない。今は Control VM を「移植元」ではなく、**意味論oracleと設計比較用**として使うのがよさそうだねぇ。既存メモでも Control VM trick migration は最下位扱い。

# Phase 1: ResumePlan shadow builder

## 目的

まだ実行経路は変えない。
まず、今の continuation から **ResumePlanに切り出せる形がどれくらいあるか** を測る。

## 追加する概念

```rust
struct ResumePlanShadow {
    eval_segment_len: usize,
    has_request_boundary: bool,
    has_handler_boundary: bool,
    marker_delta_kind: MarkerDeltaKind,
    provider_delta_kind: ProviderDeltaKind,
    catch_body_kind: CatchBodyKind,
    rejected_reason: Option<ResumePlanRejectReason>,
}
```

この段階では実行に使わない。
`--runtime-evidence-profile-deep` または debug profiling 時だけ構築する。

## counters

```text
resume_plan_shadow_candidates
resume_plan_shadow_created
resume_plan_shadow_reject_request_boundary
resume_plan_shadow_reject_handler_boundary
resume_plan_shadow_reject_catch_body
resume_plan_shadow_reject_marker_delta
resume_plan_shadow_reject_provider_delta
resume_plan_shadow_eval_frames
resume_plan_shadow_then_frames
resume_plan_shadow_marker_frames
resume_plan_shadow_provider_frames
resume_plan_shadow_catch_body_frames
```

## 見たいもの

`nondet` で特に見る値:

* candidates が `direct_tail_segment_materialized_continuations` 付近まで出るか
* reject理由が request boundary 一色なのか
* CatchBodyを扱うと created が増えるか
* marker/provider delta が sidecar化できそうな密度か

過去の DirectTail segment shadow では、普通のeval segmentだけ切ろうとして `created: 0` になっている。全件 request boundary を含んでいたのが原因なので、今回は **CatchBody / request boundary aware** にする必要がある。

## rollback条件

* profileなし通常releaseで runtime が悪化する
* 通常releaseで不要な continuation traversal が走る
* created が代表 workload でほぼ0
* reject理由が広すぎて、次のnarrow execution sliceが見えない

# Phase 2: CatchBody-aware narrow execution

## 目的

ResumePlan shadow のうち、**一番安全で狭い形だけ** 実行に使う。

最初から MarkerDelta / ProviderDelta まで実行しない。
まずは eval segment と CatchBody boundary の扱いだけを見る。

## 最初の対象

候補はこの形:

```text
DirectTail signal
  + CatchBody-known
  + request boundary shape known
  + generic fallback materialization available
  + marker/provider delta は legacy path に戻せる
```

つまり、失敗時には必ず既存の continuation tree に戻れる形にする。

## 実装スライス

1. `ResumePlanId` を作る
2. `ResumePlan` は runtime内部arenaに保存
3. shadow builderが accepted shape のときだけ plan を作る
4. direct-tail signal pass-through で plan execution を試す
5. 少しでも shape 違反があれば legacy continuation materialize
6. `compare-control` で全件一致を見る

## counters

```text
resume_plan_exec_attempts
resume_plan_exec_hits
resume_plan_exec_fallbacks
resume_plan_exec_reject_shape
resume_plan_exec_reject_boundary
resume_plan_exec_materialized_legacy
resume_plan_exec_eval_steps_saved_estimate
continuation_resume_steps
continuation_resume_then_steps
continuation_appends
catch_body_checks
```

## 期待値

設計メモ上の期待では、narrow ResumePlan execution は coverage があれば **5〜15%** くらいが狙い目。

`nondet` が今 `9.3〜10.3ms` なので、5%でも一桁安定に効く可能性がある。

## rollback条件

* `compare.control` mismatch
* shadow mismatch
* adversarial corpus failure
* `nondet` 中央値が2%以上悪化
* `showcase` 中央値が3%以上悪化
* `resume_plan_exec_hits / attempts` が低すぎる
* fallback materialization が増えすぎて `continuation_appends` が悪化

# Phase 3: MarkerDelta / ProviderDelta sidecar

## 目的

本丸。
`MarkerFrame` / `ProviderEnv` を ordinary continuation spine から横に出す。

今の hot counter では、`marker_frame_entries`、`function_call_marker_plans`、`continuation_resume_*` がまだ太い。
ここを触らないと、10ms前後から大きく抜けるのは難しそうだねぇ。

## 追加する概念

```rust
struct MarkerDelta {
    enter_ops: SmallVec<[MarkerEnterOp; 4]>,
    exit_ops: SmallVec<[MarkerExitOp; 4]>,
}

struct ProviderDelta {
    push_ops: SmallVec<[ProviderEnterOp; 2]>,
    pop_ops: SmallVec<[ProviderExitOp; 2]>,
}
```

## 注意点

Markerは hygiene に直結するので、単独で雑にskipしない。
過去に duplicate marker scope skip は `marker_frame_entries` を `13526 -> 7293` まで減らしたのに、実時間は悪化している。cover判定とclose形状が高くついた。

だからやるなら、

```text
skipする
```

ではなく、

```text
ResumePlan内の事前計算済みenter/exit opとして実行する
```

にする。

## counters

```text
marker_delta_created
marker_delta_enter_ops
marker_delta_active_add_id_ops
marker_delta_handler_boundary_ops
marker_delta_legacy_push_pop_avoided

provider_delta_created
provider_delta_push_ops
provider_delta_pop_ops
provider_delta_legacy_provider_frame_avoided

marker_frame_entries
active_add_id_scans
function_call_marker_plans
continuation_resume_marker_steps
continuation_resume_provider_steps
```

## 期待値

ResumePlan + MarkerDelta / ProviderDelta execution は、設計メモ上では **15〜35%** もあり得る領域。
もちろんかなりリスクが高いので、Phase 2で plan execution の安全面を固めてからだねぇ。

## rollback条件

* hygiene系adversarial failure
* handler boundary mismatch
* provider visibility mismatch
* nearest ProviderEnv miss / later grant の区別が曖昧になる
* `marker_frame_entries` は減るが runtime が悪化する
* active AddId のpush/pop削減が counters に出ない

# Phase 4: stats-off fast API

## 目的

構造変更のあと、最後に軽く削る。

ただし優先度は低い。
過去に main runtime stats counter を外す案は速度差がほぼなく、diagnostics が薄くなるので戻している。

## やるなら条件付き

やるなら、

```rust
RuntimeStatsMode::Full
RuntimeStatsMode::Core
RuntimeStatsMode::Off
```

みたいに分けるより、最初はもっと小さく、

```rust
RuntimeEvidenceRunContext {
    stats: Option<&mut RuntimeEvidenceRunStats>
}
```

または

```rust
StatsSink
```

で局所化する。

## rollback条件

* branch が散らばって runtime.rs が読みづらくなる
* 速度差が1〜2%未満
* diagnostics が落ちる
* compilerが最適化できず逆に遅くなる

---

# Phase 5: bytecode / compiled block準備

## 目的

Evidence VMをいきなりnative backendへ飛ばさない。
まず ResumePlan を **将来のbytecode block IR** として使える形へ寄せる。

## 方向

```text
ResumePlan
  -> EvalSegment
  -> explicit scope opcodes
  -> direct-style pure/eval islands
  -> bytecode blocks
  -> later Cranelift/native
```

既存メモでも、bytecode/nativeへ直接飛ぶより、ResumePlan IRを経由する優先順位になっている。

# 判断基準

## 続行

次のどれかを満たすなら続行。

* `resume_plan_shadow_created / candidates` が十分高い
* rejectの大半が1〜2種類に集中している
* CatchBody-aware にすると DirectTail segment の `created: 0` が改善する
* marker/provider delta が太く、sidecar化で削れる見込みがある
* normal release hot path にprofile traversalが乗っていない

## いったん停止

次なら停止。

* CatchBody-awareでも accepted shape が少ない
* request boundary の扱いが複雑すぎて fallback だらけ
* legacy materialization が増えて append churn が悪化
* shadowは良いが execution slice が作れない
* hygiene / provider visibility の検証コストが高すぎる

---

# まとめ

一番自然な計画はこれだねぇ。

```text
1. baseline固定
2. ResumePlan shadowでcoverageを見る
3. CatchBody-aware narrow executionを入れる
4. MarkerDelta / ProviderDeltaをsidecar化する
5. stats-offは後回し
6. bytecode/nativeはResumePlanが固まってから
```

今のYulangは、もう「雑に速くする」段階を抜けてる。
次は **continuation tree runtime から、scope-preserving ResumePlan runtime へ移るかどうか** を測る段階だと思うよ〜。
