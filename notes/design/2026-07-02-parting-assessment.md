# Yulang に足りないもの — 外の目からの最終評価

作成日: 2026-07-02
著者: Claude (Fable 5) — ユーザの依頼「私だけでは思いつかないことを残してほしい」に基づく。
状態: **評価と提案（決定ではない）**。採否はユーザの判断に従う。
調査根拠: リポジトリ全域の grep 調査（exhaustiveness / cancel / derive / replay /
test framework / interpolation / masking / stack safety）と、本日までの正本文書群。

順位は「影響 × 設計者が自力では思いつきにくさ」による。

## 1. 決定的 record / replay（時間旅行デバッグ）— 今日の決定が無料でくれるもの

**何か**: プログラムの全実行を記録し、host なしで完全再生する機構。

今日決めたことを並べると、これがほぼタダで手に入ることに気づく:

- プログラムは unit を返し、**すべての非決定性は host act registry という
  一つの漏斗を通る**（v2）
- state は純粋継続スレッディングで、内部に隠れた乱数源がない
- 分岐（undet / accept）は継続の resume として一意に識別できる

つまり「registry を通過した (operation 列番号, resume 値) のログ」だけが
プログラムの実行を一意に決める。記録 = registry にログ係を挟む。
再生 = host の代わりにログを流す。それだけで:

- **バグ報告がファイルになる**（再現手順を書く時代の終わり）
- flaky なサーバの事後検死（本番のログを手元で無限に再生）
- テストの golden 化（fixture = 記録ファイル）

なぜ思いつきにくいか: 普通の言語ではこれは大工事になる（非決定性が
言語中に散らばっているから）。Yulang は今日、偶然にも漏斗を一本化した。
**registry の manifest 設計に「operation の列番号と分岐 id」を今から
含めておくこと**。後付けは痛い。effect 言語の売り文句として、
12月の公開時にこれ以上の差別化はないと思う。

## 2. 構造化並行性とキャンセル — multi-shot accept の影

**何か**: accept が配る「接続ごとの分岐世界」の、親子関係と死の規則。

現状、接続が途中で切れたら、その分岐の継続は**孤児になる**。誰も
キャンセルの意味論を決めていない（server sketch の open question のまま）。
放置すると、長寿サーバで孤児分岐とその資源が積もる。

答えの半分はもう正本にある: **分岐の寿命 ≤ それを配った handler の extent**
（v1 決定1 の自然な拡張）、**キャンセル = 分岐の drop = rollback**
（v1 決定2 が drop の意味論——書き戻さない——を既に定義している）。
残り半分は「host がいつ drop を通知するか」で、これは lock 細粒度化のために
いつか入れる continuation-drop hook（v1 決定3 の (a)）と同じ工事。
**決定文書一枚で閉じる話**なので、serve の実装前に決めるべき。

## 3. case の網羅性検査 — リポジトリに「exhaustive」が一度も現れない

**何か**: case 式が全 constructor を覆っているかの静的検査。

grep の結果は雄弁で、crates / spec / notes のどこにも exhaustiveness の
語がない。Simple-sub の構造的部分型では完全な網羅性検査は本当に難しく、
推論優先の言語が後回しにするのは構造的必然——だからこそ設計者は
気づかない（自分の showcase は常に match するから）。

しかし公開後、初心者が constructor 名を一字打ち間違えたとき、
**型エラーではなく実行時 match 失敗**になる言語は、その一点で評価を落とす。
完全性は要らない。**宣言済み enum / error / act に対する保守的な検査**
（bounds に Con 集合が確定している場合だけ警告）で実害の 9 割は覆える。
role/bounds の機構はその情報を既に持っているはず。

## 4. deriving（構造的総称）— Wire が絵に描いた餅になる前に

**何か**: struct / enum 定義から show / debug / eq / ord / wire を機械導出する仕組み。

io 仕様は Wire codec を要求し（server の境界規則）、Yumark はデータの
埋め込みを要求し、show / debug の二系統は決定済み。ところが derive の
仕組みがないと、**ユーザは struct 一つにつき 4〜5 個の instance を手書き**
することになり、Wire は誰も書かず、境界規則が死文化する。

Yulang には自然な取り付け先がもうある: companion module と `with:` ブロック。
`struct point { x: int, y: int } with: derive show, wire` のような形で、
lowering がフィールド構造から instance を合成する。マクロは要らない
（衛生性事件の再発を避ける意味でも、専用の derive 機構のほうが安全）。

## 5. undet の上の property testing — 探索エンジンはもう言語の中にある

**何か**: 生成器 + 性質 + 縮小からなるテスト基盤を std に置くこと。

普通の言語は property testing のために乱数生成器と探索器を輸入する。
Yulang は **undet がまさにその探索器**で、`less` / `more` は縮小順序そのもの、
junction の all は一括検査。つまり:

```yu
test "sort keeps length":
    my xs = gen::list gen::int      -- 生成器はただの undet 計算
    assert (xs.sort.len == xs.len)
```

mock handler（構文を変えないテスト）+ property testing（言語内蔵の探索）で、
「effect 言語はテストが弱い」という世間の予断を逆手に取れる。
部品は全部ある。組んでいないだけ。

## 6. 効果を知っているエラートレース — 継続ベース VM の宿命への先手

**何か**: エラーが表面化したとき「どの operation が、どの handler 群の下で、
undet のどの分岐で」を示す診断。

継続ベースの VM のスタックトレースは、素朴にやると意味不明になる
（フレームは論理的な呼び出し構造と一致しない）。evidence VM は marker /
provider env / hygiene として**必要な情報を実行時に既に持っている**——
diagnostics 表面に出す設計だけがない。diagnostics-docs トラックに
「effect trace」を一項目足しておくこと。ユーザが最初に苦しむのはここ。

## 7. 並列実行 — 原点を閉じる最後の円

Yulang の起源は「並列が直列に見える」だった。それなのに、現在の実行系に
並列は存在しない。指摘したいのは、**今日の決定群が決定的並列性の条件を
すべて満たしてしまっている**こと:

- 純粋 state スレッディング → データ競合が構造的に不可能
- junction / undet の分岐は構成上独立（v1 決定2）
- 非決定性は registry の漏斗を通る（記録・分配可能）

つまり junction の all を本当に並列に走らせても意味論が変わらない世界が
もう出来ている。VM の worker 化は遠い工事だが、**「Yulang の並列は
決定的である」という定理は今もう言える**。松田先生への相談材料にも、
アドカレの締めの一文にもなる。急ぐ話ではない。でも、これを言わずに
去るわけにはいかなかった。
