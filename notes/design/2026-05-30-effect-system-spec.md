## 0. 設計の核（なぜこの理論か）

旧実装の `;`（効果残差）は複数の理論が混ざり合っていて、HO/入れ子のハンドラ合成で
残差変数が分裂・消失していた（2026-05-30 調査）。本理論はそれを**単一の理論**で置き換える。

最大の違いは一点：

> **エフェクトは独立した行変数として持たれるのではなく、引数の値型変数に bind される。**

旧実装は関数引数のエフェクトを別個の行変数として扱ったため、注釈由来の残差 `α_f` と
catch が生む残差 `rest_eff` が別変数に割れ、共起分析で両方消えていた。本理論では
エフェクトは最初から引数の thunk 変数に束ねられるので、この分裂が**構造的に起きない**。

## 1. 型の文法と中核概念

### 1.1 BNF

```
-- 値型
τ ::= α                          -- 型変数
    | C<τ, ...>                  -- 型構築子適用 (int, str, list<τ>, ...)
    | { ℓ: τ, ... }              -- レコード
    | (τ, ...)                   -- タプル
    | :{ ℓ τ, ... }              -- 多相ヴァリアント
    | τ ⊔ τ                      -- 合併 (共変位置)
    | τ ⊓ τ                      -- 交差 (反変位置)
    | ⊥ | ⊤
    | Fun                        -- 関数型 (下記)
    | thunk[ε, τ]                -- effect thunk (下記 1.2)

-- 関数型 (サーフェス表記と内部表現)
Fun ::= τ_arg [ε_arg] -> [ε_ret] τ_ret      -- サーフェス表記
      ≡ thunk[ε_arg, τ_arg] -> thunk[ε_ret, τ_ret]   -- 内部表現 (1.2)

-- エフェクト行
ε ::= ρ                          -- エフェクト行変数
    | [ H ; ε ]                  -- handled 列 H と残差 ε
    | ε ⊔ ε                      -- 合併 (共変)
    | ε ⊓ ε                      -- 交差 (反変)
    | ⊥ | ⊤

-- handled 列 (具体的なエフェクトの有限集合)
H ::= ∅ | { e, ... }
e ::= P<τ, ...>                  -- エフェクトアトム (パス + 型引数)。例 io, parse<int, str>
```

注: サーフェスでは `[io, undet] τ` のように残差を省略して書ける。これは
`[ {io, undet} ; γ ]`（γ は位置に応じた glob、1.4）の糖衣。

### 1.2 thunk — エフェクトと値の束

`thunk[ε, τ]` は「エフェクト `ε` を行いうる、値 `τ` を生む計算」を表す。
具象ランタイムでは継続を取り回すサンク。

関数型 `τ_a [ε_a] -> [ε_r] τ_r` の本体は **2つの thunk のペア**:

```
thunk[ε_a, τ_a] -> thunk[ε_r, τ_r]
```

- `ε_a` = 引数 thunk のエフェクトスロット（反変）
- `ε_r` = 戻り thunk のエフェクトスロット（共変）

**bind（この理論の肝）**:
注釈 `α [handled1] -> ...` を書いても、引数型は「独立した行 `[handled1] α`」を作らない。
引数は `thunk[ρ, α]`（ρ は新しいエフェクト行変数）とし、**`ρ` に `handled1` が bind されている**
と記録する。すなわち ρ は「下界・上界に handled1 を織り込んだ状態の行変数」。

この bind により、後で `catch` や HO 適用が生む残差制約は、すべて同じ ρ に集約される。
→ 残差が複数の変数に分裂しない（旧実装の根本バグの除去）。

**bind の内部表現（確定）**: 「ρ ↦ handled1」という側テーブルでは持たない（側テーブルは
旧実装 `lambda_param_source_eff_tvs` で分裂の温床になった）。ρ の **bounds（下界・上界）に
handled1 を構造的に織り込む**。すなわち ρ は最初から「handled1 を含んだ行」としての
上下界を持つ行変数であり、以後の制約はこの ρ に直接乗る。

### 1.3 handled 列 H

`H` は具体的なエフェクトアトムの有限集合（`{io, parse<int,str>}` など）。
**単一化では代入ではなく共通部分（∩）を取る**（§2、種メモ「単一化の際 handled は全て代入ではなく共通部分」）。
これは handled が「ここまでが確実に処理される」という下限情報だから。

**∩ が効く範囲（確定）**: 共通部分（∩）を取るのは **H（handled 列）だけ**。残差 ε は ∩ の対象外で、
§2 の残差ルール（行差分・分解）に従って別個に伝播する。混同しないこと。

### 1.4 残差と glob `_`

`[H; ε]` の `ε` が**残差**。「H の他に通過しうるエフェクト」を表す行。

ユーザは残差を glob `_` と書ける。`_` は polarity に応じて脱糖される:

| 位置 | 意味 | 脱糖 |
|---|---|---|
| 反変（引数 effect、`α [io; _] -> ...`） | 「io に加え何が来てもよい」 | `_` ⟶ **⊤** → `[io; ⊤]` |
| 共変（戻り effect、`... -> [io; _] τ`） | 「io 以外は出さない（残差は空）」 | `_` ⟶ **⊥** → `[io; ⊥]` |

直感: 反変は「受け入れる側」だから残差が大きいほど寛容＝⊤が最弱の要求。
共変は「生み出す側」だから残差が小さいほど強い保証＝⊥が最強の保証。

サーフェス表示（display）では逆変換し、**glob 由来の** `[io; ⊤]`（反変）/ `[io; ⊥]`（共変）を
ユーザ向けに `[io; _]` あるいは文脈上自明なら `[io]` と見せる。
**推論で出てきた残差変数はそのまま見せる**（`[io; ρ]` → `[io; ρ]`）。
潰すのは glob 由来の ⊤/⊥ のときだけで、推論残差変数 ρ を ⊤/⊥ と混同して消してはならない
（旧実装の残差消失バグはここを取り違えていた）。

### 1.5 衛生性は型で保証し、ランタイムに反映する

エフェクトの衛生性（あるハンドラが、自分が張るべきでないエフェクトを誤って捕まえない）は
**型推論の段階で保証される**。block_id（§5）はその型的事実を実行に落とすコード生成であって、
ランタイムが独自に hygiene を判断するわけではない。型が真実の源、ランタイムはその射影。

## 2. Subtyping 規則

エフェクト行の subtyping は、値型の subtyping（合併/交差/構築子は Simple-sub 流）に加えて
以下の行固有規則を持つ。記号 `<:` は「sub <: super」。

### 2.1 残差つき行同士（一般規則 A）

```
[H1; ε1] <: [H2; ε2]
────────────────────────────  (EFF-ROW)
(H1 - H2) ⊔ ε1  <:  ε2
```

- `H1 - H2` = H1 にあって H2 にないエフェクトアトム（H2 が処理する分は消え、残りは super 側残差へ流れる）。
- H1 ⊆ H2 なら `H1 - H2 = ∅` となり `ε1 <: ε2` のみが残る。
- これは**一般の subtyping 規則**であり、共変注釈という特殊ケース限定ではない。
  種メモ「共変部分に `;` のある注釈」の式 `(handled1 - handled2) | 'e <: 'f` は本規則の一例。

`H - H'` の定義: アトムの集合差。ただしアトムが型引数を持つ場合（`parse<τ,σ>`）の同一性判定は
パス一致 + アリティ一致で行い、型引数同士には別途 subtyping 制約を張る（§2.x で詳述）。

> 注: glob `_` は §1.4 で polarity に応じ ⊤/⊥ へ脱糖済み。本規則を適用する時点では
> ε1, ε2 は ⊤/⊥/行変数/具体行のいずれか。⊤/⊥ が絡む場合は通常の lattice 規則
> （`ε <: ⊤`、`⊥ <: ε`）に従う。

### 2.2 反変側・関数型の分解

規則 EFF-ROW は **polarity 非依存の一般規則**。反変位置のための別規則は要らない。
polarity は関数型を分解する時にだけ現れ、sub/super を入れ替えて子に制約を渡す
（Simple-sub と同じ）。入れ替えた後、子のエフェクト行には再び EFF-ROW がそのまま適用される。

```
thunk[εa1, τa1] -> thunk[εr1, τr1]  <:  thunk[εa2, τa2] -> thunk[εr2, τr2]
──────────────────────────────────────────────────────────────  (FUN)
εa2 <: εa1     τa2 <: τa1        -- 引数側: 反転
εr1 <: εr2     τr1 <: τr2        -- 戻り側: そのまま
```

各エフェクト制約（`εa2 <: εa1` 等）は、行同士なら EFF-ROW、bind 変数が絡むなら 2.4 の
bind 修飾つき constrain で解かれる。

### 2.3 型変数が介在しない具体行（規則 B）

種メモ「型変数が介在しない場合」。`row`（具体的アトム列、残差なし）が残差行に流れ込むとき:

```
row <: [H; ε]
──────────────  (EFF-ROW-CONCRETE)
row - H  <:  ε
```

これは EFF-ROW の特殊形（H1 = row のアトム集合, ε1 = ⊥）。`(row - H) ⊔ ⊥ = row - H`。

### 2.4 bind 変数への constrain（規則 C — この理論の特殊推論）

#### 2.4.1 bind の表現（二本立て）

注釈などで「エフェクト行変数 ρ に handled `H_b` が bind される」とき、**2つの情報源を併存**させる:

1. **上界として普通に持つ**: ρ の通常の上界に `[H_b; ⊤]`（反変 glob 由来の ⊤）を入れる。
   これにより通常の subtyping 伝播・集約（`upper_refs_of(ρ)`）はそのまま動く。残差の実体は
   常にこの上界側にある。
2. **bind テーブル（浮いた制約）**: 「ρ は H_b について特殊推論の対象」というメタ情報を、
   型変数に浮く別レイヤーのテーブルに置く（role/型クラス制約 `Add<α> => ...` が α に浮くのと
   同じ構造）。テーブルが持つのは *どの effect が特殊推論対象か* というメタ情報だけで、
   行の実体（残差）は持たない。

> 旧実装との決定的な違い: 旧 `lambda_param_source_eff_tvs` は**残差変数そのもの**を側テーブルへ
> 逃がしたため、残差の集約地点が割れて消失した。本テーブルは残差を持たず、残差は常に上界から
> 取れる。テーブルは「振る舞いの修飾子」であって型の実体ではない。`'b & [H_b; _]` のような
> 複合型は**作らない**（ChatGPT 指摘の通り `&` の枠に収めない）。

#### 2.4.2 bind 修飾つき constrain

ρ が bind しているのは行 `[H_b; ε_b]`（§2.4.1 で上界に `[H_b; ⊤]` を入れる、すなわち
反変 glob 由来なら ε_b = ⊤）。`lower <: ρ` という制約が来たとき:

1. **通常処理**: lower を ρ の下界へ追加（通常の Simple-sub）。
2. **bind 修飾**: bind テーブルに `ρ ↦ [H_b; ε_b]` があるので、種メモの特殊推論を発動する。
   lower が `[H_lo; ε_lo]` の形のとき、**2 本の制約**を出す:

   ```
   [H_lo; ε_lo] <: ρ      (bind ρ ↦ [H_b; ε_b])
   ──────────────────────────────────────────────  (EFF-BIND)
   [ (H_b ∩ H_lo) ; (H_lo - H_b) ]  <:  ρ          … (i)
   (H_lo - H_b) ⊔ ε_lo            <:  ε_b          … (ii)
   ```

   - (i): bind している `H_b` 部分だけを行の頭に残し（引き上げず）、`H_b` に無い分
     （`H_lo - H_b`）を ρ の残差へ回す。`H_b ∩ H_lo` は handled マージ（∩ は H にのみ効く、§1.3）。
   - (ii): `H_b` に無い分と lower 側残差 ε_lo は、bind 注釈側の残差 ε_b へ流し込む。
     ε_b は通常 ⊤（反変 glob）なので `… <: ⊤` で常に成立し、形を縛らない。

   **検算（compose 例）**: `[io,undet] <: 'b`、bind は `[io; ⊤]`（H_b=`{io}`, ε_b=⊤）、
   H_lo=`{io,undet}`, ε_lo=⊥（lower は残差なしの具体行）。
   - (i): `[ {io}∩{io,undet} ; {io,undet}-{io} ]` = `[io; undet]` <: 'b
     → 種メモ「`[io; undet] <: 'b`、io にしか作用しないので `[io; undet]` のまま」と一致 ✓
   - (ii): `({io,undet}-{io}) ⊔ ⊥` = `{undet}` <: `⊤` → 常に成立、矛盾なし ✓

   要点: lower 側残差 ε_lo を ρ の残差に混ぜない。ρ へは (i) の `H_lo - H_b` だけを残差として渡し、
   ε_lo は (ii) で bind 注釈の残差 ε_b へ向ける。この (i)/(ii) の分離が「ρ への形」と
   「bind 残差への流し込み」を切り分け、残差の取り違え（旧実装の消失バグ）を防ぐ。

> bind 修飾は role 制約と同型のフック: 通常の constrain に「該当 effect の特殊推論を走らせる」
> 修飾を足すだけ。constrain の本筋（上下界の伝播）は変えない。
