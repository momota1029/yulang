## 0. 中心原理（設計順序の決定）

host act API の要求は二つある（ユーザ提示、2026-07-03）:

1. **VM に Rust として実装を注入できること。**
2. **Cranelift の生成物とリンクできること**（そのほうが高速なため）。

> **決定: 2 を先に設計し、1 をその subset とする。**
> ただし先に固めるのは「リンクの機構」ではなく **「呼ばれる側の形」= 実装関数の ABI 契約**。
> symbol 解決・配布・リンク方式は開けたままにできるが、callee の署名・値表現・戻り方は
> 両経路で同一でなければならない。

> **manifest = symbol の名前空間、registry = symbol 解決。**
> native 経路は manifest 由来の symbol 名で直接リンクし、VM 経路は registry が
> 同じ署名の関数ポインタを表引きする。VM の registry とは「リンカがやる symbol 解決を
> 実行時に表でやっているだけ」である。

> **同一の実装関数が VM からも native からも呼ばれる。**
> 前例は OCaml の C primitives: bytecode VM は表を介して、native compiler は直接リンクで、
> 同一の C 関数を呼ぶ。経路の差は間接か直接かだけ。

VM 先行で設計しない理由: VM から始めると `Box<dyn Fn>`、closure キャプチャ、runtime への
`&mut` アクセス、内部値の直渡しに自然に寄りかかり、そのどれも native の call boundary を
越えられない。B-first は最初から regime を強制する。

## 1. 要求スケッチ（band 単位の注入）

将来像: **band が host act を宣言し、その実装（Rust など）を同梱または host から供給される。**
詳細要求は複雑で未確定だが、ABI が今から引き受けるべき制約は次の五つ。

- **R-band1（宣言側が正本）**: どの act が host かは宣言（`host` 修飾子、manifest 指示書 D2'）が
  持つ。compiler 内の表や runtime の定数が持たない。act_id は namespace の dotted path であり、
  band が入ればそのまま band 修飾を含む path になる（schema 変更不要の形に保つ）。
- **R-band2（提供形態に中立）**: 実装は「symbol 名 → 関数」の束として供給される。
  static link / dylib / embedder のプロセス内登録のどれでも、関数の形（§2〜§6）は同一。
  提供形態の選定・署名・sandbox 検査は本文書では決めない。
- **R-band3（grant は runtime 層）**: symbol が解決できることと、その capability が
  そのプログラムに許可されていることは別の層。sandbox deny は runtime 判定が必須なので、
  **意味論上の正本は runtime の capability 検査**に置く（§6）。
- **R-band4（型の越境は manifest 経由）**: 第三者の act は `from_labels` のような
  runtime 内蔵の名前解決に頼れない。構造値の越境は manifest の型レイアウト表（v2 F4）に
  収束させる。v0 はその骨格として label 参照を許す（§3、明示的な暫定）。
- **R-band5（record/replay の漏斗）**: どの経路（VM 表引き / native 直接リンク）でも
  record モードでは同じ漏斗（列番号発行）を通る。native の direct call 高速化は
  record モードでは列番号発行だけ残す（notes/todo/record-replay.md のトレードオフ決定に従う）。

## 2. A1: 実装関数の形（callee ABI）

sync tier（現行の全 operation）:

```rust
// v0 は Rust ABI のまま。ただし機械的に extern "C" へ落ちる形を維持する:
// 薄いポインタ・monomorphic・戻り値は判別付き構造体。
pub type HostOpFn = fn(ctx: &mut HostCtx, payload: &BoundaryValue) -> HostOutcome;

pub enum HostOutcome {
    Return(BoundaryValue),   // sync tier の正常系。ドメインエラーも result 構築値としてここ
    Suspended,               // 予約（suspend tier、scheduler 工事まで綴り未固定）
    HostError(String),       // 契約違反系（payload decode 不能など）。capability failure ではない
}
```

制約（v2 R1〜R3 の関数レベル翻訳）:

- **monomorphic な関数ポインタ。** generics / trait object / closure キャプチャは不可。
  状態は ctx（§4）経由。closure が要る実装は (fn ptr + 登録時に預けた state) に分解する。
- **越境する値は BoundaryValue のみ**（§3）。GC ポインタ・`Rc`・`RuntimeEvidenceValue` を
  直接受け渡ししない（R3）。
- **再入なし。** 関数は Outcome を返して戻るだけ。ctx に perform / resume / eval の口を
  置かない（R2、F8 直交性）。resume が要る tier は Outcome::Suspended + scheduler queue
  （将来工事）であって、同期呼び戻しでは決してない。
- **panic 禁止。** host 実装は panic しない契約。runtime は防御として `catch_unwind` で
  受け、`HostError` 相当の structured failure に落とす（native 側の防御手段は未決、§9）。
- ドメインエラー（`io_err::not_found` 等）は失敗ではなく**普通の返り値**
  （result 構築値、v2 F3）。`HostError` はドメインに翻訳できない契約違反にだけ使う
  （現行の `yulang.host-io-error` 系譜）。

suspend tier は v0 では **discriminant の予約のみ**。token の形・scheduler との受け渡しは
scheduler 工事（v2 thin path 5）で固定する。

## 3. A2: 境界の値表現 BoundaryValue（v0 骨格）

実装関数が見る唯一の値表現。VM の内部値（Rc 木）や native の GC 値との変換は
runtime 側 codec の責務で、実装関数は変換の存在を知らない。

```rust
pub enum BoundaryValue {
    Unit,
    Bool(bool),
    Int(i64),
    Float(f64),
    Str(String),
    Bytes(Vec<u8>),
    Tuple(Vec<BoundaryValue>),
    Constructor { ctor: CtorRef, payloads: Vec<BoundaryValue> },
    HostHandle { type_id: u32, handle: u64 },   // v2 F5 の host 型。v0 では未使用の予約
}

pub enum CtorRef {
    Label(String),   // v0 暫定: "std.io.file.io_err.not_found" 形式の dotted label。
                     // runtime 側 codec が既存の label→DefId 解決で構築する。
    // Tag(...)      // 将来: manifest 型レイアウト表のタグ（R-band4 の最終形）
}
```

- プリミティブは**所有コピー**で越境する（R3。pinning 不要、copying GC は自由に動ける）。
- `CtorRef::Label` は **v0 だけの明示的な暫定**。std の act は runtime に label 解決の
  下地（`from_labels`）が既にあるので成立するが、band の act では成立しない。
  manifest 型レイアウト表（manifest 指示書 Stage 3 / 本文書 Stage γ）が入った時点で
  `Tag` へ移行し、`Label` を退役させる。
- List / Record 等は**必要になった operation が現れた時点で追加**する。先回りで増やさない。
- record/replay の wire encode は、この表現の直列化として定義される予定
  （BoundaryValue が最初の本気の消費者を record に先んじて務める）。

## 4. A3: ctx（host の外部状態）

v2 の「handler パラメータ = host の外部状態」の実体。

```rust
pub struct HostCtx { /* registry 所有の具象 struct（薄い参照で渡る） */ }

impl HostCtx {
    pub fn state_mut<T: 'static>(&mut self) -> Option<&mut T>;  // 登録時に預けた act 状態
}
```

- act 実装は**登録時に**自分の状態（`Box<dyn Any>` 相当）を registry に預ける。
  呼び出し時は `HostCtx` 経由で取り出す。fat pointer は registry の内側に閉じ、
  call boundary を渡るのは具象 struct への薄い参照だけ（C ABI 互換の形を保つ）。
- ctx に runtime のサービス口は**置かない**（診断用の最小窓口が要る場合も、
  perform / resume / eval に触れる形は禁止）。
- 実装補足: console の出力先は現在 runner 所有の buffer
  （runtime.rs の `self.stdout`）なので、Stage α で「登録時に預ける共有 sink」へ
  置き換える必要がある。実装事項であって契約事項ではない。

## 5. A4: symbol 命名（manifest からの決定的導出）

native リンクで使う symbol 名は manifest から決定的に導出する。

- 規則（v0）: Itanium mangling に倣った長さ接頭辞方式。
  `yu_host_` + act path の各セグメントを `<len><seg>` で連結 + `_` + `<len><op_id>`。
  例: `std.io.file.file` の `load` → `yu_host_3std2io4file4file_4load`。
  識別子は lowercase で数字始まりがないため、長さ接頭辞で一意に復元できる
  （`__` 連結案はセグメント内の `_` と衝突するので不採用）。
- **column（列番号）は symbol に使わない。** column は宣言集合が変わると振り直るが、
  symbol は宣言単体から安定であるべき。column は record/replay 専用、symbol はリンク専用。
- native: JIT では symbol 登録（cranelift-jit の symbol 定義）、AOT では未解決 symbol と
  して出してリンカに解決させる。どちらかの選定は native 解封時。
- VM: registry は (act_id, operation_id) で表を引く。symbol 文字列は VM 経路では
  実体化しなくてよい（同じ名前空間の別綴りにすぎない）。
- mangling の最終綴りは v0 unstable の対象（native 解封時に calling convention と併せて確定）。

## 6. A5: 失敗の層と link の関係

v2 F3 の二層に「symbol 解決」を重ねたときの線引き:

- **grant / sandbox deny の正本は runtime 層**（R-band3）。native でも「symbol は解決させて
  grant 検査を runtime に残す」を意味論上の基準とする。
- **未提供（実装が存在しない）**: VM では登録がない = capability failure（現行どおり）。
  native で link 時に未解決 symbol として早期検出することは**許す**（開発体験として良い）が、
  それは前倒しの診断であって意味論の置き換えではない。stub 解決・弱 symbol などの
  実装手段は未決に置く。
- **operation 層のドメインエラー**: BoundaryValue の result 構築値として返る（§2）。
- **契約違反**（payload decode 不能・panic）: `HostError` → runtime の structured failure。
  capability failure とは混ぜない。

## 7. A6: VM subset の定義

> **VM 注入とは、§2〜§6 の関数を registry の表に登録して間接呼び出しする経路のこと。
> それ以上の権利はゼロ。**

- VM だから許される特別な型・特別な再入・内部値の直渡しを**作らない**。
  これが「subset」の意味であり、native 解封時に書き直しが発生しないことの保証。
- 既存の file / console 実装（runtime.rs の `file_load` 等 9 関数）は、この ABI に
  載せ替える最初の住人になる（Stage α）。
- 登録 API（形は実装時に確定、内容はこれで全部）:

```rust
pub struct HostOpRegistration {
    pub act_id: &'static str,
    pub operation_id: &'static str,
    pub f: HostOpFn,
}
// + act 単位の state 預け入れ。builtin 束は evidence-vm が
// builtin_host_registrations() として提供し、CLI はそれを registry に渡す。
// テストは同じ口から mock 実装や独自 act を登録できる（これが FFI smoke の原型）。
```

## 8. 段階

- **Stage α — ABI 骨格と載せ替え（挙動不変）**:
  `BoundaryValue`（v0 骨格）+ `HostOpFn` + `HostCtx` + 登録集合 API を evidence-vm に導入し、
  既存 file / console 実装 9 関数を載せ替える。console の出力 sink を登録形へ。
  既存 fixture 全通過が受け入れ条件（挙動変更ゼロの canary）。
- **Stage β — manifest 指示書との合流**:
  manifest 指示書（改訂1）の Stage 1〜2。registry は「plan の manifest × 登録集合」になり、
  runtime から act path の静的テーブルが消える。
- **Stage γ — band 注入の本体（別承認）**:
  manifest 型レイアウト表（`CtorRef::Tag` 移行、`Label` 退役）、配布形態、
  band 修飾 act の manifest 生成。deriving / wire、realm/band パッケージング、
  scheduler の進み方に依存するため、着手前にユーザと要求を再確認する。

## 9. 決めないこと（実装しないと分からないこと）

- `extern "C"` 化の時期と正確な綴り、calling convention 細部、GC safepoint との交差。
- native 側の panic 防御手段（abort / unwind 境界）。
- 配布形態（static link / dylib / 署名 / sandbox 検査）と band パッケージングの接続。
- suspend tier の token・scheduler queue の形。
- `HostCtx` の診断窓口の要否。
- BoundaryValue の最終形（v0 unstable の中心。List/Record 追加のタイミング含む）。

## 10. Rollback / 改訂条件

- native 解封時に calling convention / safepoint の都合で v0 の形が保てない
  → **v1 改訂**（v0 unstable の枠内の正規手続き。壊れたのではなく予定どおり）。
- 実装が ctx 経由の runtime 再入・継続の inspection を必要とした → 止めて v2 R2/F9 へ戻る。
- BoundaryValue で表現できない越境が必要になり、内部値を直接渡したくなった
  → 止めて本文書 §3 へ戻る（R3 違反は subset 関係の死）。
- テスト期待値が実装出力と食い違い、期待値側を変えたくなった → 止めてユーザに確認。

## 11. Validation（骨子）

1. **載せ替え canary**: Stage α 後、既存の file / console / file-resource /
   unsupported-host fixture が一つも変わらない。
2. **FFI smoke の原型**: テストが登録 API から独自 act の実装（+ テスト用 manifest）を
   登録し、Yulang 側の perform が BoundaryValue 経由で往復する。std を触らずに通ること。
3. **panic 防御**: panic する実装を登録 → structured failure（プロセスは死なない）。
4. **mangling**: 決定的・単射（現行 9 op + 衝突しやすい人工名のコーパスで unit test）。
5. **再入不能は構成で保証**: ctx の API 面に runtime へ触れる口がないことをもって検査に代える
   （型で閉じる。実行時検査は置かない）。
