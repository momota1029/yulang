# std::fs の最小設計

決定日: 2026-05-15

## 設計方針

runtime には最小限の primitive のみを置く。派生関数（行分割・テキスト読み込み・JSON 等）は Yulang で書く。
`std::list` で言うところの「runtime に `cons`/`nil`/`match`、それ以外は Yulang」と同じ思想。

## open/close を露出させない理由

ファイルディスクリプタを開きっぱなしにする API は、

- 複数回 open / close されるとエラーになる
- これが非決定性計算（`undet` エフェクト等）と相性が悪い
- ユーザーに「いま開いているか」を意識させるのは不親切

ので、**毎回 path から読み書きする** スタイルにする。少し遅いが、一番綺麗。

## read API（案 A）

引数:
- path
- byte range（読みたい範囲）

戻り値:
- エラー番号
- 文字列
- 有効な byte range（文字列として valid な範囲まで。途中で UTF-8 が切れていたら手前まで）

## path 型

- 内部表現は **バイト列**。文字で表せないファイル名（非 UTF-8 のパス）も扱えるようにするため
- **パスリテラル構文は作らない**。リテラル化すると、書ける範囲が「文字で表せるもの」に縛られるので。
- 提供するのは path 型に対する操作（join, parent, components, ...）

## なぜ案 A か

| 案 | 内容                                    | 採否 |
| -- | --------------------------------------- | ---- |
| A  | バイト列の読み書きのみ（毎回 path）     | ◎    |
| B  | open/read/close を露出（syscall に近い） | × open/close の取り回しが非決定性と噛み合わない |
| C  | read_to_string まで runtime に出す      | △ runtime を太らせる割に得るものが少ない |
