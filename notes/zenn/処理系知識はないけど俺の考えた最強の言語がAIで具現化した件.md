処理系の専門教育は受けていません。研究室にいたわけでも、実装本を一通り読破したわけでもない。それでもTaPLは読んだし、Algebraic Effectsの論文も追っているし、Simple-Subも眺めたことはある。でも、どれも理論寄りで、いざ実装に降ろそうとすると別の壁が立ち上がる、というのが正直なところでした。

そういう「読んではいるけど実装は遠い」温度感の人、たぶん結構いると思うんです。

そんな自分が、代数的エフェクトとlet多相とrole制約を備えた言語の処理系を作っています。「Yulang」と名付けたこの言語、今のところパーサ・型推論・インタプリタ・native（Cranelift JIT + AOT）まで、それなりに動いている。先に断っておくと、これ、AIと一緒に書いていなければ絶対に到達できなかった場所です。

この記事で書きたいのは、言語デザインそのものではなく、**実装フェーズをどう進めたか**の話。そして、その過程でじわじわ実感した一番大事なメッセージはこれです。

> 理論で完璧であろうとしなくていい。「上手く行くんじゃない？」という直感があれば、走り出していい。

理論は今やLLMが補強してくれます。詰めは彼らに任せられる。人間に残るのは、直感と、それを疑える目です。

## 前提

この記事の射程外として、いくつか前提を置きます。

- パーサは書けている、あるいはAIに書かせて動いている
- 型推論戦略は決まっている（Yulangの場合はSimple-Sub）

「言語を設計するところ」の話はここではしません。実装フェーズに入った後の進め方の話です。

## 第一フェーズ：型推論を実装する

### 論文をmarkdownに変換する

最初にやったのは、Simple-SubのPDFをmarkdownに変換することでした。

PDFのままだと、LLMに食わせたときに数式や推論規則の参照精度がガクンと落ちます。レイアウト崩れで「あの式の右辺」みたいな会話が成立しなくなる。なので、まずLLMと自分が**同じテーブルに乗れる**ようにしておく。

PDF → markdown変換ツールは色々試しましたが、最終的に**Marker**が一番きれいに変換してくれました。数式と推論規則をある程度保ったまま渡せる。これがあると、その後の会話で「あの規則の前提」「この導出」が指差しで通じるようになります。

### playground crateを作る

次にやったのは、`yulang-infer-playground` というplayground crateを作ること。CLIに `.yu` ファイルを食わせると、即座に型推論結果が返ってくる、それだけのものです。

```
$ cargo run -q -p yulang-infer-playground test.yu
```

地味です。地味ですが、これが**LLMと協業する上での足場**になります。

playgroundがあると、LLMにこう頼めるからです。

> 「playgroundでこういうケース試して」

たったこれだけで、LLMがケースを作り、走らせ、結果を確認し、おかしければ自分で原因を考え始める。自分で `.yu` ファイルを書く必要も、エッジケースを思いつく必要もない。**自分の役割は「出力を見て、違和感があったら指摘する」だけ**になります。

書き手から判定者へ。これがplaygroundを早めに整備することの真の意味でした。

### LLMに外注する

playgroundが整ったら、あとは外注を回します。外注先は大きく分けて二種類でした。

**一つ目：論文に書かれていない実装ディテール**

論文って、原理は書いてあるけど実装の細部は読者の宿題になりがちです。例えばSimple-Subでlet多相を扱うとき、binding群をSCC（強連結成分）で分割してから一般化するわけですが、論文にはSCCをどう切るかなんて書いていません。「CSの基礎は知っている前提」で省略されている。

普通だと、ここで「あー、SCCね、Tarjanのアルゴリズム実装するのか……」となって時間を溶かす。

僕は「SCC分割書いて」と一言頼みました。一瞬で出てきました。

地味なエピソードに見えますが、これは**「論文と実装の間の暗黙知をLLMが埋めてくれる」**という事実の象徴です。理論の本筋に集中して、橋渡し部分はLLMに任せる。これが「理論で完璧であろうとしなくていい」の具体的な意味でもあります。

**二つ目：テストとしての資産化**

playgroundで「その場の探索」をしているだけだと揮発的で残らない。なので、検証して正しかったケースは即座にテストとして書き起こさせます。

parser v7だけで154テスト。これは一人で積める量じゃないです。LLMがエッジケースを命名し、テストとして書き、僕は出力を見るだけ。テスト名すら考えなくていい。

この**テスト資産が、後の衛生性事件で命を救うことになる**のですが、その話は後ほど。

## 第二フェーズ：インタプリタを実装する

型推論が動き始めたら、次はインタプリタ。

これはもっと楽でした。「インタプリタ作って」と一言。意味論にこだわりがある部分（代数的エフェクトの継続まわりとか）だけ補足。あとは、ただの木で評価するインタプリタが出てきます。動く。

そして、**またplaygroundを作らせて、テストを繰り返す**。

ここで気づきました。「playground + LLM + テスト」というのは、型推論限定のテクニックではなく、**フェーズ越しに使い回せるテンプレート**なのだと。同じパターンを違うレイヤーに適用するだけで、実装が進んでいく。

## 第三フェーズ：nativeを実装する

インタプリタが意味論の参照実装になった時点で、面白いことが起きていました。

Cranelift JITを「インタプリタと同じ出力を出すように」書く。さらに別にnative版（AOT）も書く。両方とも、インタプリタとの差分テストで検証する。

つまり、**新規にテストを書く必要がない**。インタプリタ層に積み上がったテスト群が、そのまま下層実装の正しさを保証する基準になります。

同じパターンが繰り返されながら、**フェーズが進むにつれて工程が削れていく**。これがテンプレート化の本当の威力でした。

## クライマックス：衛生性事件

ここからが本題、というか、この記事が書かれている直接の理由です。

型推論・インタプリタ・nativeの3フェーズがほぼ書き上がった状態だったある夜、ふと気づきました。

**エフェクトの衛生性が、上手く行っていない。**

具体的には、あるエッジケースで、ハンドラのスコープがずれて、捕捉してはいけないエフェクトを掴んでしまう挙動が起きうる。型の意味論レベルから直さないと根本解決できない、ということが、ぼんやり見えてきました。

絶望しました。

3フェーズ全部直すかもしれない、という可能性に直面したからです。インタプリタは意味論変えれば全部書き直し。nativeのランタイム表現も、衛生性の表現を変えるなら大きく変わる可能性がある。「言語なんか作るべきじゃなかった」とまで思いました。

その日はマックを食って寝ました。

---

仮眠から起きたら、解けていました。

ChatGPT Proに相談していたところ、Simple-Subに依らない方策を導いてくれていて、Codexが**表層型をほぼ変えないで意味論を修正する実装**まで思いついていた。

最初、何が起きているか分かりませんでした。

僕は当然、疑いました。「本当に解けているのか？」「今まで自分がエッジケースに触れてこなかっただけで、テストが通っているだけじゃないのか？」と。

ちゃんと見ました。エッジケースを書き、新しい意味論で意図通りの挙動になるか、衛生性が崩れていた具体例が塞がっているか、確認しました。**本当に解けていました**。エッジケースもLLM側が把握していて、不足を感じた箇所はドキュメントに明記させたら対応してくれました。

ここで何が効いたか。**playground + テストの蓄積**でした。

- 表層型がほぼ変わらない解法だったので、既存のテスト群がそのまま検証に使えた
- エッジケースを試すplaygroundがあったので、新意味論の確認が即座にできた
- 3フェーズの実装そのものが**ほぼ無傷で済んだ**

「テスト資産が救った」のではないんです。**実装全体が救われた**。前夜の絶望からすると、ちょっと現実感のない着地でした。

## AI時代に人間に残る仕事：直感

ここまでの経験から、人間に残る仕事が何なのか、はっきりしてきました。**直感**です。

### ポジティブな直感「これ、上手く行くんじゃない？」

Yulangを作ろうと思ったとき、僕は理論を完璧に詰めてから始めたわけではありません。「代数的エフェクトとSimple-Subとrole制約、組み合わせたら面白そう」という、ふんわりした感触で走り出しました。

普通なら無謀です。理論的な相性を確認しないと、後で破綻する可能性が高いから。

でも今は、走り出してからLLMと一緒に詰めればいい。理論の整合性チェックも、必要な補題の構成も、彼らが手伝ってくれる。

**直感が出たら、走り出していい時代になった**。完璧であろうとしなくていい。

### ネガティブな直感「これ、上手く行ってないんじゃ？」

衛生性事件の起点は、ネガティブな直感でした。「なんかこの挙動おかしくない？」という違和感。これは LLM 側からは（まだ）出てきません。

LLMは「与えられた問題」は驚くほど解けますが、**「ここに問題がある」と自分から提起する力はまだ弱い**。コードを見て「これ実は意味論的に怪しいですよ」と切り出してくれることは、ほぼないです。

だから、**ネガティブな直感は人間が握り続けている重要な役割**になります。「これ、上手く行ってないんじゃ？」という違和感を見逃さないこと。これがAI時代に最も価値のあるスキルかもしれない、と今は思っています。

### 疑える目

最後に、LLMが「解けた」と言ってきたとき、それを疑えること。

僕は衛生性事件のとき、解けたという報告を最初は信じきれませんでした。「テスト通ってるだけで、エッジケースに触れていないだけじゃないのか」と。

この疑いがあったからこそ、ちゃんと検証して、不足箇所をドキュメント化させて、本当に解けている状態に持っていけた。

任せきりにできる人と、信じた上で検証できる人は、別の生き物です。後者であろうとしてください。

## おわりに

理論を完璧に詰めなくても、言語は作れる時代になりました。

必要なのは、

- LLMと同じテーブルに乗るための準備（論文のmarkdown化）
- 探索を外注できる足場（playground crate）
- 蓄積される資産（テスト）
- 直感（ポジティブとネガティブの両方）
- 疑える目

このセットだけです。

理論で完璧であろうとしなくていい。「上手く行きそう」が来たら走り出して、「上手く行ってないかも」が来たら立ち止まる。詰めは彼らに任せていい。

手元のYulangが、表層型をほぼ変えずに型理論ごと刷新された朝、僕は少しだけ新しい場所にいた気がしました。

同じく、論文は読めるけど実装は遠い、と感じているあなたへ。直感を信じて、走り出していいですよ。
