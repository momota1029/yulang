[Yulang](yulang.momota.pw) という言語を作っています。代数的エフェクト・let 多相・role 制約を備えた言語で、パーサから native（Cranelift JIT + AOT）まで動いています。

本記事の主題は、この処理系を **LLM と協業して実装するために組んだ方法論** にあります。具体的には次の四点セットを、フェーズを越えて使い回しました。

1. 論文を markdown 化して LLM と同じテーブルに乗せる
2. playground crate を最初に整備し、人間を「書き手」から「判定者」に降ろす
3. 検証済みケースを即テストとして資産化する
4. 上層の意味論を参照実装にして、下層は差分テストで通す

筆者は処理系の専門教育を受けていません。TaPL を読み、Algebraic Effects と Simple-Sub の論文には目を通していますが、実装本を読破したわけでも研究室にいたわけでもありません。「論文は読めるけど実装は遠い」という温度感のまま、ここまで来てしまいました。本記事はその到達点ではなく、**到達するために何をしたか** の話です。

## 前提

この記事の射程外として、いくつか前提を置きます。

- パーサは書けている、あるいはAIに書かせて動いている
- 型推論戦略は決まっている（Yulangの場合は[Simple-Sub](https://github.com/LPTK/simple-sub)）

「言語を設計するところ」の話はここではしません。実装フェーズに入った後の進め方の話です。

## 方法論：四点セット

実装フェーズ全体を貫いた四点セットを、最初に整理しておきます。後続の各フェーズは、このテンプレートの変奏として読めるようになっています。

### ① 論文を markdown 化して LLM と同じテーブルに乗せる

PDF のままだと、LLM に食わせたときに数式や推論規則の参照精度がガクンと落ちます。レイアウト崩れで「あの式の右辺」みたいな会話が成立しなくなる。最初にやるべきは、変換ツールで論文を綺麗な markdown に整え、自分と LLM が **同じテーブルに乗れる** 状態を作ることです。指差しで議論できる土台になります。

### ② playground crate を最初に整備する

実装本体に手をつける前に、CLI から `.yu` ファイルを食わせて結果を見るだけの小さな crate を作ります。これがあると、LLM に「playground でこういうケース試して」と一言頼むだけで、ケース生成・実行・結果確認まで全部やってくれる。人間の役割が **書き手から判定者へ** 降りる、という構造変化が起こります。

### ③ 検証済みケースを即テストとして資産化する

playground での探索だけだと、知見が揮発します。「これは通るべき」と判断したケースは即座にテストとして書かせる。テスト名すら LLM に命名させていい。parser だけで 154 テスト、というのは人間が一人で積める量ではありませんが、この外注ルートだと自然に貯まります。

### ④ 上層の意味論を参照実装にして、下層は差分テストで通す

型推論で積んだテストが、インタプリタの検証に使える。インタプリタが正しく動くようになれば、それが参照実装になって、JIT や AOT は **新規テストを書かずに差分テストで** 通せる。フェーズが進むほど、同じテスト資産が下層に降りていく構造です。

## 第一フェーズ：型推論を実装する

最初のフェーズである型推論には、四点セットの ①〜③ がそのまま乗りました。

まず点①。Simple-Sub の論文を Marker で markdown 化しました。PDF → markdown 変換ツールは色々試しましたが、最終的に Marker が一番きれいに、数式と推論規則をある程度保ったまま変換してくれた。これがあると、その後の会話で「あの規則の前提」「この導出」が指差しで通じるようになります。

次に点②、`yulang-infer-playground` という crate を立てます。CLI に `.yu` ファイルを食わせると、即座に型推論結果が返ってくる、それだけのものです。

```
$ cargo run -q -p yulang-infer-playground test.yu
```

地味です。地味ですが、playground が整った時点で外注が一気に楽になります。「playground でこういうケース試して」と一言頼むだけで、LLM がケースを生成し、走らせ、結果を見て、おかしければ自分で原因を考え始める。筆者の役割は **「出力を見て、違和感があったら指摘する」だけ** になります。書き手から判定者へ。

それと並行して、**実装の周辺アルゴリズムの外注** が回り始めます。論文は原理を書きますが、それを動くものに下ろすとき、論文の射程外の道具がたくさん必要になる。

例えば筆者は、Simple-Sub の let 多相の扱いを、binding 群を SCC（強連結成分）で分割してから一般化する形に拡張したいと思っていました。Simple-Sub 自体は OCaml 系の `let` / `let rec` がある言語に対して定義されているので、SCC 分割は論文の話ではなく、筆者側の拡張です。

SCC は Tarjan のアルゴリズム——「CS の基礎は知っている前提」の標準技法です。普通なら「あー、SCC ね、実装するのか……」と時間を溶かすところを、筆者は「SCC 分割書いて」と一言頼みました。一瞬で出てきました。

地味なエピソードですが、これは **論文と実装の継ぎ目を LLM が埋めてくれる** という事実の象徴です。理論の本筋に集中して、橋渡し部分は LLM に任せる。

最後に点③。playground での探索だけだと知見は揮発するので、検証して正しかったケースは即座にテストとして書き起こさせます。parser v7 だけで 154 テスト——これは一人で積める量ではありません。LLM がエッジケースを命名し、テストとして書き、筆者は出力を見るだけ。テスト名すら考えなくていい。

この **テスト資産が、後のストレステストで実装全体の命を救うことになる** のですが、その話は後ほど。

## 第二フェーズ：インタプリタを実装する

型推論が動き始めたら、次はインタプリタです。

このフェーズは型推論よりずっと楽でした。「インタプリタ作って」と一言。意味論にこだわりがある部分（代数的エフェクトの継続まわり）だけ補足する。あとは、ただの木で評価するインタプリタが出てきます。動く。

そして、**また playground を作らせて、テストを書かせる**。点②③が型推論フェーズと同じ手付きで回り始めます。点①が要らないぶん、工程は確実に減っています。

ここで気づきました。「playground + LLM + テスト」というのは、型推論限定のテクニックではなく、**フェーズ越しに使い回せるテンプレート** なのだと。同じパターンを違うレイヤーに適用するだけで、実装が進んでいく。

そしてこの段階で、インタプリタは **意味論の参照実装** になります。これが次のフェーズで点④を発動させる土台になります。

## 第三フェーズ：native を実装する

インタプリタが意味論の参照実装になった時点で、点④が効き始めます。

Cranelift JIT を「インタプリタと同じ出力を出すように」書く。さらに別に native 版（AOT）も書く。両方とも、インタプリタとの差分テストで検証する。

つまり、**新規にテストを書く必要がない**。型推論とインタプリタの層に積み上がったテスト群が、そのまま下層実装の正しさを保証する基準になります。

同じパターンが繰り返されながら、**フェーズが進むにつれて工程が削れていく**。これがテンプレート化の本当の威力でした。

## ストレステスト：衛生性事件

ある夜、エフェクトの衛生性が壊れていることに気づきました。

### 衛生性とは

ここで言う衛生性は **「エフェクトを知らない関数はハンドルしない」** という性質です。これがあると、複雑なハンドラを組んだ時に「中で何を部品として使ったか」を呼び出し側が覚えておく必要がなくなります。代数的エフェクトを足場にする言語にとっては、生命線です。

### 最小再現

```yulang
our f() = return 0

our g h = sub:
    h()
    return 1

my a = sub:
    my b = g f
    println b.show
    2

a
```

衛生性が成立しているなら、このプログラムは **何も印字せず、`0` を返すはず** です。`g` は引数 `h` の中身（`f`）を知らないので、`f` 由来の `return 0` は `g` の `sub:` をすり抜けます。けれども `a` は `f` を直接知っているので、`a` の `sub:` が `f` の `return 0` を捕まえる。結果として `a` は `0` を返し、その先の `println` は実行されない——それが衛生性の保証する世界です。

ところが当時の実装では、**`0` が印字され、`2` が返ってきました**。`f` の `return 0` が、それを知らないはずの `g` の `sub:` に捕獲されていたのです。Soundness そのものが壊れていました。

### 原因：`compose` という矛盾

問題の核には、シンプルだけれど厄介な式がありました。

```yulang
my compose f g x = f(g x)
```

この `compose` には素直に次のような型がつきます。

```
(a [e] -> [f] b) -> (c -> [e] a) -> c -> [f] b
```

読み下すと「`g x` で発生したエフェクト `e` を、`f` が受け取って `f` のエフェクト系に流す」式に見えます。エフェクトをパイプで繋ぐ、ごく自然な書き方です。

ところが、ここに衛生性を厳密にぶつけると問題が起きます。`f` の本体は `e` を「知らない」関数として書かれているかもしれません。その場合、衛生性のルールでは `f` は `e` をハンドルしてはいけない——つまり、**エフェクトをパイプで繋ぐコード自体が書けない言語**になってしまう。

逆に衛生性を緩めると、冒頭の `g f` の例で起きた捕獲事故が出る。

これがそのまま矛盾でした。Simple-Sub と素朴な型注釈の組み合わせでは、この矛盾を解く道具立てが足りていなかった、ということです。

### 絶望

実装は型推論・インタプリタ・native の 3 フェーズ。型の意味論レベルから直すとなると、全部に波及しかねません。特に native のランタイム表現が大きく変わる可能性が怖かったです。「言語なんか作るべきじゃなかった」とまで思いました。慰めに倍ビッグマックを食べて仮眠を取りました。

### 翌朝

仮眠から起きると、ChatGPT Pro に投げていた相談に答えが返っていました。投げた質問はシンプルです。

> 今の方式をブラッシュアップして穴を塞ぐにはどうするか？ ただし表層の型は変えないこととする。

数十分後に、Simple-Sub に依らない方策が返ってきていて、Codex のほうでは表層型をほぼ変えずに意味論を修正する実装案まで進んでいました。

解法の骨子はこうです。Simple-Sub の **外側** に、「**何がハンドル可能なエフェクトか**」を型クラスのように **ぶら下げる**。ぶら下がっているので推移律が自然に成り立ち、衛生性の境界は型注釈の有無で明示的に切り替わる。そしてこの追加情報は **ほとんどの場合は邪魔なので非表示** にしてある——既存のコードは型推論をそのまま通過します。

つまり、Simple-Sub の表層は無傷で、衛生性を担保する裏の制約だけが追加された、という整理です。

### 検証

最初は信じきれませんでした。「テストが通っているだけで、エッジケースに触れてこなかっただけではないか」と。

冒頭の `g f` の例を実際に走らせ、**何も印字されず `0` が返ってくる** ことを確認しました。続けて気になるエッジケースを試し、不安な箇所は「ここが心配だ」とドキュメントに明記させて、その都度詰めていきました。

本当に解けていました。

### 四点セットが救ったもの

ここで効いたのは冒頭で挙げた四点セットの蓄積そのものです。

- 表層型がほぼ変わらない解法だったから、**既存のテスト群がそのまま検証に使えた**（点③）
- playground があったから、新意味論の挙動を **走らせて即確認** できた（点②）
- 3 フェーズの実装はほぼ無傷で済み、native のランタイム表現にも触らずに済んだ（点④の差分テスト構造のおかげ）

「テスト資産が救った」というより、**実装全体が救われた**、というのが正確な表現です。前夜の絶望からすると、現実感のない着地でした。

## AI 時代に人間に残る仕事：直感

LLM に大半の詰めを外注できる時代になって、人間側に残るのは何か。本記事の経験から言えるのは、**直感**（ポジティブとネガティブの両方）と **疑える目** です。

### ポジティブな直感「これ、上手く行くんじゃない？」

Yulang を始めたとき、筆者は理論を完璧に詰めてから走り出したわけではありません。「代数的エフェクトと Simple-Sub と role 制約、組み合わせたら面白そう」というふんわりした感触で出発しています。

従来であれば、理論的な相性を確認しないと後で破綻する、というのが定石でした。今は、走り出してから LLM と一緒に詰める動き方が現実的になっています。理論の整合性チェックも、補題の構成も、対話相手として LLM が伴走してくれる。

走り出すための直感に、以前ほどの理論的裏打ちが要らなくなっている、ということです。

### ネガティブな直感「これ、上手く行ってないんじゃ？」

衛生性事件の起点は、ネガティブな直感でした。「なんかこの挙動おかしくない？」という違和感です。**これは現状の LLM 側からは出てきません**。

LLM は与えられた問題を驚くほど解きますが、「ここに問題がある」と自分から提起する力はまだ弱い。コードを見て「これ実は意味論的に怪しいですよ」と切り出してくれることは、まずない。

だから、「上手く行ってないんじゃないか」という違和感を拾える人間が、依然として必要です。

### 疑える目

もうひとつ、LLM が「解けた」と言ってきたとき、それを **疑える目** が要ります。

衛生性事件のとき、筆者は最初、解決報告を信じませんでした。「テスト通ってるだけで、エッジケースに触れていないだけじゃないのか」と。この疑いから検証に入り、不足箇所をドキュメント化させて、本当に解けている状態に着地できた。

任せきりと、信じた上で検証することは別物だ、ということです。

## おわりに

四点セット——markdown 化された論文、playground、蓄積されるテスト、参照実装と差分テスト——と、人間側に残る直感と疑える目。本記事で扱った道具立てはこれだけです。

これは Yulang に特化した話ではなく、論文 + LLM + 自作処理系というセットアップなら、おそらく他の言語実装にも転用できる構造だと考えています。

衛生性事件の翌朝、表層型をほぼ変えずに型理論ごと書き換わった Yulang を眺めながら、「論文は読めるけど実装は遠い」という温度感のままだった自分が、少しだけ違う場所にいることに気付きました。同じ温度感の方にとって、本記事のセットアップがその移動を少し早めるなら、書いた甲斐があります。
