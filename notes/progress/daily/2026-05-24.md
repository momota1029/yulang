# 2026-05-24

## Runtime finalize instance cache surface

既存 compiled-unit 永続 cache へ runtime-finalize instance を載せる前段として、
`yulang-runtime-finalize` 内に cache surface を追加した。

- `FinalizeInstanceKey`: `binding + type substitutions`
- `CachedFinalizeInstance`: monomorphic `scheme` / `body` / `callee_type` / `result_type`
- `FinalizeInstanceCacheSurface`: serde 可能な永続化候補 surface
- `FinalizeInstanceCache`: in-memory cache + policy + profile

`finalize_module_with_cache` は、append 時に cache hit した instance の materialized body を再利用する。
miss 時は従来どおり graph solve / materialize を行い、policy に通る instance だけ cache に入れる。
肥大化対策として、default policy は最大 4096 entries、1 body あたり最大 2048 expr nodes を上限にする。
これは persistent artifact へ保存する前の safety valve で、cache は advisory のため skip しても意味は変わらない。

さらに `FinalizeInstanceArtifactCache` を追加し、既存 compiled-unit cache と同じ
`CompiledUnitManifest` set を key にして、runtime-finalize instance surface を sidecar file として読めるようにした。
path は既存の `compiled-units/v*/parser-v*/` 配下に `runtime-finalize-v*/instances-*.bin` を作る。
infer の compiled-unit artifact 本体に runtime-ir body を混ぜると層が逆流するため、artifact 本体は触らず、
同じ manifest identity を使う隣接 cache として扱う。
呼び出し側は `read_cache_for_manifests` / `write_cache_for_manifests` で
`FinalizeInstanceCache` を直接 load/store できる。

追加 regression:

- `my id x = x; id 1` を同じ `FinalizeInstanceCache` で 2 回 finalize すると 2 回目は cache hit する。
- `FinalizeInstanceCacheSurface` へ出したあと、復元した cache からも同じ instance が hit する。
- 実際に finalize して作った `FinalizeInstanceCache` を `FinalizeInstanceArtifactCache` へ書き、
  manifest key から読み戻した cache で solver が同じ instance を hit する。
- fake std manifest set で `FinalizeInstanceArtifactCache` の write/read が roundtrip する。
- compiled artifact/parser format が混ざった manifest set は reject する。

未解決:

- `"hello".println` を std ありで `runtime-finalize` まで通す通常スタック test は、現状の
  finalize instance cache だけでは成立しない。compiled std artifact を使って infer/core
  merge までは通常スタックで進むが、その後の `yulang_runtime::lower_core_program` が
  stack overflow する。これは finalize cache 以前の core -> runtime IR lowering cache が
  まだないため。

## Cached std runtime IR entrypoint

`yulang-compile` に、compiled dependency artifact cache から runtime IR 直前まで進める入口
`runtime_ir_module_from_virtual_source_with_dependency_cache` を追加した。

- cache-key SourceSet で dependency manifest を作る。
- compiled dependency bundle があれば、trusted compiled runtime surface を user program へ merge して
  `yulang_runtime::lower_core_program` だけを走らせる。
- bundle がなければ CLI と同じく full SourceSet を取り直して lower し、dependency unit artifact /
  bundle / semantic bundle を書く。
- 返り値は runtime IR module、cache hit かどうか、dependency manifest set を持つ。
- `runtime_ir_module_from_virtual_source_with_dependency_cache_read_only` も追加した。
  これは既存 compiled dependency cache だけを読み、full lower / cache warmup はしない。
  full bundle がなくても dependency-closed な unit artifact 群があれば bundle を復元して使う。

注意点として、miss 時に cache-key SourceSet のまま full lower すると、std graph cache 由来の空 source が
混ざり、`"hello".println` の method surface が解決されない。miss 時の full SourceSet 取り直しは、
この症状側の補正ではなく、CLI の dependency cache path と同じ責務境界へ戻す修正。

`yulang-runtime-finalize` には ignored integration test として
`prewarmed_std_runtime_ir_cache_can_enter_runtime_finalize` を追加した。
これは既に温まっている project compiled dependency cache だけを使って `"hello".println` の runtime IR を作り、
その module を新 `finalize_module` に渡す。cache が未作成または壊れていれば即失敗し、
test 内で cold std lower は走らせない。

## Cache invalidation after stale artifact failures

既存 user cache から、現在の runtime と意味が合わない artifact が読める症状を確認した。

- source cached control VM から `ExpectedString(Int("20"))` / `ExpectedFloat(String(...))` が出る。
- compiled dependency runtime surface merge で `ConflictingPrimitiveType { family: Frac }` が出る。

temp cache で同じ入力を走らせると stale artifact 固有の panic は消え、現在の pipeline の型エラーへ落ちる。
これは cache 内容そのものを直す問題ではなく、意味が変わった artifact を同じ version で読めてしまう問題。

対応:

- `COMPILED_UNIT_ARTIFACT_FORMAT_VERSION` を `17` から `18` へ上げた。
  old `compiled-units/v17` は current binary から読まれない。
- `CONTROL_VM_ARTIFACT_VERSION` を `8` から `9` へ上げた。
  source cached control VM と明示 load artifact の両方で、古い control artifact を拒否する。

これで stale cache の症状は current pipeline の通常エラーと分離できる。
残っている `branch result type mismatch: expected int, got std::range::bound` と
`cannot match a tuple pattern against std::list::list_view<()>` は cache ではなく現在の std / lowering 側の別バグ。

## Nominal enum variant payload surface for runtime lowering

上の current pipeline 側エラーは、runtime lowering が nominal enum の case payload 型を復元できないことが原因だった。
`std::range::bound::included x -> x` は constructor binding が runtime env に残る場合だけ復元できるが、
`std::list::list_view::node(left, right)` のように `view_raw : list<a> -> list_view<a>` だけが見える場合、
pattern payload が `Any` / scrutinee の nominal 型に落ち、tuple pattern と局所 binding 型が壊れていた。

対応:

- `CoreGraphView` に `enum_variants: Vec<EnumVariantGraphNode>` を追加した。
- infer export は enum variant constructor の frozen scheme から、解決済みの enum path / tag / type params / payload surface を出す。
  構文上の `SigType` から直接作ると相対 `path` が `std::path::path` に解決されないため、必ず frozen scheme の resolved type surface を使う。
- runtime lowering の `variant_payload_expected` は、構造 variant に加えて graph の nominal enum payload surface を参照し、
  `list_view<str>` の `node` payload を `(list<str>, list<str>)` へ置換する。
- constructor binding が存在する場合の旧 fallback も残した。
- compiled dependency artifact の serialized graph 形状が変わったため、
  `COMPILED_UNIT_ARTIFACT_FORMAT_VERSION` は `18` から `23` へ上げた。
  作業途中の `v19` / `v20` / `v21` / `v22` cache は、未解決の payload surface export や stale bundle で温まっている可能性があるため読まない。
- `CoreGraphView::enum_variants` は current artifact では必須にした。
  同じ format version の artifact がこの runtime surface を欠いていた場合、空 surface として読まず deserialize error にする。
- `CompiledRuntimeBundle` の merge が `graph.enum_variants` を落としていた。
  個別 unit JSON には nominal enum payload surface が入っていても、bundle cache を読むと消えるため、
  `std::list::list_view::node(left, right)` が `list_view<()>` に戻っていた。
  bundle merge でも enum variant surface を重複排除しながら引き継ぐようにした。
- source/control VM artifact も作業途中の悪い runtime lowering 結果を持つ可能性があるため、
  `CONTROL_VM_ARTIFACT_VERSION` は `9` から `10` へ上げた。

確認:

- `RUSTC_WRAPPER= cargo test -q -p yulang-runtime`
- `RUSTC_WRAPPER= cargo test -q -p yulang-vm control`
- `RUSTC_WRAPPER= cargo test -q -p yulang-infer lowers_case_over_nominal_list_view_with_apply_in_body`
- `RUSTC_WRAPPER= cargo test -q -p yulang-compile`
- `RUSTC_WRAPPER= cargo test -q -p yulang-runtime-finalize`
- `cargo fmt --all --check`
- `printf '["あ"].debug\n' | RUSTC_WRAPPER= cargo run -q -p yulang -- dump --runtime-ir --no-cache`
- `printf '((1..20).each + (1..20).each).list.say\n' | RUSTC_WRAPPER= cargo run -q -p yulang -- dump --runtime-ir --no-cache`

確認:

```bash

RUSTC_WRAPPER= cargo test -q -p yulang-runtime-finalize finalize_instance_artifact_cache_rehydrates_solver_cache -- --nocapture
```

## Runtime finalize small executable coverage

`yulang-runtime-finalize` の新 solver で、小さい source program を runtime IR finalize 後に
VM へ渡して実行する regression を追加した。

- `my x = if true { 1 } else { 2.0 }; x` は branch result を `float` へ join し、VM 結果 `1.0` になる。
- nested lexical block は block tail の値を返し、VM 結果 `2` になる。
- 多相関数 body 内の lexical block は、emit された単相 alias body の再 scan 後も VM 結果 `3` になる。

この過程で、runtime-finalize の `materialize_expr` が `Block` の古い外側 annotation を残したまま
`BindHere` の保存/削除を判断していた問題を直した。`Block` は materialize 済み tail の runtime type を
result type として再構成し、tail が thunk の場合だけ `BindHere` が残る。

確認:

```bash
RUSTC_WRAPPER= cargo test -q -p yulang-runtime-finalize
RUSTC_WRAPPER= cargo test -q -p yulang-compile
cargo fmt --all --check
git diff --check
```

prewarmed std cache を使う ignored test では、scalar ref assignment / ref read assignment /
nested block ref assignment / `"hello".println` の runtime-finalize entry は個別に通ることを確認した。
一方、二重 loop 内の ref update は current no-cache runtime lowering でも
`std::flow::loop::for_in` の effect mismatch に落ちるため、runtime-finalize の handler leak とは分けて扱う。
