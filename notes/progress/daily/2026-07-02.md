## Public contract manifest first slice

- Context:
  - ChatGPT Pro recommended turning the Public Contract Spine into an
    executable contract rather than adding more language surface.
  - Runtime and diagnostics already had shared `.yu` fixtures, but the
    fixture set was still driven by Rust test code instead of a repository
    manifest.
- Implementation summary:
  - Added `tests/yulang/cases.toml` as the first public contract manifest.
  - The manifest currently covers focused CLI `run` cases for:
    - `nondet_once_triple`
    - `optional_record_defaults`
    - `attached_impl_pick`
    - `list_update`
  - It also covers focused CLI `check` diagnostics for:
    - `type_annotation_mismatch`
    - `unresolved_value_name`
    - `catch_missing_arm_body`
  - Added `public-signature` manifest support and canaries for:
    - data-position effectful function public signature
    - nested data-position effectful function public signature
    - `std.control.var.ref.update`
    - `std.text.parse.choice`
    - `std.control.flow.loop.for_in`
    - `std.control.nondet.nondet.#act-method:once`
    - `std.control.nondet.nondet.#act-method:list`
  - Extended the manifest to include the remaining focused public diagnostics
    fixtures for unresolved type names, unsupported type syntax, top-level
    mutable bindings, parser syntax recovery, rule lazy quantifiers, and
    malformed `catch` arms.
  - Moved representative runtime/compile error sources into shared fixtures and
    covered unhandled effects, non-call calls, non-record field access, pattern
    mismatch, missing record fields, unresolved methods, and ambiguous methods.
  - Fixed the cached runtime build path so specialization failures go through
    the same route-error formatter as the no-cache path. Without this, cached
    `run` reported raw subtype constraints instead of the public
    `compile error [yulang.*]` message and hint.
  - Added repo-root manifest entries for public examples `examples/01_*.yu`
    through `examples/13_console.yu`, plus `examples/showcase.yu`.
    Console output is checked without `--print-roots`; showcase stays a compact
    contains check rather than a large stdout snapshot.
  - Clarified standard API contract staging:
    - `tests/yulang/cases.toml` is for current executable behavior, not TODO or
      ignored placeholders.
    - file/server/host capability cases should enter the manifest only when
      native or unsupported-host public CLI behavior can be observed.
    - provisional helpers should not receive stable contract tags.
  - Added a source-level public signature invariant helper that scans every
    public signature type in selected contract-spine std modules and rejects
    private stack evidence (`#...` / `AllExcept...`) in the type part. The scan
    strips public symbol names and bodies first, so `#act-method` names remain
    valid while leaked type evidence is still caught.
  - Hardened the manifest runner schema:
    - case names must be unique;
    - every case must assert either CLI output or a public signature type;
    - hardening smoke already runs the manifest through its `cargo test -q -p
      yulang public -- --test-threads=1` gate because the manifest test is a
      public CLI test.
  - Connected diagnostics manifest entries to the structured `SourceDiagnostic`
    payload:
    - check cases can now assert diagnostic count, code, label, related count,
      and related origins;
    - current public diagnostics fixtures assert the shared payload without
      broadening stdout snapshots.
  - Removed the older inline public examples CLI bridge after the manifest
    became the stronger executable owner for `examples/01_*.yu` through
    `examples/showcase.yu`. This avoids running the same public examples twice
    in `cargo test -p yulang --test cli`.
  - Updated solver invariant docs to point at the manifest public-signature
    cases and the all-public-signature scan for contract-spine std modules.
  - Added a manifest-driven CLI integration test that reads the manifest with
    `toml`, runs each CLI case through the `yulang` binary, uses repo `lib` as
    std root for runtime cases, keeps runtime cache isolated, and checks public
    signature cases through the source dump API.
  - Updated `tests/yulang/README.md`, `docs/status.md`, `notes/todo/testing.md`,
    and `tasks/current.md` so the manifest is visible as the first executable
    contract entrypoint.

## File-resource rollback fixture probe

- Context:
  - Contract v1 requires `text_with` rollback when the callback exits through an
    effect before normal scope exit.
- Probe:
  - A minimal native fixture shape was tested locally:
    - read initial file text;
    - enter `std::io::file::text_with`;
    - update the managed ref buffer;
    - `fail edit_err::abort`;
    - close the error with `edit_err::wrap` outside the `text_with` call;
    - read the backing file again.
- Result:
  - `check` succeeds with quoted temp-file substitution.
  - mono specialization fails with:
    `conflicting type candidates for slot 0: [std::io::file::file, edit_err] vs [edit_err]`
- Decision:
  - Do not add the fixture to `tests/yulang/cases.toml` yet. The manifest must
    stay executable and should not carry expected-failure placeholders.
  - Do not add a broad effect-row exception in specialization. The failure is
    narrower: generated error wrapping and `text_with` callback residuals are
    valid at the public type boundary, but specialize still collapses the
    callback file residual against the wrapped error residual in this shape.
  - The next fix belongs near specialize2's existing effect-subtraction /
    callback-residual handling, not in the file API wrapper.

## Contract manifest schema validation in the CLI

- Context:
  - The public contract manifest policy had Rust-side checks in
    `crates/yulang/tests/cli.rs`, but a packaged `yulang contract` binary could
    still start from a malformed manifest shape.
  - Contract v0 / v1 evidence should fail at the release CLI boundary too, not
    only in repository tests.
- Implementation summary:
  - Added manifest validation to `crates/yulang/src/contract.rs` immediately
    after TOML loading and before case / contract filters are applied.
  - The CLI now rejects:
    - duplicate case names;
    - non-canonical or unknown tags;
    - kind/tag mismatches;
    - `stable-core` cases mixed with preview / migration canary /
      compile-error tags;
    - malformed `runtime-error` tag combinations;
    - `standard-api` cases without exactly one stability tag and a narrower API
      area tag;
    - `file-resource` runtime cases without exactly one host scope and
      `resource-lifetime`;
    - public-signature cases without `expect_type`;
    - check cases without compact diagnostic code / severity / primary range /
      related count expectations.
  - Added CLI tests for unknown manifest tags and file-resource runtime cases
    missing host scope.
