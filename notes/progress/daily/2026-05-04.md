## Principal Substitution Candidates

ChatGPT へ渡すため、直近の実装結果とテスト結果をここにまとめる。

### Implemented

- `ApplyEvidence` に `substitution_candidates` を追加した。
- `complete_principal` が principal callee / arg / result slot から `PrincipalSubstitutionCandidate` を export するようにした。
- candidate は `var`、`relation` (`Lower` / `Upper` / `Exact`)、`ty`、`source_edge`、`path` を持つ。
- `ApplicationCallee` / `ApplicationArgument` の source edge と derived expected edge も candidate 入力にした。
- closed ではない structural candidate も evidence として残し、runtime solver 側だけが open candidate を無視する。
- `SubstitutionSpecialize` は spine 内の複数 `ApplyEvidence` から candidates を集め、次の場合だけ substitution にする。
  - `Exact` が 1 種類の closed type に一致する。
  - `Lower` と `Upper` の両方に同じ closed type がある。
  - 片側候補、衝突候補、open / `Any` / value `Never` は bind しない。
- core IR verbose 表示では、候補が `subst-candidates=[...]` として見える。

## Direction Guardrail: One-Pass Typed IR

The main goal is still to move toward a typed IR that can be generated with as
little looping and later rediscovery as possible.

The current evidence / ShapeTable work is on track only insofar as it moves
decisions earlier:

```text
infer/lower records boundaries
complete_principal exports expression-local shape/evidence
runtime lower consumes that evidence instead of rediscovering it
monomorphize relies less on demand/fixpoint recovery
```

Future changes should be judged by this question:

```text
Does this move a type or adapter decision from a later heuristic loop into
complete_principal / ShapeTable / explicit evidence?
```

If the answer is yes, it supports the one-pass typed IR direction. If a change
only adds profile output, strengthens runtime heuristics, globally solves open
candidate graphs, or special-cases std helpers without reducing later recovery,
it is probably sideways.

The current `ShapeTable` is therefore not the destination. It is a small
checkpoint toward a shaped core where every expression has intrinsic/contextual
value and effect shape, principal instantiation data, and adapter/coercion
boundary information that runtime can eventually trust.

- The real `1 + true` CLI run now reports a source frame:

```text
error: result type mismatch in call to `+`: expected bool -> bool, got int -> int
  at 2:3
```

The line number is based on the diagnostic source used by the lowerer. The
important part for this checkpoint is that runtime `TypeMismatch` can now travel
through source-edge-only `ApplyEvidence` to a source frame for the real operator
case, not only the synthetic unit test. Keeping this evidence source-only avoids
the `std::undet` / role-method regression that occurred when full principal
evidence was attached to rewritten concrete impl calls.

## Correction: Principal-Unify Is the Runtime Subset

The fast subset target is not a Core IR interpreter. It is a lightweight
principal-unification / elaboration pass that uses exported inference evidence
to produce executable, monomorphic runtime IR without rediscovering type
relations through the old demand/refine loop.

The incorrect direct Core IR evaluator was removed. The new slice adds a
separate `principal_unify` implementation under the monomorphize pipeline and
routes `YULANG_PRINCIPAL_ELABORATE=1` through it instead of the old
substitution-specialize implementation.

Current behavior:

- uses `PrincipalElaborationPlan` / `ApplyEvidence` as the only main-path
  source of substitutions
- clones bindings with closed substitutions from the plan
- rewrites the apply spine to the specialized binding
- recursively rewrites cloned bodies with the same evidence-driven path
- does not run runtime shape inference helpers on the principal path
- does not invent adapter holes; adapter needs become explicit skip reasons
- old demand/refine remains fallback outside strict mode

## Strict Principal-Unify Reaches `07_junction`

The direct principal-unify path now runs `examples/07_junction.yu` in strict mode
without old demand fallback:

```sh
YULANG_PRINCIPAL_ELABORATE=1 \
YULANG_PRINCIPAL_ELABORATE_STRICT=1 \
RUSTC_WRAPPER= cargo run -q -p yulang -- --run --runtime-phase-timings examples/07_junction.yu
```

Result:

```text
[0] 1
mono_passes: 2, effective_passes: 2, specializations: 10
demand_queue: attempted=0, pushed=0
skip_reachability surviving_actionable_targets=0 surviving_benign_targets=2
```

What changed:

- `principal_unify` now treats `BindHere` in callee position as a transparent
  forced-callee wrapper when reconstructing an apply spine. This lets the direct
  path see lowered shapes like `((bind_here ((Fold::fold xs) ())) f)` as the
  original `Fold::fold xs () f` spine.
- Partial role-method calls are skipped so the outer full call can drive role
  resolution and effect wrapping.
- Role impl resolution falls back to ordinary principal specialization if an
  optional effect-context wrapper cannot be built. That keeps role calls from
  surviving as runtime requests.
- Handler-adapted specializations rebuild their call sites with the same
  handler-adapted callee type that was emitted. This fixed the stale-call-type
  bug where `junction` was specialized as effectful but its call was rebuilt as
  if the argument were pure.
- `principal_arg_adapter` no longer trusts a `RuntimeType::Thunk` annotation by
  itself. If an expression is typed as a thunk but is not syntactically a thunk
  and is not an apply whose callee actually returns an effect thunk, the adapter
  materializes an explicit `Thunk` node. This fixed the root `Ord::lt` call,
  where the value had contextual thunk type but still executed as a plain bool
  apply.
- Skip target actionability is now classified in `principal_unify`, so surviving
  `skip-non-generic-callee` revisits of already-specialized bindings are benign.
