## Principal Substitution Candidates

ChatGPT へ渡すため、直近の実装結果とテスト結果をここにまとめる。

### Implemented

- `ApplyEvidence` に `substitution_candidates` を追加した。
- `complete_principal` が principal callee / arg / result slot から `PrincipalSubstitutionCandidate` を export するようにした。
- candidate は `var`、`relation` (`Lower` / `Upper` / `Exact`)、`ty`、`source_edge`、`path` を持つ。
- `ApplicationCallee` / `ApplicationArgument` の source edge と derived expected edge も candidate 入力にした。
- closed ではない structural candidate も evidence として残し、runtime solver 側だけが open candidate を無視する。
- `SubstitutionSpecialize` は spine 内の複数 `ApplyEvidence` から candidates を集め、次の場合だけ substitution にする。
  - `Exact` が 1 種類の closed type に一致する。
  - `Lower` と `Upper` の両方に同じ closed type がある。
  - 片側候補、衝突候補、open / `Any` / value `Never` は bind しない。
- core IR verbose 表示では、候補が `subst-candidates=[...]` として見える。

## Direction Guardrail: One-Pass Typed IR

The main goal is still to move toward a typed IR that can be generated with as
little looping and later rediscovery as possible.

The current evidence / ShapeTable work is on track only insofar as it moves
decisions earlier:

```text
infer/lower records boundaries
complete_principal exports expression-local shape/evidence
runtime lower consumes that evidence instead of rediscovering it
monomorphize relies less on demand/fixpoint recovery
```

Future changes should be judged by this question:

```text
Does this move a type or adapter decision from a later heuristic loop into
complete_principal / ShapeTable / explicit evidence?
```

If the answer is yes, it supports the one-pass typed IR direction. If a change
only adds profile output, strengthens runtime heuristics, globally solves open
candidate graphs, or special-cases std helpers without reducing later recovery,
it is probably sideways.

The current `ShapeTable` is therefore not the destination. It is a small
checkpoint toward a shaped core where every expression has intrinsic/contextual
value and effect shape, principal instantiation data, and adapter/coercion
boundary information that runtime can eventually trust.

## Source Frames for Runtime Apply Mismatch

Continued the source-edge bridge so real operator applications keep enough
evidence for runtime diagnostics.

### Implemented

- Role/operator apply rewrites now preserve source-edge-only `ApplyEvidence`
  instead of rebuilding source-level applies with `evidence: None`. This keeps
  diagnostic source edges without leaking generic role-method principal evidence
  into the concrete impl call.
- `collect_apply_spine_with_apps` preserves each original TIR app node so export
  can reuse its `callee_edge_id`, `arg_edge_id`, expected callee slot, and
  expected argument slot.
- Runtime lowering now propagates the current apply adapter source into nested
  type mismatch errors. This preserves the `+` callee label and source edge ids
  when a mismatch is discovered while lowering the callee side of an outer apply.
- Infix/prefix/suffix/index suffix applications now call `make_app_with_cause`
  with the suffix source range instead of losing the range through `make_app`.
- The real `1 + true` CLI run now reports a source frame:

```text
error: result type mismatch in call to `+`: expected bool -> bool, got int -> int
  at 2:3
   2 | 1 + true
     |   ^^^^^^
```

The line number is based on the diagnostic source used by the lowerer. The
important part for this checkpoint is that runtime `TypeMismatch` can now travel
through source-edge-only `ApplyEvidence` to a source frame for the real operator
case, not only the synthetic unit test. Keeping this evidence source-only avoids
the `std::undet` / role-method regression that occurred when full principal
evidence was attached to rewritten concrete impl calls.

## PrincipalElaborationPlan Skeleton

Added the first exported plan shape for the direct elaboration path.

### Implemented

- Added `core_ir::PrincipalElaborationPlan` and attached it to
  `ApplyEvidence`.
- Added related plan data:
  - `PrincipalElaborationArg`
  - `PrincipalElaborationResult`
  - `PrincipalAdapterHole`
  - `PrincipalAdapterKind`
  - `PrincipalElaborationIncompleteReason`
- Export now builds a conservative plan whenever an apply has
  `principal_callee` evidence.
- The initial plan records:
  - target path when the principal callee head is known
  - principal callee type
  - closed substitutions
  - one argument slot with intrinsic/contextual bounds and source edge
  - result intrinsic bounds
  - incomplete reasons for missing target, missing substitutions, open
    candidates, missing expected arg, and open result bounds
- Runtime type substitution now substitutes the plan alongside apply evidence.
- `CoreShape` now counts:
  - applies with principal elaboration plans
  - complete plans
  - incomplete plans

This is still non-behavior-changing. Runtime does not execute the plan yet, and
adapter holes are still empty. The purpose is to make the next strict
elaboration step visible without adding runtime shape inference.
