## lowering

- `if` / `elsif` / `else` を nested `Case` へ lowering する処理を追加した。
  - 条件は `bool` に exact constraint。
  - `else` がない `if` は then の値を捨て、全体を `unit` に寄せる。
- bare `true` / `false` を bool literal として lowering するようにした。
- qualified value path (`m::x`, `std::int::eq` など) を `ModuleTable::value_path_at` で解決し、通常の `RefId` use として lowering するようにした。
- plain string literal を `Lit::Str` として lowering するようにした。
  - string interpolation はまだ未対応。
- `{x: expr, y: expr}` 形式の record literal を `Expr::Record` へ lowering するようにした。
  - record spread はまだ未対応。
- debug std source に `std.int` / `std.float` / `std.bool` の primitive 宣言 module を追加した。
  - `std::int::mod` は `pub (mod)(x: int, y: int): int` として宣言できるようにした。
- act operation 宣言を companion scope の value DefId として登録し、`catch` arm の bare operation
  pattern をその DefId から act family へ解決するようにした。
  - `return a, _ -> ...` は operation 名 `return` ではなく、宣言元 act `sub` の effect row item になる。
  - operation value は通常 binding lowering の cursor から除外する。
- `never` を builtin Bottom 型として annotation / signature lowering から `Pos::Bot` / `Neg::Bot` へ落とすようにした。
- top-level の分割束縛を lowering するようにした。
  - `my (x, y) = ...` は各名前を root value として登録し、hidden `#destructure` を一度だけ評価する。
  - `my (mod) = ...` や `pub (mod)(x: int, y: int): int` のような括弧付き keyword 名を value binding として扱える。
- `builtin_op::...` を compiler builtin namespace として lowering し、`poly::Expr::PrimitiveOp` に落とすようにした。
  - primitive kind は旧実装の `PrimitiveOp` に合わせた enum として `poly` に保持する。
  - `builtin_op` の signature は lowering 時の lexical type lookup を通して作るため、`std::text::str::str` などの std 型 path と裸 `str` が混ざらない。
  - `bool` は literal / primitive op が直接使う core builtin 型として登録した。
- 型注釈 / signature で tuple 型を読めるようにし、`(str, int)` を `Pos::Tuple` / `Neg::Tuple` へ lowering するようにした。
- std source の scalar/text primitive 宣言を `builtin_op::...` body へ置き換えた。
  - `int` / `float` / `bool`
  - `str` / `char` / `bytes` / `path`
- `as` tail の型注釈を expression lowering で読み、`AnnConstraintLowerer` へ接続するようにした。
- list literal / list pattern の spread を lowering するようにした。
  - `[..xs, y]` は `xs` を `list item` として制約する。
  - `[x, ..xs]` などの pattern は prefix/spread/suffix を `Pat::List` に落とす。
- local の分割束縛を block 内でも lowering するようにした。
  - `my (x, y) = ...`
  - `my _ = ...`
- `expr with:` block を先に lower し、`our`/`pub` binding だけを tail expression へ見せるようにした。
- effect row annotation / signature の separator 判定を修正した。
  - `,` は row item separator、`;` を含む `Separator` だけを tail separator として扱う。
- string literal の型を裸 `str` ではなく `std::text::str::str` の structured std path にした。
  - `std::core::fmt` の `Display unit` / `Debug unit` などの signature mismatch が消えた。
- `case` / `catch` guard を poly IR に残すようにした。
  - `CaseArm` / `CatchArm` を tuple ではなく struct にし、`guard` を `ExprId` として保持する。
  - guard は bool exact に制約し、pattern local を見えるまま lower する。
  - guarded catch arm は complete coverage として数えない。
- `if` 条件と `case` / `catch` guard に `std::control::junction::junction::junction` を自動挿入するようにした。
  - 条件式そのものではなく `junction(condition)` の結果を bool exact に制約する。
  - `junction` は std の実体に解決済み `RefId` を張り、通常の関数適用として lowering する。
- pattern lowering を広げた。
  - `PatOr` / `PatAs` tail
  - string / plain `RuleLit` literal pattern
  - record pattern と optional record field default
  - polyvariant pattern
- expression lowering に polyvariant head を追加した。
  - `:ok value` を `Expr::PolyVariant("ok", [value])` として lowering する。

## 型推論: stack subtract 置換

- `spec/2026-05-31-effect-variable-subtractable.md` の stack 重み表現に合わせて、旧 `non-subtract` 風の外部制約を型内 `StackWeight` と weighted subtype edge へ寄せた。
  - `Pos::Stack` / `Neg::Stack` の peel で左右の `ConstraintWeights` を更新する。
  - 変数境界は weight 付きで保存し、lower/upper replay でも weight を合成して伝播する。
  - `α @L <: @R [S; β]` は、weight が空でないとき raw row を直接保存せず、row residual gamma を作って `γ @W <: β` へ分解する。
  - row item の stack intersection は effect family path で判定し、payload の詳細は ordinary type constraint 側へ回す。
- `StackWeight` / `Scheme` / instantiate / dump を stack 表現へ追随させた。
  - scheme に stack id を保存し、instantiate 時に type var と同様 fresh 化する。
  - 表示は `stack(...)` と短い row 表示の両方で崩れないようにした。
- compact / generalize を stack 重み前提へ更新した。
  - concrete shape に weight を持たせず、covariant var occurrence に寄せる。
  - subtract fact は「共変側に非空 stack entry が残る id」だけを scheme へ残す。
  - 共起分析に variable sandwich flattening を戻し、正負両方で同じ var に挟まれる残差を代表へ畳むようにした。
  - 負位置の `A & [H; A]` を `A` へ落とす row 正規化案は取り下げた。
  - simplification / sandwich / 量化対象の判定を birth level ではなく current level に直し、extrude で外側へ下がった変数を量化・消去しないようにした。
- `ref::update` の canary が期待形へ戻った。
  - `std::control::var::ref('a & 'c, 'b) -> ('b -> ['c] 'b) -> ['c, 'a] ()`
- impl method receiver の annotation var が child level で新規作成されないよう、impl receiver annotation だけ root level で不足 var を作る経路を分けた。
  - `impl 'a: Box` 内の `x.eq` が `Eq 'a` prerequisite として回収される。
- std 再編に追随して、古い test/build include のうち compile を止めていた箇所を更新した。
  - `crates/yulang-infer/src/source/tests.rs` の `ops.yu` include を `lib/std/core/ops.yu` へ変更。

## 型推論: stack subtract 仕様外候補

`spec/2026-05-31-effect-variable-subtractable.md` と現実装を照合した結果、まだ仕様外に見える箇所がある。
特に `#id` の量化と instantiate の扱いが、現在の `SchemeSubtractFact` 表現に引きずられている。

### `#id` 量化の表現が足りない

spec では、量化時に型変数と同じように `#id` も scheme に保存し、instantiate 時に fresh 化する。
しかし現在は `poly::Scheme.subtracts: Vec<SchemeSubtractFact>` が、次の二つを兼ねている。

- stack id の量化対象
- `S-subtract(α,#id)` 相当の subtract fact

この二つは本来別の情報に見える。
`stack(T, @S)` 内に残る `#id` があるだけなら、その id は fresh 化対象ではあるが、必ずしも `SchemeSubtractFact` として表面に出るとは限らない。
現実装では `instantiate.rs` が `scheme.subtracts` に現れた id だけを fresh 化するため、stack 内にだけ残る id が別 instantiate 間で共有される危険がある。

関係箇所:
- `crates/infer/src/generalize.rs`
  - `GeneralizedCompactRoot.subtracts: Vec<(TypeVar, SubtractId)>`
  - `prepare_quantified_subtracts`
  - `finalize_scheme_subtracts`
- `crates/poly/src/types.rs`
  - `Scheme.subtracts`
  - `SchemeSubtractFact`
- `crates/infer/src/instantiate.rs`
  - `for fact in &scheme.subtracts { self.fresh_subtract(fact.id); }`

次に決めること:

- `Scheme` に stack id quantifier を別フィールドとして持たせるか。
- それとも `SchemeSubtractFact` を「stack id quantifier + optional fact」のような表現へ変えるか。
- instantiate は `scheme.predicate` / `role_predicates` / `recursive_bounds` に含まれる `StackWeight` も走査して fresh id を作るべきか。

### `Subtractability::Empty` を live stack entry と見ていない

spec では `pop(0)[Empty]` は「何も引いてはならない」寿命境界として導入される。
また、instantiate 後の共変側に `pop(p)[H1, ..., Hn]` で `n > 0` の entry が残るなら、述部側に同じ id の `pop(u32::MAX)[]` を付ける、と書いている。

現在の generalize cleanup は `Subtractability::Empty` を live stack entry として扱っていない。

関係箇所:
- `crates/infer/src/generalize.rs`
  - `collect_live_stack_ids_in_type`
  - `stack_item_keeps_subtract_id_live`

現在の実装:

```rust
fn stack_item_keeps_subtract_id_live(subtractability: &Subtractability) -> bool {
    !matches!(subtractability, Subtractability::Empty)
}
```

これは `Empty` を「空の stack entry」と誤読している可能性がある。
`Empty` は effect 集合として空であって、stack entry が存在しないという意味ではない。

ただし、ここだけを `true` に変えると、上の `#id` 量化/fresh 化問題にぶつかる。
先に scheme の stack id 表現を決める必要がある。

### instantiate が述部側 `pop(u32::MAX)` を付けていない

spec では、量化値の scheme に live な共変 stack entry が残る場合、instantiate 後の述部側へ `pop(u32::MAX)[]` を付与する。
現実装は逆に、predicate root を `pop(u32::MAX)` で包まないとコメントしている。

関係箇所:
- `crates/infer/src/instantiate.rs`
  - `instantiate_scheme_with_roles`

現在のコメント:

```rust
// scheme 内の live な共変 stack entry はそのまま use-site へ持ち込む。
// predicate root を pop(u32::MAX) で包むと、分解時に重みが全位置へ配られ、
// scheme 自身の push (AllExcept 等) を合成で消してしまう。
```

これは spec と直接衝突しているように見える。
一方で lowering 側では、定義済み関数の述部へ `pop` を入れている箇所がある。
そのため、実装が「scheme instantiate ではなく lowering 時点で述部 pop を作る」設計へ寄っている可能性もある。
もしそうなら spec 側にその設計を書き直す必要がある。

次に決めること:

- `pop(u32::MAX)` は instantiate 時に付与するのか。
- 定義の lowering/generalize 時点で述部側に保存するのか。
- 両方を避けて、scheme 内に「use-site で付ける pop」を明示的に持つのか。

### `declared_subtracts` filter が spec にない

generalize では、scheme に保持する subtract id を `machine.subtract_declared(id)` で絞っている。
コメント上は「注釈・データ宣言が直接導入した id だけ保持し、instantiate clone で再登録された fact は消す」という意図。

関係箇所:
- `crates/infer/src/constraints.rs`
  - `declared_subtracts`
  - `declared_subtract_fact`
  - `subtract_declared`
- `crates/infer/src/generalize.rs`
  - `prepare_quantified_subtracts`

これは現在の `SchemeSubtractFact` が stack id quantifier と subtract fact を兼ねているための実装上の都合に見える。
spec の「`#id` も量化する」とは別の基準なので、必要なら仕様化が必要。
ただし、scheme の stack id 表現を分ければ、この filter 自体を単純化できる可能性がある。

### non-generic subtract id の dead pruning 例外

`prepare_quantified_subtracts` は、dead と判定した id でも `non_generic_subtract_ids` に含まれるものは weight から剥がさない。

関係箇所:
- `crates/infer/src/generalize.rs`
  - `prepare_quantified_subtracts`
  - `non_generic_subtract_ids`

cleanup 規則には「non-generic なら消さない」という例外は書かれていない。
これが外側の未量化変数に属する stack id を守るための正しい境界処理なら、仕様へ書く必要がある。
そうでなければ、古い `subtract fact` 表現を守るための保護機構になっている可能性がある。

### row residual gamma の再利用

`α @L <: @R [S; β]` の分解で作る残差変数 `γ` は、`RowResidualKey` で同じ形の分解結果を再利用している。

関係箇所:
- `crates/infer/src/constraints.rs`
  - `row_residuals: FxHashMap<RowResidualKey, TypeVar>`
  - `add_effect_row_upper_bound`

これは無限増殖や同じ分解の重複を避ける実装上の正規化で、現在は仕様にも追記済み。
key は `source/tail/retained row/weight` ではなく、`source/U family/@W'` として見る。
`β` は残差 slot の行き先なので key に含めず、別の `β` へ流す場合は同じ `γ` に上界を追加する。

- 同一 machine 内で、同じ source/U family/@W' の row residual は同じ `γ` を再利用する。
- これは制約の意味を変えず、同一分解の重複生成を避けるための hash-cons である。

### post-subtract cleanup / post-strip について

一度は `prepare_quantified_subtracts` 後に再度 compact simplification を走らせる処理が必要に見えた。
しかし見直すと、原因は `#[cfg(test)] generalize_compact_root` が実 entrypoint と違い、pre-simplify なしで private helper に入っていたことだった。

対応:

- test helper を実 entrypoint と同じく `generalize_prepared_compact_root_with_roles` 経由にした。
- `post_subtract_cleanup` / `post_strip` の追加再簡約は不要として削除した。
- spec への追記は不要だったので戻した。

これは「仕様外候補」ではなく、今回の確認で取り下げたもの。

## 型推論: stack id 量化の実装

`#id` の量化を `SchemeSubtractFact` から分離し、`poly::Scheme.stack_quantifiers` として保持する形にした。
これにより、`stack(T, @S)` にだけ現れる `#id` も、subtract fact の有無に関係なく instantiate で fresh 化できる。

実装したこと:
- `GeneralizedCompactRoot.stack_quantifiers` と `Scheme.stack_quantifiers` を追加。
- generalize の cleanup 後に、量化対象の型変数上にある共変位置の非空 stack entry を持つ id だけを `stack_quantifiers` として集計。
- `Empty` は「空のeffect集合」であって「stack entryなし」ではないので、`pop(0)[Empty]` も live stack entry として扱う。
- `stack_quantifiers` に含まれる id は、`SchemeSubtractFact` に出ていなくても instantiate で fresh 化する。
- `stack_quantifiers` に含まれない id は lazy fresh しない。型変数と同じく、scheme が量化した id だけを fresh 化対象にする。
- instantiate 後、量化値の predicate root に対応する fresh id の `pop(u32::MAX)[]` を付与する。
- ancestor finalize の経路でも、親の代入・sandwich・cleanup 後に `stack_quantifiers` を再集計する。

仕様側にも、`α @L <: @R [S; β]` の分解で同じ `source/U family/@W'` から作る残差 `γ` は同一 machine 内で再利用すると追記した。
これは新しい型規則ではなく、同じ残差 slot を何度も増やさないための hash-cons として位置づける。

- `cargo test -q -p infer`
- `cargo test -q -p poly format`

## 型推論: self skeleton と residual floor の再確認

- `spec/2026-05-31-effect-variable-subtractable.md` の「本体の型集めを始める前に、関数自身の型変数を下から抑える」規則に合わせた。
  - top-level / local の `my f(x: ...) = ...` は、引数注釈と述部 stack をまとめた骨格型を先に作り、`skeleton <: f` を登録する。
  - 本体を lower した後、本体 value/effect を骨格末端の value/effect へ流し、result annotation はその骨格末端へ接続する。
  - 双方向の等式制約は入れていない。
- `@W - U` は、stack だけでなく floor にも反映する。
  - pop だけの entry から `U` を引いた場合は、`floor[AllExcept(U)]` を残す。
  - stack/floor が残る entry でも、`U` を引いた後に残る共通部分を floor に保存する。
    これにより、stack 要素が後続の `pop` で消えても、消費済み budget が `All` へ戻らない。
  - これにより、`pop(u32::MAX)[]` を通った後でも「すでに `U` を引いた」という情報が残る。
- row residual は tail-free に共有する。
  - key は `source` / retained effect family path set / residual weight。
  - payload は family 判定に使わず、通常の subtype 制約へ任せる。

## 型推論: 注釈付き local effect view

`my loop(x: [io] _) = catch io::out(_), k -> nondet::list(loop(k()))` の形で、
`'a [io; β&[nondet; β]] -> [β] list 'a` のように外側の residual から `nondet` が引けてしまう問題を確認した。

原因:
- 注釈付き引数 `x: [handled] _` の local effect を共有 outer 変数として保存していた。
- `catch x` が `outer <: [handled; ρ]` を raw row upper として outer に溜める。
- その outer が自己再帰や別 handler の引数 effect として見えるため、catch の内部用 row upper が関数境界へ漏れた。
- さらに、消費済み stack 要素が `pop` で消えると、残差が `All` 相当に戻り、別 effect を引ける経路が生まれた。

対応:
- `LocalEffect::Stack { inner, weight }` を導入し、注釈付き local は共有 outer 変数ではなく stack view として持つ。
- local 名参照の通常 effect slot は `inner` を返し、`catch` の scrutinee 制約だけが view から `stack(inner, weight) <: [handled; ρ]` を張る。
- defined function の self skeleton は、引数 effect として outer ではなく inner slot を使う。
- `@W - U` 後の stack entry には、残り budget の共通部分を floor として保存する。
- dump では scheme 量化由来の `pop(u32::MAX)` だけを持つ stack wrapper を短く表示し、`ref::update` のような hygiene stack を surface 型へ漏らさない。
