# 2026-06-25

## Control VM handler hygiene

`tests/yulang/yulang-adversarial-corpus/02_repeated_callback_hygiene.yu` exposed
a control-vm/interpreter mismatch:

```text
control-vm:  run roots [0]
interpreter: run roots [6]
```

The mono output already carried the callback adapter evidence:

```text
hygiene[add_id[0, tick, own]]
```

So the failure was in runtime guard transport, not in inference or
specialization.

The root cause was in `crates/control-vm/src/runtime/flow.rs`.
When the VM created a carried guard for an adapter marker, it treated every
handler frame outside the marker entry as already exposed. In nested same-path
handlers this exposed the inner `strip_tick` handler too early, so it was
allowed to capture a callback-origin `tick::ping`.

The interpreter already used the narrower rule: expose a matching handler alias
only when an earlier guard at the marker entry proves that alias is visible.
The VM now mirrors that rule and only exposes the outermost matching handler in
that alias case.

Added regression coverage:

```text

```

## CLI control VM stack gate

Reproduced the `for` / ref-update stack overflow noted on 2026-06-24 with a
minimal CLI workload:

```yu
{
    my $x = 0
    for i in 0..:
        if i == 100: last
        &x = $x + 1
    $x
}
```

Before the fix, the release CLI aborted with:

```text
thread 'main' has overflowed its stack
fatal runtime error: stack overflow, aborting
```

The source-level runtime tests already run selected control VM cases on a
larger helper stack. The CLI path did not: normal `run`, `run-control`,
`run-control-std`, and artifact execution called the control VM on the process
main stack. The control VM still has recursive eval/apply paths, so this was a
CLI stack policy gap rather than a new type/effect regression.

Added a CLI-only control runner that executes the control VM and root
formatting on a dedicated 64MiB stack, then returns only printable output,
errors, stats, and timing data across the thread boundary. Runtime values stay
inside the VM thread; this avoids making `control_vm::Value` cross-thread.

This is not the final trampoline fix. It makes CLI/release behavior match the
known runtime limitation and keeps the remaining runtime rewrite explicit.

## Trampoline experiment and explicit callback contracts

Measured the current CLI stack workaround before attempting a real trampoline.

```text
continuation_frames_cloned: 100409
marker_scope_frame_touches: 436413
```

A temporary call-depth probe found `rust_max_call_depth=1903` for the ref loop
and `176` for showcase. The probe itself added measurable overhead, so it was
not kept.

Then tried a local trampoline shape that collected nested `Value::Marked`
callee layers and closed them in reverse after applying the inner callee once.
That passed `control-vm` unit tests, but broke the parser combinator smoke:

- `run_control_source_text_with_embedded_std_choice_recovers_from_parse_fail`
- `run_control_source_text_with_embedded_std_repeats_parser_until_delimiter`
- `run_control_source_text_with_embedded_std_repeats_parser_until_eof`

The failure was semantic, not just a missing frame pop. Parser backtracking
depends on the exact order in which marker frames are opened, closed, and
carried across requests. Batching marked layers changes which handler boundary
is visible to `parse::fail`. Do not retry this as the trampoline shape.

After reverting the experiment, the same parser smoke still failed on clean
HEAD. The underlying regression was that explicit argument-effect contracts
were carried as ordinary own-path guards. For `std.text.parse.choice`,
`p: () -> [parse] _` should allow the `choice` body's `catch p()` to catch
`parse::fail`; instead the carried guard blocked that matching handler and let
the failure escape to `run_str`.

The runtime now treats `PreserveMatchingPath` contract markers as explicit
callback contracts: when they carry a request, all matching handlers visible at
the marker entry are exposed to that carried guard. Ordinary own-path guards do
not get this exposure. This lets `choice` catch `fail`, while operations that
`choice` does not handle, such as `snapshot` and `item`, still flow to outer
handlers like `run_str`.

## `act copy` source visibility

Stabilized the source-level `act copy = source with:` visibility rule:

- A source-level copy inherits only the source body's `pub` / `our` surface.
- Source `my` operation signatures and companion members are not registered in
  the destination companion.
- The destination `with` body is still a direct body of the destination act, so
  its own `my` members remain available to its own declarations.
- Compiler-internal synthetic copies keep their implementation-copy behavior.
  This is needed for generated `var` / label-sub acts, where private helper
  methods such as `std.control.var.var.var_ref` and `run` are implementation
  details consumed by lowering rather than source-level exports.

The original leak was that `materialize_act_copy` registered the source body
verbatim into the destination companion, and `lower_act_body_contents` then
lowered the same source declarations verbatim. A `with` body could therefore
resolve a source-private name such as `hidden` as `copy.hidden`.

The fix adds an explicit copied-source mode to both module-map registration and
act body lowering. Source-level copies use an export-only mode; synthetic
compiler copies use an internal mode. The operation signature collection follows
the same mode, so private source operations do not remain visible in the copied
effect family.

Added `spec/2026-06-25-act-copy.md` and linked it from `spec/README.md`.

## Compiled-unit typed surface

Added the next compiled-unit layer for exported schemes. This still does not
change the normal compile path; it only makes `.yuunit` artifacts carry the
typed information that a future cache hit will need.

Implemented:

- `infer::CompiledTypedSurface`, mapping namespace value symbols to
  `poly::types::Scheme`.
- `infer::CompiledTypeArena`, a serialized arena form that stores only
  `pos` / `neg` / `neu` node vectors and avoids serializing hash-cons maps.
- `infer::CompiledTypedIndex` for symbol-to-scheme lookup, plus
  `CompiledTypeArena::to_type_arena` for process-local arena reconstruction.
- `CompiledTypeImporter`, which remaps serialized `PosId` / `NegId` /
  `NeuId`, type variables, and stack subtract IDs into a live `infer::Arena`.
- `CompiledTypedSurface::import_value_defs`, which allocates fresh bodyless
  `poly::Def::Let` entries carrying imported schemes for exported value
  symbols.
- Typed surface extraction from `BodyLowering`, using the live `ModuleTable`
  only to map namespace symbol/order back to the source `DefId`.
- `.yuunit` envelope support for typed surface data, plus `typed_hash` and a
  compiled-unit format bump.
- Deterministic typed-surface hashing over the serialized type node vectors
  and scheme contents.

## Compiled-unit runtime surface

Before wiring cached dependency units into the normal lowerer, checked what the
typed import view is missing. Value schemes are not enough for execution:
dependency bodies, `poly::Arena::effect_operations`, constructors, field
projection metadata, runtime roots, and resolved refs/selects all live in the
lowered poly graph. Connecting only bodyless imported `Def::Let`s would make
name lookup appear to work while leaving runtime/specialization metadata behind.

Added:

- `infer::CompiledRuntimeSurface`, currently the lowered per-unit
  `poly::Arena` plus `DumpLabels`.
- `.yuunit` envelope support for the runtime surface, with compiled-unit format
  bumped to 4.
- Cache round-trip coverage that checks the runtime surface's def graph and
  labels survive serialization.

Added `runtime_hash` to the `.yuunit` manifest. The hash walks the lowered
`poly::Arena` structure directly instead of hashing serialized bytes, and sorts
HashMap/HashSet-backed metadata before hashing. Labels are intentionally kept
out of this semantic runtime hash; they still round-trip as part of the runtime
surface.

The next clean step is a remap/merge path that can bring cached dependency
bodies into a live arena.

Also generalized `CompiledTypeImporter` so it can import into both

## Parser choice effect contract sharing

Fixed the `std.text.parse.choice` public signature regression where the two
parser callback arguments and the result exposed different `parse` effect
variables unless the implementation carried extra annotations.

The root issue was that closed effect rows in callback annotations were lowered
as separate stack-inner variables each time they appeared. For `choice`, the
same annotation row:

```text
[parse 'item 'err 'pos 'snap]
```

appears in both callback arguments and in the result effect. Those appearances
are the same annotation contract, so the lowerer now interns closed annotation
effect rows within the annotation scope and reuses the same `TypeVar`.

The fix is deliberately limited to closed rows. Tail rows such as
`[loop; 'e]` still use the existing projection/erasure path, so canaries like
`for_in` and the small callback-tail tests keep their residual-only public
surface.

Specialization also needed a runtime-side fix. Once the public type is shared,
the callback argument can become boundary-equivalent to the expected type, but
the explicit annotation contract still has to install function-adapter hygiene
markers. `specialize2` now reads the expected callee argument's
`ArgEffectContract` from the call spine and keeps a function boundary even when
the value types are equivalent if that boundary carries contract markers.

During the fix, a visibility-based branch (`pub`/`my`) was tried and removed.
That branch was too ad hoc: visibility is not the reason the type should
change. The remaining rule is based on annotation row structure and call-site
contract metadata instead.

Regression coverage:

- `dump_poly_std_parse_choice_public_signature_hides_stack_evidence`
- `run_control_source_text_with_embedded_std_choice_recovers_from_parse_fail`
- `dump_poly_std_flow_for_in_public_signature_hides_stack_evidence`
- `dump_poly_std_ref_update_public_signature_hides_stack_evidence`

## Source/lowered signature bridge for cached prefixes

Started separating lowering-time signatures from CST ownership. Constructor
payload annotations and act operation signatures now use `StoredSignature`,
which can be either:

- `Source(Cst)` for the normal source-lowering path;
- `Lowered(SignatureType)` for cached/imported prefix metadata.

This keeps the existing source behavior intact while giving the compiled-unit
cache path a place to import already-lowered signatures without reparsing cached
source. The lowering paths that consume these signatures now accept both forms:

- constructor expression and constructor pattern payload wiring;
- error variant payload helpers;
- act operation value references;
- catch operation payload/continuation contracts.

The catch operation path was moved from AnnType-only constraint lowering to
`SignatureType`-based value connection, so source and lowered signatures share
the same constraint-generation route after parsing.

Added a cache-prefix canary that clones a compiled dependency's `ModuleTable`,
replaces the test constructor and operation annotations with lowered
`SignatureType` values, imports the runtime surface into a fresh prefix, and
then lowers a root file using the imported constructor and handled operation.
This exercises the lowered-signature path without pretending that the full
serializable lowering surface is done.

Still not done:

- `act_template` / `act_copy` bodies still require source CST when a downstream

## Lowered role method signatures for cached prefixes

## Compiled lowering surface skeleton

Added `CompiledLoweringSurface` as the fourth compiled-unit surface, between
namespace and typed/runtime surfaces. The first exported metadata is the act
type-variable table:

```text
type_symbol + type_path -> ['a, 'b, ...]
```

This intentionally does not try to serialize the whole `ModuleTable`.
Namespace/typed/runtime surfaces already cover names, schemes, and executable
poly bodies. The lowering surface is for the remaining declaration metadata
that downstream source lowering needs but the final poly arena does not carry.

The CLI `.yuunit` artifact now stores:

- syntax surface
- namespace surface
- lowering surface
- typed surface
- runtime surface

The compiled-unit format was bumped from 5 to 6, and the manifest now records a
`lowering_hash`. The round-trip cache test fixture includes a parameterized act
and asserts that its `act_type_vars` entry survives serialization.

`CompiledLoweringSurface::apply_to_module_table` can now restore these act
type-variable entries to a live `ModuleTable`. This is not the final cached
dependency import yet, but it pins the direction: lowering metadata is imported
as structured declarations, not recovered by reparsing source templates.

This is still only a skeleton. The next useful contents are constructor payload
signatures, act operation signatures, and role method signatures in lowered
form. Act copy body templates remain a separate problem because they are not
just metadata; they are source-body materialization rules.

## Compiled-unit prefix plus source suffix

Added the next lowerer seam for dependency cache hits: a compiled-unit prefix
can now be combined with a freshly parsed source suffix that includes the root
file and local `mod` children.

New entrypoints:

- `infer::lowering::lower_loaded_files_with_prefix`
- `yulang::build_poly_from_compiled_unit_prefix_and_collected_sources`

The `sources` crate already had `load_suffix_with_syntax_prefix`; this change
adds the matching body-lowering side. The prefix provides namespace/lowering/

This is still not wired into normal CLI cache policy. It is the structural
