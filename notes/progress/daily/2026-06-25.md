# 2026-06-25

## Control VM handler hygiene

`tests/yulang/yulang-adversarial-corpus/02_repeated_callback_hygiene.yu` exposed
a control-vm/interpreter mismatch:

```text
control-vm:  run roots [0]
interpreter: run roots [6]
```

The mono output already carried the callback adapter evidence:

```text
hygiene[add_id[0, tick, own]]
```

So the failure was in runtime guard transport, not in inference or
specialization.

The root cause was in `crates/control-vm/src/runtime/flow.rs`.
When the VM created a carried guard for an adapter marker, it treated every
handler frame outside the marker entry as already exposed. In nested same-path
handlers this exposed the inner `strip_tick` handler too early, so it was
allowed to capture a callback-origin `tick::ping`.

The interpreter already used the narrower rule: expose a matching handler alias
only when an earlier guard at the marker entry proves that alias is visible.
The VM now mirrors that rule and only exposes the outermost matching handler in
that alias case.

Added regression coverage:

```text
cargo test -q -p yulang run_control_source_text_with_embedded_std_keeps_repeated_callback_hygiene -- --test-threads=1
```

Manual checks after the fix:

```text
./target/release/yulang run --no-cache --print-roots tests/yulang/yulang-adversarial-corpus/02_repeated_callback_hygiene.yu
./target/release/yulang run --no-cache --print-roots --interpreter tests/yulang/yulang-adversarial-corpus/02_repeated_callback_hygiene.yu
tests/yulang/yulang-adversarial-corpus/probe.sh
```

Both runtimes now return `run roots [6]`, and the adversarial probe passes.

## CLI control VM stack gate

Reproduced the `for` / ref-update stack overflow noted on 2026-06-24 with a
minimal CLI workload:

```yu
{
    my $x = 0
    for i in 0..:
        if i == 100: last
        &x = $x + 1
    $x
}
```

Before the fix, the release CLI aborted with:

```text
thread 'main' has overflowed its stack
fatal runtime error: stack overflow, aborting
```

The source-level runtime tests already run selected control VM cases on a
larger helper stack. The CLI path did not: normal `run`, `run-control`,
`run-control-std`, and artifact execution called the control VM on the process
main stack. The control VM still has recursive eval/apply paths, so this was a
CLI stack policy gap rather than a new type/effect regression.

Added a CLI-only control runner that executes the control VM and root
formatting on a dedicated 64MiB stack, then returns only printable output,
errors, stats, and timing data across the thread boundary. Runtime values stay
inside the VM thread; this avoids making `control_vm::Value` cross-thread.

This is not the final trampoline fix. It makes CLI/release behavior match the
known runtime limitation and keeps the remaining runtime rewrite explicit.

Regression coverage:

- `public_regression_ref_update_loop_runs_on_cli_vm_stack`
- `scripts/release-smoke.sh` now runs the same 100-iteration ref-update loop.
- `scripts/hardening-smoke.sh` now calls `scripts/release-smoke.sh` by default.

## Trampoline experiment and explicit callback contracts

Measured the current CLI stack workaround before attempting a real trampoline.
The 100-iteration ref-update loop showed:

```text
runtime_execute: 190.4ms / 181.9ms / 180.2ms
marker_scope_max_depth: 709
request_resume_steps: 3904
continuation_frames_cloned: 134524
marker_scope_frame_touches: 1294750
```

`examples/showcase.yu` showed:

```text
runtime_execute: 53.7ms / 55.3ms / 54.4ms
marker_scope_max_depth: 54
request_resume_steps: 13530
continuation_frames_cloned: 100409
marker_scope_frame_touches: 436413
```

A temporary call-depth probe found `rust_max_call_depth=1903` for the ref loop
and `176` for showcase. The probe itself added measurable overhead, so it was
not kept.

Then tried a local trampoline shape that collected nested `Value::Marked`
callee layers and closed them in reverse after applying the inner callee once.
That passed `control-vm` unit tests, but broke the parser combinator smoke:

- `run_control_source_text_with_embedded_std_choice_recovers_from_parse_fail`
- `run_control_source_text_with_embedded_std_repeats_parser_until_delimiter`
- `run_control_source_text_with_embedded_std_repeats_parser_until_eof`

The failure was semantic, not just a missing frame pop. Parser backtracking
depends on the exact order in which marker frames are opened, closed, and
carried across requests. Batching marked layers changes which handler boundary
is visible to `parse::fail`. Do not retry this as the trampoline shape.

After reverting the experiment, the same parser smoke still failed on clean
HEAD. The underlying regression was that explicit argument-effect contracts
were carried as ordinary own-path guards. For `std.text.parse.choice`,
`p: () -> [parse] _` should allow the `choice` body's `catch p()` to catch
`parse::fail`; instead the carried guard blocked that matching handler and let
the failure escape to `run_str`.

The runtime now treats `PreserveMatchingPath` contract markers as explicit
callback contracts: when they carry a request, all matching handlers visible at
the marker entry are exposed to that carried guard. Ordinary own-path guards do
not get this exposure. This lets `choice` catch `fail`, while operations that
`choice` does not handle, such as `snapshot` and `item`, still flow to outer
handlers like `run_str`.

Updated the design note:

```text
notes/design/2026-06-24-explicit-effect-contract-metadata.md
```

## `act copy` source visibility

Stabilized the source-level `act copy = source with:` visibility rule:

- A source-level copy inherits only the source body's `pub` / `our` surface.
- Source `my` operation signatures and companion members are not registered in
  the destination companion.
- The destination `with` body is still a direct body of the destination act, so
  its own `my` members remain available to its own declarations.
- Compiler-internal synthetic copies keep their implementation-copy behavior.
  This is needed for generated `var` / label-sub acts, where private helper
  methods such as `std.control.var.var.var_ref` and `run` are implementation
  details consumed by lowering rather than source-level exports.

The original leak was that `materialize_act_copy` registered the source body
verbatim into the destination companion, and `lower_act_body_contents` then
lowered the same source declarations verbatim. A `with` body could therefore
resolve a source-private name such as `hidden` as `copy.hidden`.

The fix adds an explicit copied-source mode to both module-map registration and
act body lowering. Source-level copies use an export-only mode; synthetic
compiler copies use an internal mode. The operation signature collection follows
the same mode, so private source operations do not remain visible in the copied
effect family.

Added `spec/2026-06-25-act-copy.md` and linked it from `spec/README.md`.

Validation:

```text

cargo run -q -p yulang -- --no-cache check-poly-std /tmp/yulang-act-copy-private.yu
```

The CLI reproduction now reports unresolved `hidden` in `copy.visible` and
`copy.leak` instead of materializing `copy.hidden`.

## Web docs implementation sync

Updated `web/docs` to track the current implementation instead of the older
standard-library namespace:

- Added source-level `act copy = source with:` visibility notes to the English
  and Japanese effect/module references.
- Documented that `act copy` creates a distinct effect family and inherits only
  the source `pub` / `our` surface; source `my` members do not become visible in
  the destination companion or `with:` body.
- Replaced stale module paths such as `std::list`, `std::range`, `std::ops`,
  `std::undet`, `std::console`, and `std::fs` with the current paths:
  `std::data::*`, `std::core::ops`, `std::control::*`, and `std::io::*`.
- Updated VitePress sidebar labels to show the current standard-library module
  paths while keeping the existing page URLs stable.
- Updated filesystem docs from the old `fs` / `fs_err` surface to the current
  `file` effect and `io_err` error.
- Renamed remaining tutorial-only `fs_err` examples to `path_err`, so no docs
  example looks like the old standard filesystem API.
- Updated console docs away from removed `print_native` / `println_native`
  names toward the current `out` / `err` / `warn` / `die` effect families and
  wrappers.
- Normalized source-facing effect row syntax: concrete effect heads use commas
  (`[file, io_err]`), while row tails use semicolon plus a quoted type variable
  (`[file, io_err; 'e]`).
