## Root cause

今回のハングの直接原因は、`ref_update` そのものではなく、effect subtraction の private evidence が
public scheme へ漏れて、bounds replay の static boundary が unbounded cycle になったこと。

具体的には、`std.control.var.ref.update` は data field に

```text
update_effect: () -> [ref_update a; e] ()
```

のような effectful function を持つ。旧 lowering では、この field の latent return-effect tail に
public な `e` 自体を使い、さらに `e` へ `AllExcept(ref_update ...)` の subtract fact を登録していた。
そのため synthetic getter / receiver / local helper / callback を一周したとき、public scheme 内に

```text
PWeight(pop(1); take(AllExcept(ref_update)), e) <: e
```

のような weighted self-alias が入った。

最小因子は次の三つだった。

- data-position の effectful function: `r.update_effect()`
- local wildcard handler: `my loop(x: [_] _) = catch x: ...`
- handler arm 内 callback: `ref_update::update v, k -> loop:k:f v`

この形では、lexical boundary 内では balanced だった `pop(s); take_s(...)` が、receiver の反変位置や
var-var alias replay を通って `right-pop(s)`, `right-pop(s)^2`, ... のように増殖しうる。
`pop(n) -> pop(1)` clamp を外すと、exact replay key では候補が無限に増えて solver が止まらなくなる。

6/22 の一時 guard は主に var-var direct replay の left-pop-only cycle を抑えるものだった。
今回の増殖は right-pop / ordinary bounds replay 側にも出るため、その guard だけでは止まらない。

## Implemented fix

採った方針は、`ref_update` API を捨てることではなく、data-position に埋め込まれた関数の
返り effect tail を private / projected にすること。

- Type method の戻り型注釈を、body lowering 中の raw body effect へ直接つながないようにした。
  result annotation は receiver と args を含む method scheme の結果位置に対する deferred check として扱う。
- data-position の effectful function tail は、public tail `e` ではなく private tail `κ` で lower する。
  `[ref_update a; e]` は field 内部では概念的に `[ref_update a; PWeight(s, κ)]` になり、
  `declared_subtract_fact` は `κ` 側にだけ置く。
- field row に bare public tail を混ぜる案は採らなかった。これは runtime で
  `std.control.var.ref_update.update` が unhandled effect として漏れ、list update が壊れた。
- private tail だけにして public `e` へ戻さない案も採らなかった。これは ordinary residual が戻らず、
  list update の候補衝突を起こした。
- 最終形は、field row には `κ` だけを置き、ordinary row 内容だけを `κ <: e` の片方向 projection で
  public tail へ戻す形。これは solver graph 上の双方向 equality ではなく、private evidence を閉じるための
  projection として扱う。
- Generalize の final cleanup で、構造出現と declared fact が `Empty` だけの stack id は public scheme から
  落とすようにした。`All` / `AllExcept` / `Set` を持つ id はこの cleanup の対象外。

この結果、`std.ref_update` は public 表示で次のように落ちる。

```text
std::control::var::ref('a & 'c, 'b) -> ('b -> ['c] 'b) -> ['c, 'a] ()
```

`#...` や `AllExcept(ref_update ...)` は public scheme に出ない。

## Replay stabilization

Pro 案のうち、replay 側も最後まで入れた。

- `ConstraintWeights::compose_for_replay()` は W-Mix 後の exact label を保つ。
  `pop(n) -> pop(1)` のような意味論的 clamp は入れていない。
- var-var replay を専用 direct path から通常の `SubtypeConstraint { lower, upper, weights }`
  queue へ戻した。これで ordinary replay と var-var replay の dedup 意味論を分けない。
- ただし exact queue key だけでは、act method / parser choice の same-boundary alias cycle が
  bounds table の保存時に増殖した。そのため、bounds table への保存直前だけ、
  同一 endpoint / 同一 subtract id / 同一 family の alias replay cycle を既存 bound で subsume する。
- この subsumption は surface type の等式ではない。同じ static boundary を worklist cycle で再訪したときに、
  `pop^n` / `take^n` だけを増やす bound を保存しないための停止性規則である。
  別 id / 別 family は潰さない。

追加した主な canary:

- `ref.update` が public scheme に private stack evidence を出さないこと。
- nested data function でも同じ private projection が効くこと。
- `std::control::flow::for_in` の callback annotation が止まること。
- `std::text::parse.choice` の callback annotation が止まること。
- push-only / pop-only / balanced alias replay cycle が有限で、別 boundary は区別されること。

途中で起きた receiverless role impl の回帰も直した。
receiverless impl method は requirement value upper を接続する必要があるが、receiver method では同じ接続を
入れると mono 側で `UnsupportedBoundary` になったため、`connect_impl_method_requirement` に
`connect_value_upper` を渡す形に分けた。
