## Evidence VM known-state operation route proof D1

- Implemented the plan-only `KnownStateOperationRouteProof` slice from the
  follow-up consultation:
  - `poly::SyntheticVarEffect` now carries optional compiler operation defs for
    generated local-var `get` / `set`.
  - compiled-runtime import remaps those defs when a cached/imported arena is
    loaded.
  - lowering registers the generated synthetic-var operation defs for the
    compiler local ref effect.
  - specialize emits `known_state_accesses` from compiler-certified operation
    call sites.
  - Evidence VM joins those access facts with `KnownHandlerPlan::State` and
    emits per-control-call `KnownStateOperationRouteProof` objects.
- D1 remains plan-only:
  - route-proven local ref operations still keep `GenericFallback` execution.
  - `KnownOperationPlan::direct_ready` remains `false`.
  - rejects move from `no-candidate-handler` / `no-known-state-access-proof` to
    `direct-execution-disabled` once a route proof exists.
- Important boundary:
  - `RuntimeEvidenceSurface` currently carries poly expr ids, while Evidence VM
    uses control expr ids.
  - Therefore D1 uses compiler-issued `synthetic_var + operation role + operation
    def existence` as the stable bridge and emits per-control-call proofs.
  - A later D2/D3 implementation should carry operation origin/def into the
    control-side operation plan before enabling request-free direct execution.
