## Evidence VM catch body env-preserving proof

- Added a static `env_preserving_exprs` table to `RuntimeEvidenceRunner`.
  - The table proves only this local property: evaluating an expression does not
    extend the `Env` value passed by its caller.
  - `Stmt::Let` and `Stmt::Module` remain non-preserving.
  - `Catch` and `RefSet` are preserving because they isolate their mutable body /
    reference work behind their own env clones.
  - `Apply` is preserving when the callee expression is preserving; non-immediate
    args are already evaluated in an internal cloned env.
- `eval_catch` now reuses the caller env for catch bodies whose table entry is
  preserving, and keeps the old cloned-body path otherwise.
- Added counters:
  - `catch_body_env_clone_elided`
  - `catch_body_env_clone_kept`

- Captured the current filesystem API direction:
  - `file::open` / `open_with` as file session entrypoints.
  - `.meta`, `.text`, `.bytes`, and `.raw` as session views.

## Parser recovery / public diagnostics floor

- Context:
  - While expanding diagnostics canonical fixtures, `check` on an unclosed
    parenthesis (`my x = (1`) exposed a CLI panic in module parsing.
- Root cause:
  - `DelimitedListMachine::parse_delimited_list` propagated `None` on EOF with
    `?`, leaving the opened delimiter node unfinished before the module parser
    closed the root. Rowan then saw two top-level children at `finish`.
- Implementation summary:
  - Treat EOF inside a delimited list as a recovered missing close delimiter by
    emitting an empty `InvalidToken` and returning `TriviaInfo::None`.
  - Added a parser module regression for unclosed paren recovery.
  - Added a public CLI diagnostics regression that `check` completes without
    panicking on the same malformed source.
- Semantics:
  - Parser recovery hardening only. This does not yet surface a rich parser
    diagnostic; it prevents the public CLI from panicking and leaves the next
    diagnostics slice to connect recovery nodes to user-facing messages.
