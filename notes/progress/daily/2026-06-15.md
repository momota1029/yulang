# 2026-06-15

## specialize solver rewrite decision

- `specialize` の現行 `ConstraintGraph` は、node / bound / edge を持っているが、解決段階が
  slot-local な lazy resolver になっている。
  - `TypeResolver::slot_solution` がその slot の lower / upper を読みに来た時だけ畳む。
  - 未処理の構造制約を worklist で飽和させてから substitution を作る設計になっていない。
  - 循環に当たったときに `TypeSlotKind::default_type()` を返す経路があり、graph を解いた結果ではない
    default が混ざりうる。
- これは `spec/2026-06-07-principal-monomorphization.md` の「制約グラフを解く」という設計意図に反する。
- spec の `制約グラフ` 節へ、subtype constraint worklist を空になるまで処理し、
  飽和済み graph から substitution table を読むことを追記した。
- この worklist は monomorphize demand 内の制約飽和であり、SCC は解決単位ではない。
- 実装したこと:
  1. `ConstraintGraph` に pending subtype constraint worklist を持たせた。
  2. `constrain_subtype` を、即時再帰ではなく constraint を worklist へ積む入口にした。
  3. bound / edge 追加で生まれる派生制約も worklist へ積み、role demand 解決前に飽和させるようにした。
  4. 循環 slot を `unit` / empty effect row へ default する経路を消した。
  5. call-site variant narrowing と case reachable arm から、実際に到達する値型を result slot へ流すようにした。

## runtime guard marker split

- guard marker の実行時意味を `get_id` / `frame_id[id]` / `marker[id]` /
  `add_id[depth, path, id]` の四 primitive として spec に書き直した。
- runtime 内部では、value marker を `Frame { id }` と `AddId { id, path, depth, guard_own_path }` に分けた。
  - `Frame` は dynamic `GuardIdList` の push/pop だけを行う。
  - `AddId` だけが request.guard_ids を書き換える。
  - stack handler 由来の marker は handler arm body まで包まず、handled computation 側へ掛ける。
- 現行 IR では effect row item が act family path、runtime request が operation exact path を持つため、
  `add_id` の allowed path 判定は family prefix として扱う。
- continuation resume では frame を復元しつつ、resume marker の own-family guard は弱める。
  これにより recursive handler が前回の handler frame によって自分の request を読めなくなる問題を避ける。
- runtime value formatter は hygiene marker を user-facing result に出さない。
