# 2026-06-15

## specialize solver rewrite decision

- `specialize` の現行 `ConstraintGraph` は、node / bound / edge を持っているが、解決段階が
  slot-local な lazy resolver になっている。
  - `TypeResolver::slot_solution` がその slot の lower / upper を読みに来た時だけ畳む。
  - 未処理の構造制約を worklist で飽和させてから substitution を作る設計になっていない。
  - 循環に当たったときに `TypeSlotKind::default_type()` を返す経路があり、graph を解いた結果ではない
    default が混ざりうる。
- これは `spec/2026-06-07-principal-monomorphization.md` の「制約グラフを解く」という設計意図に反する。
- spec の `制約グラフ` 節へ、subtype constraint worklist を空になるまで処理し、
  飽和済み graph から substitution table を読むことを追記した。
- この worklist は monomorphize demand 内の制約飽和であり、SCC は解決単位ではない。
- 実装したこと:
  1. `ConstraintGraph` に pending subtype constraint worklist を持たせた。
  2. `constrain_subtype` を、即時再帰ではなく constraint を worklist へ積む入口にした。
  3. bound / edge 追加で生まれる派生制約も worklist へ積み、role demand 解決前に飽和させるようにした。
  4. 循環 slot を `unit` / empty effect row へ default する経路を消した。
  5. call-site variant narrowing と case reachable arm から、実際に到達する値型を result slot へ流すようにした。

## specialize2 graph solver spec

- 既存 `specialize` を expected type 伝播で直し続ける方針をやめ、`specialize2` を新規に書く判断に切り替えた。
- `spec/2026-06-15-specialize2-graph-solver.md` を追加した。
  - 2026-06-07 の monomorphization spec は `specialize2` の根拠にしない。
  - `signature_for_scheme(..., None)` 由来の `unit` / pure effect default を禁止事項にした。
  - 到達した `MonoInferTask` ごとに、erased IR から mono 用 simple-sub 制約を作り直す。
  - 解けた concrete signature から参照先 def / typeclass member / cast の mono task を再帰的に展開する。
  - 全 task の concrete solution 後だけ `mono::Program` を生成する。
  - 変数参照は型代入制約を入れるが、それ自体を runtime result として扱わない。

## runtime guard marker split

- guard marker の実行時意味を `get_id` / `frame_id[id]` / `marker[id]` /
  `add_id[depth, path, id]` の四 primitive として spec に書き直した。
- runtime 内部では、value marker を `Frame { id }` と `AddId { id, path, depth, guard_own_path }` に分けた。
  - `Frame` は dynamic `GuardIdList` の push/pop だけを行う。
  - `AddId` だけが request.guard_ids を書き換える。
  - stack handler 由来の marker は handler arm body まで包まず、handled computation 側へ掛ける。
- 現行 IR では effect row item が act family path、runtime request が operation exact path を持つため、
  `add_id` の allowed path 判定は family prefix として扱う。
- `add_id` は対象 effect family の request には id を振らず、それ以外の visible request にだけ id を振る。
- runtime value formatter は hygiene marker を user-facing result に出さない。

## playground wasm compact std profile

- playground で例を切り替えたとき、`1 + 2` や Data & Methods のような root でも embedded std
  全体を wasm 上で推論し直して `Running` に残る問題を確認した。
- `run` の fallback を no-std → compact playground std → full embedded std にした。
  - compact profile は arithmetic / cmp / fmt role / list / range / flow / nondet / junction /
    var / console だけを通常の Yulang source として集める。
  - compact profile で lowering/runtime error が出た場合は full embedded std へ落とす。
- ブラウザ worker 実測:
  - `1 + 2`: 約 0.37s
  - Data & Methods: 約 0.22s
  - junction: 約 0.20s
- 実 UI でも初回 Tour が約 0.5s、Data & Methods への切り替えが約 0.2s で完了した。

## infer stack-weight replay cleanup

- `dump-poly-std` の制約 replay で `floor[Empty]` と `push[Empty]` が増え続け、`1 + 2` でも
  20 秒 timeout する状態を確認した。
- まず、pop を発生させる制約経路を直した。
  - lower 側の `Stack` / `NonSubtract` wrapper は、外側から剥がした順に suffix compose する。
    これで `Stack(push, NonSubtract(pop, x))` が replay 前に `push; pop` の順で相殺される。
  - 注釈と defined lambda skeleton の出力 predicate は、`Stack(pop)` ではなく
    `NonSubtract(inner, id)` として制約へ落とす。
  - 再帰 self の skeleton は、body lowering 前に注釈済み引数の空 predicate 層も接続する。
    未注釈引数は後から callback subtract が増える可能性があるので、空 predicate の直結は final まで保留する。
  - body lowering 中に unannotated callback subtract が frame に追加された時点で、
    active recursive skeleton の predicate 接続を追加する。
- それでも残った timeout は、終端値型 bound に保存された stack weight が、row subtraction に届かないまま
  bounds replay で別状態として増えていたことで起きていた。
  - `floor[Empty] pop(p)[H...]` を `floor[Empty]` に畳む正規化は元仕様にないため戻した。
  - 非 effect 具象型の terminal edge では weight を観測できないので、ここでだけ weight を空に畳むようにした。
  - effect atom は同じ `Con(path, args)` 表現を使うため、`ModuleTypeKind::Act` 由来の path 登録で
    effect family と値型を区別する。
  - 詳細調査用に `notes/progress/2026-06-15-stack-weight-pop-balance-handoff.md` を追加した。

  - `cargo build -q -p yulang`
  - `./target/debug/yulang dump-poly-std` for `1 + 2`: 約 2.82s
  - `./target/debug/yulang dump-poly-std` for `(each 1..3 + each 1..2).list.say`: 約 2.84s
