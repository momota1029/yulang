# 2026-05-03

## runtime specialization redesign

- `complete_principal` を role requirements まで見る形に広げ、`Fold<list<int>, item=t>` から associated item substitution を出せるようにした。
- `substitution-specialize` 側で、単相 role impl も候補に入れるようにした。
- 空代入の単相 impl は clone せず、元 binding に直接 rewrite する。
- role impl の receiver 判定は exact/bounds matching に寄せた。通常の numeric widening を使うと `int` receiver が `float` impl にも合ってしまうため。
- `examples/07_junction.yu` では `std::prelude::&impl#487::lt__ddmono` が消え、`std::prelude::&impl#487::lt` と call site の `thunk[std::junction::junction, bool]` wrapper に分かれた。
- opt-in 計測では `junction` が 14 passes / 9 specializations、`effect_handler` が 10 passes / 1 specialization、`types` が 10 passes / 2 specializations。
- runtime `Thunk` の effect から `param_effect` / `ret_effect` を推測する入口を試した。`std::flow::sub::sub` の残り effect 変数は拾えるが、handler body adapter がないまま clone すると effect row が増殖して `junction` が 38 passes まで悪化した。
- そのため `ExprKind::Handle` を含む binding は `substitution-specialize` から構造的に除外した。`std::flow::sub::sub` と `std::junction::junction::junction` は名前ではなく body の handler boundary で判定される。
- handler boundary の構造解析を `pipeline::handler_boundary` module に切り出した。`HandlerBindingInfo` は consumes と residual の有無、pure handler かどうかを持つ。
- `HandlerCallBoundary` を追加した。主型側の handler input/output effect と、呼び出し地点の runtime thunk input/output effect を並べて debug できる。`std::flow::sub::sub` では consumed effect が主型側に残る一方、呼び出し地点ではすでに外側の `junction` だけが見えることがあり、adapter は両方を見る必要がある。
- `HandlerAdapterPlan` を追加した。`sub` は before が `sub + junction + tail`、after が `junction + tail`、`junction` handler は before が `junction + tail`、after が `tail` になる。まだ handler clone は解禁せず、adapter 入力を固定した段階。
- `YULANG_SUBST_SPECIALIZE_HANDLERS=1` で handler clone に plan を流せるようにした。`sub` を clone すると 38 passes に悪化するが、`consumes_input_effect=true` かつ `residual_after` が空の handler だけに絞ると壊れず、`junction` は 14 passes / 9 specializations のまま。
- `YULANG_SUBST_SPECIALIZE_HANDLER_CALL_ADAPTER=1` を追加した。unsupported handler plan の場合は handler clone を作らず、call site の第 1 引数 thunk を `plan.residual_before` の effect へ持ち上げて旧 demand に渡す。`sub` で 38 passes には悪化しないが、`junction` は 14 passes / 9 specializations のままで削減なし。
- `examples/02_refs.yu` をテスト対象に加えた。直接スカラー参照代入 `&x = ...` が未定義名になっていたため、lower の `VarBindingUsage` を read/write に分け、書き込みでも `var_ref` helper と参照 binding を作るようにした。
- `02_refs.yu` は元の形で `[0] (11, 21)` になった。通常実行は 37 passes / 39 specializations、置換 opt-in は 38 passes / 41 specializations。参照 helper 周りはまだ旧 demand-driven 単相化に強く乗っている。
- ChatGPT Pro への相談結果を `tasks/current.md` に反映した。今後は simple substitution だけでなく、制約生成時に `CastHole` / `AdapterHole` 相当の evidence を置き、runtime で `Coerce` / thunk wrapper / `BindHere` / handler adapter を埋める elaboration として見る。
- `ExpectedEdge` の invariant テストを追加した。通常 edge は solver の lower/upper に同じ `actual <= expected` 境界を持つこと、effect edge は `actual_eff <= expected_eff` を持つこと、kind と reason が対応していることを見る。
- synthetic struct / enum constructor と struct field projection の `ExprKind::Coerce` 生成を `LowerState::representation_coerce` に寄せ、`ExpectedEdgeKind::RepresentationCoerce` を記録するようにした。representation coerce は通常の subsumption constraint ではないので、制約追加はせず観測点だけ置く。
- direct ref assignment の `AssignmentValue` edge に右辺 span を入れた。`my ($x) = 1; &x = 2` の回帰テストで span と solver constraint を確認する。
- CLI 型エラーに `ExpectedEdge` 由来の context 行を追加した。`my x: int = "s"` は `context: annotation expected int; expression provides std::str::str`、`f "s"` は `context: function argument expected int; expression provides std::str::str` を出す。direct cause が残る edge だけを使い、伝播越し assignment error の復元はまだしない。
- ChatGPT の次レビューを反映した。ExpectedEdge は 9割近いが、次は diagnostic edge selection の tie-break、edge actual/expected の compact 表示、`complete_principal` での ExpectedEdgeEvidence、RepresentationCoerce と CoerceEvidence の対応強化が優先。
- runtime lower の thunk / `BindHere` / forced-effect thunk 挿入点を `RuntimeAdapterProfile` として数えるようにした。`YULANG_DEBUG_RUNTIME_ADAPTERS=1` で `value-to-thunk`、`thunk-to-value`、`bind-here`、`reused-thunk`、`forced-effect-thunk` が見えるので、ExpectedAdapterEdge がまだ覆っていない runtime 側 adapter 境界を観測できる。
- `catch` lowering で handler boundary の adapter evidence を置き始めた。handled effect がある `catch` は `HandlerResidual`、arm body の result join は `HandlerReturn` として `CoreProgram.evidence.expected_adapter_edges` まで残る。
- continuation application は `ResumeArgument` adapter edge として記録するようにした。resume call の通常 `ApplicationArgument` edge とつながるので、handler arm 内の resume 引数境界を debug できる。
- `YULANG_USE_EXPECTED_ARG_EVIDENCE=1` の opt-in を monomorphize の apply evidence arg signature hint にも接続した。閉じた `ApplyEvidence.expected_arg` だけを優先し、open な expected arg は従来の `evidence.arg` へ落とす。
- `--runtime-phase-timings` に expected arg evidence profile と runtime adapter profile を統合した。`examples/10_effect_handler.yu` では present=111、converted=108、usable_by_table=1、used=0 で、現状は evidence が多くても runtime_usable gate でほぼ落ちていることが見えた。
- `DemandQueueProfile` を追加し、`--runtime-phase-timings` に demand queue の attempted / pushed / open / closed / duplicate / closed-cover skip を出すようにした。`examples/10_effect_handler.yu` は attempted=4、pushed=3、skipped_duplicate=1 で、現状は closed-cover skip がまだ出ていない。
- expected arg evidence の unusable 理由を分解した。`examples/10_effect_handler.yu` では ignored_table_open=107、ignored_table_uninformative=9、ignored_table_not_runtime_usable=0 なので、runtime_usable が増えない主因は「closed ではない」こと。
- runtime lower の expected arg table gate を edge 全体の `runtime_usable` から expected slot 単体の closed / informative / runtime type usable 判定へ切り替えた。actual slot が open でも expected slot が閉じていれば利用する。`examples/10_effect_handler.yu` は opt-in なしで usable_by_table=70、opt-in ありで usable_by_table=71 / used_as_arg_type_hint=25 / used_as_lowering_expected=63 まで増えた。pass 数は 9 / specializations 1 のままなので、evidence は読めているが demand 削減へはまだ届いていない。
- monomorphize profile に `effective_passes`、pass ごとの progress、追加 binding path、DemandEvidence profile を追加した。`examples/10_effect_handler.yu` は opt-in ありで expected_arg_used=244 だが expected_arg_changed_signature=0 / same_signature=244。つまり expected arg evidence は demand signature hint として読まれているが、signature 自体は変えていない。追加 specialization は `listen__ddmono0` の 1 個だけ。
- expected adapter evidence profile と specialization の target/signature 表示を `--runtime-phase-timings` に追加した。`examples/10_effect_handler.yu` は expected_adapter_evidence total=76 / runtime_usable=0 / closed=0 / handler_residual=11 / handler_return=30 / resume_argument=8。runtime 側では value_to_thunk=3 / thunk_to_value=14 / bind_here=14 が出ているが、expected adapter evidence の value_to_thunk / thunk_to_value / bind_here は 0。残った specialization は `listen__ddmono0 <= listen :: thunk[out]int -> str -> (int, str)`。
- runtime adapter profile を ApplyEvidence 由来で分解した。`examples/10_effect_handler.yu` は apply_evidence_thunk_to_value=10 / apply_evidence_bind_here=10 / apply_evidence_value_to_thunk=0。runtime `BindHere` の多くは apply 境界で thunk を値へ戻すために入っているが、expected adapter evidence 側にはまだ `ThunkToValue` / `BindHere` が出ていない。
- `ApplyEvidence.arg_source_edge` 由来の文脈を runtime adapter profile へ伝播した。`examples/10_effect_handler.yu` は apply_evidence_bind_here=10 のうち source edge 文脈まで追えるものが 1 個だけ。残りは final apply argument の evidence ではなく、apply evidence context 内の別経路で発生している。
- `ExpectedEdgeKind::ApplicationCallee` を追加し、apply lowering で `callee.tv <= expected_callee_tv` を `ExpectedEdge` として記録するようにした。core `ApplyEvidence` には `callee_source_edge` / `expected_callee` を載せ、runtime adapter event には `callee_source_edge` も表示する。
- `examples/10_effect_handler.yu` の runtime adapter event は、baseline で `apply.lower-callee` の `ThunkToValue/BindHere` 全 9 組、opt-in + coalesce で残る 4 組が `callee_source_edge=Some(...)` まで辿れるようになった。match はまだ 0 なので、次はこの `ApplicationCallee` edge から semantic adapter evidence を作るか、runtime event 側で observed adapter として扱うかを分ける。
- `YULANG_USE_EXPECTED_CALLEE_EVIDENCE=1` を追加し、DemandEmitter / DemandChecker の apply parameter hint に `ApplyEvidence.expected_callee` の parameter slot を混ぜる経路を作った。`examples/10_effect_handler.yu` の opt-in + coalesce では expected_callee_present=249、used=242 だが changed_param_signature=0 / same_param_signature=242。callee evidence も demand signature 自体は変えていないため、次は observed adapter / semantic `ForceThunkToValue` 側が本命。
- runtime adapter event から `ObservedAdapterEvidence` を作るようにした。`ThunkToValue + BindHere` の pair は semantic に `ForceThunkToValue` 1 件へ畳む。`examples/10_effect_handler.yu` baseline では runtime events total=20 に対し observed evidence total=10 / force_thunk_to_value=10 になり、二重カウントではなく意味的 adapter 数として見られる。
- `DerivedExpectedEdgeEvidence` を core `PrincipalEvidence` に載せた。`--core-ir --verbose-ir` で derived expected edges を見られる。runtime timings では derived evidence の kind / polarity counts と、runtime adapter event の source edge に derived child があるかを出す。`examples/10_effect_handler.yu` baseline は derived_expected_evidence total=1108、function_param=509 / function_return=509、adapter event 20 件のうち matched_derived_parent=14。

## next

- 残る旧 demand は `std::flow::sub::sub` と `std::junction::junction::junction` が中心。
- handler body 専用に `residual_before` / `residual_after` / return thunk wrapper を作る最小 adapter を設計する。
- adapter の入口は `HandlerBindingInfo` と `HandlerCallBoundary` にする。
- 次は `HandlerAdapterPlan` を使って、substitution clone 内の先頭 handler に `HandleEffect` と return wrapper を流し込む。ただし clone 解禁は opt-in で、row 増殖しないことを `junction` で見てからにする。
- `sub` のような「消費 effect が主型側にだけ残る handler」は、単純な clone ではまだ危ない。call site arg を handler principal input へ持ち上げるか、handler 本体ではなく call site 側に adapter を置くかを次に分けて試す。
- call site adapter だけでは pass 数は減らない。次は旧 demand 側が handler binding を再処理する理由を debug し、handler demand を stable/cached として扱えるかを見る。
- `02_refs.yu` は今後の回帰・計測対象に残す。参照 helper を新パスへ移すか、handler と同じく旧 demand 側へ明示的に残すかを切り分ける。
- 次の設計単位は `complete_principal` を principal elaboration evidence の入口へ広げること。まず apply evidence と handler boundary で intrinsic/contextual 型と adapter hole を持てる形を考える。
