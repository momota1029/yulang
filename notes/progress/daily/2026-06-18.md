## source collection module discovery skip

- source collection は実ファイルを二度 `read_to_string` してはいなかった。
  `Collector` が読んだ `source` を `CollectedSource` として後段へ渡し、`sources::load` はその in-memory source を読む。
- ただし collect 側の `discover_module_loads` は module load request の有無を見るだけなのに、
  各ファイルを full parse していた。`sources::load` も header parse + full parse を行うため、collect/load の時間が似ていた原因は二重 read ではなく二重 parse。
- module load request は `mod foo;` / `use mod foo::*` だけなので、source text に `"mod"` が無いファイルでは parse せず空を返すようにした。
  false positive は parse するだけで、false negative は keyword が ASCII `mod` である限り起きない。
- `showcase` infer-only repeat 5 では `collect` が 39〜41ms から 8ms 前後へ低下。

## embedded std loaded-prefix cache

- CLI の `.yuir` / `.yuvm` cache は source collection 後の whole source-set artifact cache なので、
  `check` や playground の in-memory embedded std route では `sources::load` の full parse が残っていた。
- `sources::load_with_loaded_prefix(prefix, suffix)` を追加した。
  `prefix` は suffix に依存しない dependency-closed な loaded file set という不変条件を持ち、
  suffix の op table は prefix の header / export surface から組み直す。
  これは将来の source-SCC compiled-unit import の小さい前段として扱う。
- `yulang::source` は embedded full std / embedded playground std の loaded prefix を thread-local に cache し、
  root source だけを suffix として parse する入口を追加した。
  既存の `CollectedSource` route はそのまま残し、FS std route や CLI artifact cache の意味は変えていない。
- wasm playground の colorize と compact playground std run path は、新しい loaded-prefix route を使う。
  full std fallback と no-std route は既存 route のまま。
- 一時 example で旧 route と新 route を同一プロセス比較した。
  - load-only（120 iter, debug）: 旧 `168.410ms/iter`、新 `3.169ms/iter`、`53.147x`。
  - build poly まで（6 iter, debug）: 旧 `1169.368ms/iter`、新 `998.976ms/iter`、`1.171x`。
  - build 全体は lowering/infer が支配的なので改善は限定的。colorize / parse warm path では大きく効く。

## instantiate predicate direct lower-bound path

- `instantiate_subtype_predicate` / constraint replay の内訳を見るために counters を追加した。
  - instantiate predicate の root shape: var / stack / non-subtract / fun / con / other。
  - lower / upper bound 追加時の replay input / enqueue / var-var replay。
  - direct var-var replay 時の replay input / enqueue / empty replay skip。
- `showcase` では instantiate predicate は `Fun` 853、`Con` 43、`Stack` 4 で、
  `Var` / `NonSubtract` は 0 だった。
- predicate root が直接 `lower <: target-var` の lower bound 追加へ落とせる形なら、
  `Neg::Var(target)` を allocate して `subtype` queue 経由で drain する代わりに、
  `add_lower_bound(target, lower, empty)` を直接呼ぶ入口を追加した。
  対象は `Con` / `Fun` / record / polyvariant / tuple / row。
- `Stack` / `NonSubtract` / `Union` / `Var` は直接化しない。
  `Stack` と `NonSubtract` は weight / pre-pop の意味を持ち、`Union` は分配が必要で、
  `Var` は lower/upper 両側の var-var bound 登録が必要なため。
- direct var-var replay は、両側 weight が空で transitive closure を作らない replay を clone 前に除外するようにした。
  これは既存の「empty var-var replay は閉じない」挙動を、queue へ入れる前に早めただけ。

## control VM frame-runtime direction

- 旧作 bench に近い `use std::control::nondet::*; { my xs = (each 1..20 + each 1..20).list; () }`
  を release / no-cache で測ったところ、`vm_eval` は 160ms 台だった。

  arithmetic / list merge より、request resume closure と marker frame 再導入が支配的と見る。
- 旧作 `archive/crates/yulang-vm/src/vm/control.rs` は `ControlContinuation { frames, guard_stack, lookup_stack }`
  と `ControlFrame` を持ち、request が出たら `push_frame(request, frame)` で continuation に
  データ frame を積む。現行の `Rc<dyn Fn>` resume chain とはここが大きく違う。
- 現行 control IR / lowerer は維持し、runtime だけを frame-based continuation へ寄せる方針を
  `notes/design/control-vm-frame-runtime-plan.md` に残した。
- 先に試した hybrid 案は棄却した。
  - `Request` に `frames` を足し、`continue_with_request` の nested resume closure を
    frame push に置き換えた。
  - `request_resume_steps` は 20x20 で約 35k から約 13k へ落ちた。
  - ただし `Vec<RequestFrame>` 版は 20x20 が 190ms 台へ悪化し、`SmallVec<[...; 4]>` 版は

  - `SmallVec<[...; 1]>` 版も 20x20 が 180ms 付近へ戻ったため、closure chain の上に
    frame list を載せる半端な形は採用しない。
  - 次は `Request.resume` を残したまま包むのではなく、marker / handler / request resume を
    最初からデータ frame として持つ runtime へ進める。

## control VM cached catch arm keys

- frame runtime 本体へ入る前の小さい slice として、catch arm の operation path を
  `ExprId` 単位で runtime cache するようにした。
- `CatchArm` は IR / artifact 上では `Vec<String>` のまま残し、runtime entry で
  `RuntimeCatchArm { operation_key: Option<InternedPath>, ... }` へ変換する。
  以後の request matching は per-request `intern_path` ではなく cached `InternedPath` を読む。
- arm list も `Rc<[RuntimeCatchArm]>` で持つため、pass-through resume が `Vec<CatchArm>` を
  丸ごと clone しない。
- `use std::control::nondet::*; { my xs = (each 1..20 + each 1..20).list; () }`
  は release / no-cache で `vm_eval` 148.8ms。
  直前の 160ms 台より少し改善した。
- `showcase` repeat 3 は `vm_eval` 341.3 / 346.6 / 394.2ms。
  揺れはあるが、少なくとも明確な悪化ではないため採用する。
- `Value::EffectOp` / `Thunk::Effect` に `InternedPath` を持たせる実験もしたが、
  20x20 nondet discard が `vm_eval` 164.8ms へ悪化したため棄却した。
  effect op 生成側の key cache ではなく、marker / request resume の frame 化を優先する。

## control VM path id prefix check

- 20x20 nondet discard を追うため、`bench/nondet_20_discard.yu` を追加した。
  `use std::control::nondet::*; { my xs = (each 1..20 + each 1..20).list; () }`
  で root formatting を外した VM-heavy ケースとして使う。
- `InternedPath` を path 全体の `id` と prefix id table で持つようにした。
  path equality は `id` 比較、prefix 判定は `path.prefix_ids[prefix_len - 1] == prefix.id` で済む。
- 20x20 nondet discard の `path_prefix_segments` / `path_eq_segments` は 0 になった。
  `vm_eval` は 139.8 / 148.4 / 145.4ms で、直前の 142.6 / 138.3ms 付近から大きくは変わらない。
- `showcase` は `vm_eval` 365.8 / 364.3 / 403.8ms。
  path 比較だけでは主因を倒せていない。残る支配要因は引き続き
  request resume closure、marker frame resume、active add-id scan。

## control VM active marker guard fast path

- `mark_request_with_active_markers` は、request が live guard をすでに持つ場合、
  以後の active marker を skip する規則を持っていた。
  ただし従来は live guard 判定の前に毎回 path prefix check をしていたため、
  skip される marker にも prefix work が乗っていた。
- live guard の有無を loop の前に計算し、guard が付いた後は prefix check 前に skip するようにした。
  意味は変えず、不要な path work だけを落とす。
- 20x20 nondet discard では `path_prefix_checks` が 74360 から 20033 へ低下した。
  `vm_eval` は 150.0 / 146.7 / 141.2ms。
- `showcase` では `path_prefix_checks` が 118496 から 38778 へ低下した。
  `vm_eval` は warm で 382.6 / 381.7ms 付近。劇的ではないが、支配要因の一部を削った。

## control VM shared resume marker plan

- marker frame が request を外へ投げるとき、resume closure は `resume_markers.clone()` で
  `Vec<ValueMarker>` を毎回 deep clone していた。
  20x20 nondet discard では `marker_frame_resume_steps` が 49980 あるため、
  multi-shot continuation で同じ marker plan clone を何度も踏む。
- 通常 value path はそのまま `Vec<ValueMarker>` を使い、request resume 用に作った
  continuation marker plan だけを `Rc<[ValueMarker]>` で共有するようにした。
  以前棄却した「marker frame 全体を Rc 化する」案とは違い、request resume の multi-shot clone だけを狙う。
- `ActiveFrame` は handler の文字列 path を使っておらず、`handler_key.is_some()` と
  interned key だけを読んでいたため、`Option<Vec<String>>` を削って `Option<InternedPath>` だけにした。
- 20x20 nondet discard は `vm_eval` 121.0 / 129.1 / 130.9ms。
  直前の 140〜150ms 台から明確に改善した。
- `showcase` は `vm_eval` 378.9 / 355.4 / 352.9ms。
  初回は揺れるが、warm sample は 350ms 台前半まで落ちた。

## control VM resume marker reuse

- shared resume marker plan は、作成時点で `markers_for_continuation_resume` 済みである。
  それにもかかわらず、resume 後の実行が再び request を外へ投げるたび、
  `close_shared_marker_frame_result` が同じ marker plan を再正規化していた。
- `with_shared_marker_plan` を `with_shared_resume_marker_plan` に改名し、
  resume 用 marker plan だけを受ける invariant を名前で固定した。
  shared resume path から出た request は同じ `Rc<[ValueMarker]>` をそのまま次の resume marker plan に使う。
- 先に試した active marker plan 自体の `Rc` 化は、20x20 nondet discard が
  `vm_eval` 148.2 / 141.3 / 130.8 / 149.7 / 141.7ms と悪化気味だったため棄却した。
  active frame state の広い共有ではなく、multi-shot resume marker だけを狙う方針を維持する。
- 20x20 nondet discard repeat 5 は `vm_eval` 110.1 / 107.5 / 112.4 / 118.2 / 117.8ms。
  counter は変わらないため、主に marker plan 再正規化 / allocation の削減と見る。
- `showcase` repeat 5 は `vm_eval` 277.3 / 285.8 / 275.1 / 293.6 / 261.0ms。
  350ms 台からさらに落ち、frame runtime 本体へ入る前の小さい slice として採用する。

## control VM direct binary primitive apply

- 旧作 control VM と同じ方向の小さい slice として、
  `Apply(Apply(PrimitiveOp, lhs), rhs)` と
  `Apply(Apply(InstanceRef(PrimitiveOp), lhs), rhs)` を direct binary primitive として評価するようにした。
  path 名ではなく、control IR の構造と解決済み instance entry だけを見る。
- lhs / rhs の評価順は既存の curried apply と同じまま保ち、どちらかが request を出した場合も
  `continue_with` で既存 continuation 経路へ戻す。
  変えるのは `PrimitiveValue { args }` の部分適用 value を作らず、2 引数が揃った時点で
  `apply_primitive` へ直接渡す点だけ。
- direct `PrimitiveOp` 直書きだけを見る実験は、20x20 nondet discard で発火せず、
  showcase でも 16 箇所程度しか減らなかった。
  std primitive は主に `InstanceRef` 経由で来るため、解決済み instance entry まで見る必要があった。
- 20x20 nondet discard repeat 5 は `apply` が 26198 から 20230、
  `prim_part` が 1702 から 0、`expr` が 49989 から 46582 へ低下した。
  `vm_eval` は 111.9 / 96.7 / 98.9 / 109.2 / 111.9ms。
- `showcase` repeat 5 は `apply` が 90247 から 67295、
  `prim_part` が 6148 から 0、`expr` が 164641 から 152322 へ低下した。
  `vm_eval` は 246.9 / 230.5 / 263.8 / 238.3 / 225.7ms。
- まだ `request_resume_steps` と `marker_frame_resume_steps` は残る。
  direct primitive は call/value 側の無駄を削る slice で、frame runtime 本体の代替ではない。
