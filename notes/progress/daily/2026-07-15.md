## Yumark Markdown cold-compile performance: 8 verified optimizations landed

- Following the suffix-safety-oracle project's abandonment after Stage 0 falsified its core hypothesis, pivoted to directly speeding up std recompilation through ordinary performance engineering, with no new caching or soundness-critical architecture.
- Profiled, implemented, verified, committed, and pushed eight optimizations targeting role resolution/generalization costs for the Yumark Markdown workload's `proof` definition and its `YumarkRender` role demand. That demand initially accounted for about 96-97% of role-resolution cost and, after other costs shrank, was found to account for about 85% of total generalization cost:
  1. `3fcb9cc1` — skipped role candidate compaction for non-concrete demands by moving an existing early-return check earlier.
  2. `bf539514` — prechecked compacted role candidate input heads for definite nominal/builtin mismatches before deep matching, and eliminated a redundant clone in identity interval substitution.
  3. `ae6afc58` — reused exact final role-solve dispositions across generalization iterations through a conservative cache gated on exact equivalence, never a heuristic.
  4. `5588abb2` — added the analogous precheck at the raw, pre-compaction candidate representation level.
  5. `6529b7e8` — removed an unnecessary `TypeSubst` rollback clone in deterministic constructor matching after verifying that no caller relied on rollback semantics there.
  6. `cfa017f6` — added a cheaper raw-input-exact-match fast path on top of `ae6afc58`'s gate, skipping normalized-view reconstruction entirely when raw inputs are unchanged.
  7. `df1c23f3` — added a conservative guard to skip an unchanged method-role pass, using constraint/role epochs plus a new generation counter bumped at every selection/candidate mutation site. This reduced 190 no-progress passes to 101 and cut method-role-solve cost by 46%.
  8. `51e9a722` — eliminated clone-before-check waste in dominance interval collection by rejecting trivial intervals through borrowed references before cloning, and in role coalescing by moving component members with `.take()` instead of cloning.
- Cumulatively, repeated timing runs after each fix showed Yumark Markdown cold-compile `build_poly` time falling from about 35.7s to a 14.36s mean, a roughly 59.8% reduction.
- Every fix was verified with `cargo test -p infer --lib` (no regressions beyond the pre-existing five known-unrelated Yumark DefId-ordering failures in `case_02`) and a repository-std semantic comparison in the style of `check-poly-std examples/showcase.yu`, confirming zero change in typed-lets, missing-schemes, and lowering-errors counts. Higher-risk fixes also received targeted adversarial coverage: the final-role-solve reuse gates and method-role-pass skip guard each have dedicated negative tests proving that they fall back correctly when inputs genuinely change.
- Two quick profiling follow-ups after fix 8 found no further low-risk win. The remaining cost is genuinely concentrated in `proof` generalization, still about 85% of total generalization cost, but reusing intermediate iteration state would require incremental reachability/collector infrastructure with dirty tracking. Constraint replay is also about 87% duplicate entries, but is a separate, more correctness-sensitive area. Both this session and the 2026-07-14 early-fallback investigation independently identified these as dedicated larger-scope projects rather than quick bounded fixes.
- Next step for further Markdown speed gains: first scope an incremental/dirty-tracking role-solver scheduling project through read-only, decision-free characterization, following the established project discipline. Multiple sessions have repeatedly identified this as the actual remaining lever, but it has never been properly scoped as its own project.

## Shadow dirty oracle: owner-level method-role scheduling characterization

- Added a `#[cfg(test)]`-only shadow oracle beside the existing `df1c23f3` method-role pass guard. It never skips a solve. For every parent `DefId` of an unresolved selection, it predicts clean/dirty from the last no-progress observation, lets the real owner solve run, and records a hard-test mismatch if a clean prediction has a different real outcome.
- The dependency fingerprint records the exact owner solve read frontier: SCC root availability; owner raw-role epoch; owner unresolved-selection membership/use-sites; constraint bound epochs; adjacency, subtract facts, pre-pop effect families, levels, and birth levels read during main-root/raw-role/candidate compaction; method-taint projections actually queried by the owner; applied-resolution membership queries; and test-only generations for every candidate/prerequisite role bucket transitively scanned.
- Prediction deliberately reuses the method-taint index that the current forced pass already builds before owner solving. A future owner scheduler could therefore avoid owner solves, but not the shared method-taint rebuild. Candidate generations are complete for the session mutation APIs used by real lowering; direct test mutation of the public table bypasses them and is outside the fixture evidence.
- Focused cold-lowering results, with zero clean-prediction mismatches:
  - Yumark Markdown: 100 forced passes, 7,359 / 7,526 owner-checks predicted clean (97.78%). The workload owner `render_markdown_doc` was clean 94 / 95 checks (98.95%).
  - Yumark HTML: 100 forced passes, 7,359 / 7,526 clean (97.78%). `render_html_doc` was clean 94 / 95 (98.95%).
  - Repository std only: 91 forced passes, 6,317 / 6,480 clean (97.48%).
  - All observed owner-checks had an SCC root, so the aggregate clean rate is not inflated by repeated root-unavailable checks.
- Important identity correction: under the task's exact owner definition (`SelectionUse.parent`), root `proof` (`DefId(167)` in both Yumark fixtures) had zero owner-checks and therefore no meaningful clean/dirty fraction. The 95-check owner on the Markdown path is `render_markdown_doc`, not `proof`; HTML analogously uses `render_html_doc`. Earlier profiling attribution to `proof` and method-pass owner identity must not be treated as interchangeable.

  - `cargo test -p infer --lib -- --test-threads=1`: 754 passed / 5 known unrelated Yumark DefId-ordering failures, exactly the 753/5 baseline plus the new passing test.
  - `cargo build --release -p yulang`: pass. All new state and hooks are `#[cfg(test)]`; the production path retains the same owner order, solve call, and progress handling.
  - Markdown no-cache release timing: pre-change `build_poly` 5.161s; rebuilt artifact 5.424s, then 4.888s on confirmation. The values bracket the baseline, and method-role time moved from 545.1ms to 520.9ms / 487.4ms, so there is no measured production regression.

## Owner-level dirty scheduling: Stage 3 shadow scheduler

- Implemented the journal-backed owner scheduler as `#[cfg(test)]` shadow infrastructure. Its session-owned state contains terminal owner records, reverse dependency subscriptions, the current dirty set, per-key generations, mutation serial tracking, and comparison telemetry. Every owner still runs the unchanged real solve; there is no clean-prediction skip path.
- The scheduler drains the Stage 2 production mutation outbox at forced-pass entry, before every owner eligibility prediction, after every real owner solve, and at ordinary constraint-event routing boundaries. Record publication replaces all old reverse edges atomically. Selection resolution, SCC quantification, stale-owner detection, and session destruction remove records and subscriptions without tombstones.
- Found an important Stage 2/Stage 3 boundary drift while validating the design: terminal owner records survive across `drain_work` calls, while ordinary lowering can mutate their dependencies outside the Stage 2 one-pass activation window.
