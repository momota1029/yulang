## Composed nested handler contract

`compose(total_amount, all_paths): ...` の形で `flip` と `amount` の handler を合成する例を
`notes/research/consultation/examples/compose_nested_handler_contracts.yu` として追加した。

当初の `our compose(f, g: _ -> [_] _, x) = f g(x)` では、`x` は純粋値として扱われる。
そのため呼び出し側の block は `all_paths` へ渡る前に評価され、`flip::coin` が root 側で
発火して `unhandled effect flip::coin` になる。`all_paths` に computation を渡す contract は、
第三引数側の `x: [_] _` として書く必要がある。

一方で、`our compose(f, g, x: [_] _) = f g(x)` のように `g` を未注釈のままにすると、
`g(x)` 由来の residual effect は `take(Empty)` で保護される。これは `f` の内部 handler が
`g(x)` 由来の effect を偶然捕捉しないための hygiene evidence であり、`total_amount` が
`all_paths(x)` から残った `amount` を処理するこの例では強すぎる。合成例では、
`x: [_] _` に加えて `g: _ -> [_] _` も書き、`g` の戻り effect を surface contract として
明示する。

一方、specialize2 側にも別の問題があった。signature 付き lambda の解決が curried function の
外側一段だけに期待型を使い、内側の lambda を fresh な引数 effect で解いていた。そのため
`x: [_] _` 版でも、内側の `g x` が concrete function application として解けず、effect slot の
衝突に落ちていた。

修正として、`lambda_type_with_signature` が return shape を `expr_with_signature` へ渡し、
curried lambda の内側にも期待型を伝播するようにした。これで `compose` の `g` と `x` が
推論済み signature に沿って解かれ、root block は `thunk[[flip, amount], int]` として
`all_paths` へ渡る。

shadow key は、まず保守的に次の形にした。

```text
(side, pivot var, endpoint var, normalize_for_var_var_replay_key(weights))
```

同じ key が再度入った場合、もし replay を飛ばしてもよかったかを、実際の replay accepted 数で後から分類する。

- `safe_hits`: frontier hit かつ replay accepted が 0
- `unsafe_hits`: frontier hit だが replay accepted が 1 以上
- `unsafe_accepted`: unsafe hit から生まれた accepted constraint 数

`examples/showcase.yu` では次を観測した。

```text
constraint.replay_frontier_shadow_var_var_candidates: 128178
constraint.replay_frontier_shadow_var_var_hits: 0
constraint.replay_frontier_shadow_var_var_safe_hits: 0
constraint.replay_frontier_shadow_var_var_unsafe_hits: 0
constraint.replay_frontier_shadow_var_var_unsafe_accepted: 0
constraint.lower_replay_var_var: 352835
constraint.upper_replay_var_var: 266371
```

つまり、今回の duplicate は「同じ var-var bound fact が同じ pivot に再挿入されている」問題ではない。
それぞれの bound insertion は frontier key としては新しいが、opposite table と join した結果として
同じ consequence を再発見している。したがって次は bound table の local frontier ではなく、
var-var routing graph / transitive consequence 側を見る必要がある。
