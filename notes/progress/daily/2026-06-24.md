## Composed nested handler contract

`compose(total_amount, all_paths): ...` の形で `flip` と `amount` の handler を合成する例を
`notes/research/consultation/examples/compose_nested_handler_contracts.yu` として追加した。

当初の `our compose(f, g: _ -> [_] _, x) = f g(x)` では、`x` は純粋値として扱われる。
そのため呼び出し側の block は `all_paths` へ渡る前に評価され、`flip::coin` が root 側で
発火して `unhandled effect flip::coin` になる。`all_paths` に computation を渡す contract は、
この例では第三引数側の `x: [_] _` として書く必要がある。

一方、specialize2 側にも別の問題があった。signature 付き lambda の解決が curried function の
外側一段だけに期待型を使い、内側の lambda を fresh な引数 effect で解いていた。そのため
`x: [_] _` 版でも、内側の `g x` が concrete function application として解けず、effect slot の
衝突に落ちていた。

修正として、`lambda_type_with_signature` が return shape を `expr_with_signature` へ渡し、
curried lambda の内側にも期待型を伝播するようにした。これで `compose` の `g` と `x` が
推論済み signature に沿って解かれ、root block は `thunk[[flip, amount], int]` として
`all_paths` へ渡る。
