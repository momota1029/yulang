## role 解決前の compact 簡約

- `impl Add (list 'a)` のような unary constructor impl が、open payload bounds を持つ
  demand にも照合できるようにした。
  - candidate 側の `Neu::Bounds('a, 'a, 'a)` は「任意の invariant payload」として扱い、
    demand の lower / upper 両側へ同じ var を再照合して弾かない。
- `generalize_root_with_prepasses` の role 解決 view に、root と role predicate を含む
  compact 簡約を通すようにした。
  - solver 本体は書き換えず、解決判定用の compact view だけを簡約する。
  - `std::data::list::sort_split` / `sort` に残っていた余計な
    `std::data::list::list(...): std::num::Add` 制約が消えた。
- まだ残る型品質:
  - `sort_split`: `list 'd -> 'a -> (list 'b, list 'c) where 'a: Ord`
  - `sort`: `list('a & 'b & 'c) -> list('b | 'a) where ('a|'c): Ord`
  - 次は `Ord` の demand と `case view_raw` 由来の list item 共起を別件で見る。

## constructor pattern payload と impl candidate 簡約

- enum/struct constructor の payload metadata を `ModuleTable` に保持し、constructor pattern lowering が
  constructor function application ではなく宣言済み payload signature から pattern payload 型を張るようにした。
  - これで `case o: opt::some x -> x` が `any -> never` ではなく `opt 'a -> 'a` になる。
  - pattern payload は annotation signature の lower/upper の両向きで value slot と接続する。
- role impl candidate も compact role 簡約に通してから matching するようにした。
  - `impl Add (list 'a)` の candidate が impl method body 由来の完全共起 var を持って
    `list('a | v & 'a & v)` のように膨らんでも、candidate 内部で `'a` と `v` を coalesce してから demand と照合する。
  - `std::data::list::sort_split` / `sort` に戻っていた `std::num::Add` 残差は再び消えた。
- 危険だった `generalize` 側の「vars を無視して exact role arg bounds にする」fallback は外した。
  - role predicate の表面から変数を落とす対症療法になりうるため、candidate 側の共起簡約で直す方へ寄せた。
- 現在の残り:
  - `sort_split`: `list('b & 'c & 'a) -> 'a -> (list('c | 'b), list('c | 'b)) where ('b|'a): Ord`
  - `sort`: `list('a & 'b & 'c) -> list('b | 'a) where ('a|'c): Ord`
  - `Add` は消えたが、`Ord` demand と list pattern/list_view payload 由来の element 共起はまだ詰め切れていない。

## interval 由来の等式併合（共起分析の新規則）

- 不変区間 `(lo, up)` は instantiation ごとに `lo ≤ 実型 ≤ up` を要求するので、lo と up の
  両方に現れる型変数は実型と等しいことが区間単独で確定する。
  - 共起分析に `record_interval_equalities` を追加し、`CompactBounds::Interval` は中心
    `self_var` と、role 引数（中心なし）は両側変数同士を、他出現での同伴に関係なく併合する。
  - 併合は変数同士に限る。具体型との等値は変数の除去になり、区間 1 箇所の根拠で除去すると
    具体型が同伴しない他出現から変数だけが消えて型が狭まるため、除去は従来の全出現同伴に委ねた。
  - spec/2026-06-02-role-system.md の「注意」節に追記した。
- これで red だった 2 テストが緑になった:
  - `co_occurrence_uses_role_interval_to_coalesce_returned_receiver_and_argument`
  - `constructor_pattern_payload_participates_in_guard_role_method`（`view 'a -> 'a -> 'a where 'a: Ord`）
- std::data::list の型品質も向上:
  - `sort_split`: `list('b & 'a) -> 'a -> (list('b&'a), list('b&'a)) where ('b|'a): Ord`（3 var → 2 var）
  - `sort`: `list('a & 'b) -> list('a&'b) where ('a|'b): Ord`（3 var → 2 var）
  - 残る 2 var 形は junk ではなく principal と判断。`'a ≤ 'b` ＋ `Ord('b)` は「payload の
    supertype の順序で比較する」instance（subtyping × role の正当な一般性）と、空リスト
    instance（payload=never）を表現している。

## receiver demand を 1 引数 role に限定（複数引数 role の残骸根治）

- `fold_lines` に `where std::text::str::lines: Index` が残り、`fold_ref_lines` に Index 述語が
  **二重に**付いていた件の根治。
  - 原因: typeclass method selection の receiver demand（`insert_selection_receiver_role_constraint`）
    が role の引数数に関係なく inputs 1 個で demand を作っていた。`Index 'container 'key` のような
    複数引数 role では、引数の個数が合わないため `role_constraints_share_input_vars` で本物の
    demand と merge できず、`resolve_role_candidate` の照合（inputs.len() 一致が前提）も
    永久に通らない述語が残る。
  - 修正: `RoleMethodCandidate` に `input_count` を持たせ（登録時に `ModuleTable::role_inputs`
    から取得）、receiver demand は **1 引数 role のときだけ** fabricate する
    （`SelectionTarget::TypeclassMethod.receiver_role: Option<Vec<String>>`）。
    複数引数 role は method scheme の instantiation が作る demand に receiver が下界として
    流れ込むので、そちらが唯一の demand になり arity も揃う。
  - 1 引数 role は従来どおり: fabricated demand は receiver var 共有で本物と coalesce され、
    関連型は merge が名前単位で安全に統合する（非対称でも落ちない）。
  - spec/2026-06-04-method-selection.md に規則として追記した。
- 結果:
  - `fold_lines`: `lines: Index` 述語が消えた（`impl lines: Index int` に解決・discharge）
  - `fold_ref_lines`: Index 述語が二重→一重（fabricated 側の残骸が消滅）
  - 回帰テスト `multi_input_role_method_selection_resolves_concrete_demand`
- 既知の残り（別件）:
  - selection 連鎖の戻り値に床下 var の union が残る（`int -> int -> 'a|…|'e|int`、
    fold_lines の callback item も同型）。instantiation 変数（secondary level）が body 変数との
    制約で extrude されて床に落ち、polar 消去/sandwich の level 保護対象になるのが原因と推定。
    quantifiers が空になるケースの level 配管調査が必要で、今日は触らない。
  - d596（ref_lines の Fold 連鎖）の巨大 union も同根と推定。

## role prepass と impl associated candidate

- receiver-first concrete demand が key 側や receiver 側に負方向の余分な床下変数を持つ場合でも、
  role 解決前の簡約 view で床下区間等式併合と冗長床変数除去を通すようにした。
  - `Index(ref_lines 'e, int, value = v)` のような demand が、`ref_lines 'e` / `int` の
    candidate へ解決できることを `role_prepass_resolves_receiver_first_concrete_with_negative_extra_inputs`
    で固定した。
- `impl` body 内の明示 associated type（`type value = ...`）を role impl candidate の
  associated 引数として読むようにした。
  - candidate 用の通常引数・associated 引数は、impl member body の annotation solver vars と分けて lower する。
    これで `ref_lines 'e` candidate が method body の `ref_update` effect row 制約で汚れず、
    `fold_ref_lines` の `Index` 残差が消える。
  - member requirement 側は従来どおり role の associated 変数を使う。明示 associated を requirement に直代入すると
    `ref` method body の形チェックが深い再帰を踏むため、candidate の意味付けと body check を分けた。
  - 明示 associated がない impl は、既存の「method body から associated を clamp する」経路を維持する。
- 結果:
  - `fold_ref_lines`: `where ...: Index(...)` が消え、callback item が
    `std::control::var::ref('a, std::text::str::str|...)` まで具体化した。
  - `explicit_associated_type_keeps_impl_input_type_parameter` を追加し、
    `impl Index (box 'a) int: type value = 'a` が `box int -> int -> int` になることを固定した。

## effect row 型引数注釈の tail lowering

- `ref '[file] str` / `ref '[var 't] 't` の第1引数に、注釈由来の ` '[file]` / ` '[var 't]`
  が body 由来の裸 effect atom と併存していた件を修正した。
  - 原因: `AnnConstraintLowerer::lower_invariant_arg` が effect row 型引数を通常の value bounds へ流し、
    下側に `Pos::Row([file])` を置いていた。`ref 'e 'a` の `'e` は `['e]` / `[ref_update a; 'e]`
    の tail に入る effect-kind 型なので、body から来る `file` atom と同じ表現へ合流できず二重 row になっていた。
  - 修正: effect row が type application の invariant 引数に現れた場合だけ、下側を row item/tail の
    ordinary effect 型 union として下ろし、上側を concrete effect を引ける stack 付き proxy にする。
  - `std::io::file.open_text/open` は
    `ref(file|'[file]|... , str|...)` ではなく `ref file str` と表示されるようになった。
  - `std::control::var::var.var_ref` も `ref(var|'[var]|..., ...)` ではなく `ref(var 'a, 'a)` と表示される。
- `poly::dump` は `T|'a&T` のように同じ concrete に pin された invariant bounds を surface 上 `T` として表示するようにした。
  - `ref` payload の `Neu::Bounds` 形は selection probe のため内部には残すが、surface には不要な中心変数を出さない。

## 未注釈 callback 返りeffectの Empty stack cleanup

- `my h(x, f) = f x` や `std::io::file.open_in` の返りeffectに
  `stack('a, { #id -> pop(0)[Empty] })` が surface / raw scheme へ残っていた件を修正した。
  - 原因: 未注釈 local callee の返りeffect保護として入る `push Empty` と、
    定義ラムダ境界側の `pop` が compact / generalize 前に完全には相殺されず、
    callback の返りeffect変数の共変側だけに `Empty` entry が残っていた。
  - 修正: scheme 化直前の stack cleanup で、`push Empty` だけを持つ `(型変数, #id)` が
    同じ scheme の反変側に stack なしで普通に現れている場合、その entry を内部境界として落とす。
    id 全体ではなく `(var, id)` 単位で落とす。
  - `my h(x, f) = f x` は
    `'a -> ('a -> ['b] 'c) -> ['b] 'c` になる。
  - `std::io::file.open_in` は
    `(... -> ['a] 'b) -> ['a] 'b` になり、`dirty stack schemes: 0` になった。
- 回帰テスト:
  - `empty_stack_entry_with_plain_negative_var_is_internal`
  - `unannotated_callback_return_effect_surfaces_without_empty_stack`
