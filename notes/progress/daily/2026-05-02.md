# 2026-05-02

## Runtime specialization redesign

`tasks/current.md` を作成した。現在の方針は、`yulang-runtime` の demand-driven `monomorphize` を直接高速化する前に、infer/lower から runtime へ渡す型情報の責務を整理すること。

初期調査では、`crates/yulang-infer/src/solve` は union-find 型の等価クラスを持っていないことを確認した。型変数制約は主に `Infer.lower` と `Infer.upper` に格納され、`Pos::Var(a) <: Neg::Var(b)` でも `a = b` として結合はされず、互いの上下界として記録される。

commit 時には `compact_type_vars_in_order_profiled` で compact scheme を作り、`coalesce_by_co_occurrence_with_role_constraints` が後処理として型変数を消す/寄せる。これは export/lower が参照できる等価 relation ではなく、scheme の見た目を整える置換に近い。

ユーザーから、共起分析と極性消去こそが今回の肝だと確認した。極性消去で消えた変数は、その場所にあった型と同一視できる。共起分析で `Some(to)` になった変数は、その対象内では `from = to` として扱える。

重要なのは、これはグローバルな等価クラスではないこと。異なる変数や異なる export 対象では、同じ型変数が消えたり消えなかったりする。次は `coalesce_by_co_occurrence_with_role_constraints` が作る置換を、対象ごとの simplification relation として保持する設計を考える。

`crates/yulang-infer/src/simplify/cooccur.rs` に `coalesce_by_co_occurrence_with_role_constraints_report` を追加した。既存 API は維持しつつ、各 round の `subst: HashMap<TypeVar, Option<TypeVar>>` を観測できる。単体テストで、共起による `Some(to)` と極性消去による `None` を確認した。

runtime の責務について方針を補正した。runtime から推論的な処理を全部消すとは限らない。特に effect / thunk / `bind_here` の配置は、infer では軽くしか分からない可能性があるため、VM の実行形に近い側で扱ってよい。一方で、上流で分かっていた型変数関係を runtime が demand-driven に再推測する量は減らしたい。

効果確認の baseline として、`examples/07_junction.yu` は 13 mono passes / 10 specializations、`examples/10_effect_handler.yu` は 9 passes / 1 specialization、`test.yu` は 37 passes / 23 specializations だった。

`export_relevant_type_bounds_for_tv` で単独 tv ごとに coalesced bounds を使う実験をしたが、`examples/07_junction.yu` で runtime specialization 後も `std::flow::sub::sub` が多相のまま残って失敗した。変更は戻した。これは、per-tv に雑に coalesce するのではなく、apply evidence などの対象単位で同じ simplification relation を共有する必要があることを示している。

代表型を取得する report を追加した。`CoalesceRound` は `subst` に加えて `representatives: HashMap<TypeVar, CompactType>` を持つ。代表型は、その round の下界側で型変数と同居していた concrete part から集める。`a | int` のような compact type で `a` が極性消去される場合に、`a` の代表型として `int` を取得できることを単体テストで確認した。

代表型 rewrite は通常経路には入れず、`YULANG_USE_COALESCE_REPRESENTATIVES=1` の opt-in にした。デフォルトでは `cargo test -p yulang-infer` が 273 tests / doc tests ともに通る。

`YULANG_USE_COALESCE_REPRESENTATIVES=1` だけでは、`examples/07_junction.yu`、`examples/10_effect_handler.yu`、`test.yu` の runtime `mono_passes` / `specializations` は変わらなかった。

apply evidence の callee / arg / result を同じ compact 対象で coalesce する opt-in 経路として、`YULANG_COALESCE_APPLY_EVIDENCE=1` を追加した。代表型なしの共起共有だけでは pass 数は変わらなかった。

`YULANG_COALESCE_APPLY_EVIDENCE=1 YULANG_USE_COALESCE_REPRESENTATIVES=1` では runtime lower の型不一致が出た。`arg` slot に関数型の代表が流れるなど、代表型を evidence 全体へ雑に流すのは強すぎる。`Some(to)` ではなく `None` で消える変数だけに代表型を絞っても、まだ型不一致は残った。

次は代表型を「変数ごと」ではなく「evidence slot ごと」に決める必要がある。slot の極性と期待形に合う concrete part だけを採用し、callee の関数代表が value arg slot に流れないようにする。

主型 + 型代入表の実験経路を追加した。`core_ir::ApplyEvidence` は `principal_callee: Option<Type>` と `substitutions: Vec<TypeSubstitution>` を持つ。`YULANG_EXPORT_APPLY_SUBSTITUTIONS=1` で export 側が callee の主型と、arg/result から推定した型変数代入を evidence に載せる。`YULANG_USE_APPLY_SUBSTITUTIONS=1` で runtime monomorphize が、その代入済み callee 型を hint として読む。

`--core-ir --verbose-ir` では `principal=...` と `subst=[t := ty]` を表示するようにした。`examples/07_junction.yu` では実際に `subst=[t4448 := int]` や `subst=[t4144 := std::list::list<int <: _>]` のような代入が出た。

この経路は壊れず、`YULANG_EXPORT_APPLY_SUBSTITUTIONS=1 YULANG_USE_APPLY_SUBSTITUTIONS=1` で `examples/07_junction.yu`、`examples/10_effect_handler.yu`、`test.yu` は通った。ただし `mono_passes` / `specializations` はまだ変わらない。次は demand collection / generic call closure を、evidence 型の再推測ではなく、代入済み主型の param/ret 参照へ寄せる必要がある。

generic call closure の入口を、opt-in 時は代入済み主型から param/ret hint を作るように変更した。`closed_call_signature` は head の runtime 型だけでなく、`principal_callee + substitutions` 由来の引数列を seed hint として受け取れる。`cargo test -p yulang-runtime` は通った。

ただし `examples/07_junction.yu`、`examples/10_effect_handler.yu`、`test.yu` の `mono_passes` / `specializations` はまだ変わらない。次は child demand を queue に積む段階、または collect/check/emit/refine のどこで次 pass に持ち越されているかを測る必要がある。

demand collection と specialization emit の missing demand key 生成も、opt-in 時は代入済み主型の param/ret hint を seed として使うようにした。これでも pass 数はまだ変わらない。

`YULANG_DEBUG_DEMAND_CHECK=1` で見ると、`Fold.fold` の call hint 自体は `list[int]` まで閉じている。一方で `std::list::fold_impl` へは別経路から open demand が入り、`list_view[Hole]` と `Thunk[..., list_view[Hole]]` の不一致で一度失敗している。次は、この open demand が initial collect / child demand / missing demand のどこで生まれるかを切り分ける。

`YULANG_DEBUG_DEMAND_SOURCE=1` を追加した。demand source を `initial collect` / `pending missing` / `checked child from ...` として出せる。これで `std::list::fold_impl` の open demand は、主に `fold_impl` 自身の recursive child demand から出ていることが分かった。

閉じた親 demand を check している self-recursive child demand は、親の demand signature にそろえるようにした。また、同じ target に閉じた demand がすでに queue にある場合、後から来る open demand は積まないようにした。落とす条件は、既存の closed signature が open demand の具体部分を覆う場合だけに絞った。これで `examples/07_junction.yu` の `fold_impl` / `view_raw` open demand は消えた。

ただし `mono_passes` は 13 のまま。`SpecializationTable::seed_existing` の debug では、最終的な `fold_impl__ddmono13` / `view_raw__ddmono14` は既存 specialization として seed されていた。残っている主因は seed 失敗ではなく、role-specialization の 1 周目で作った specialization の参照が、その周の後半だけでは安定しきらず、2 周目で新規 specialization なしの rewrite/refine がもう一度走ること。

role fixpoint に `canonicalize-specializations` を追加する実験もしたが、収束周回は変わらず pass 数が 13 から 15 に増えたため戻した。重複 specialization 名を canonicalize するだけでは、今回の 2 周目は消えない。

`YULANG_EXPORT_APPLY_SUBSTITUTIONS=1 YULANG_USE_APPLY_SUBSTITUTIONS=1` で再計測した。`examples/07_junction.yu` は `[0] 1` で `mono_passes: 13, specializations: 10`、`examples/10_effect_handler.yu` は `[0] (9, "3\n6\n")` で `mono_passes: 9, specializations: 1`、`test.yu` は `[0] 4` で `mono_passes: 37, specializations: 23`。open demand は消せたが、pass 数の改善はまだ出ていない。

`YULANG_DEBUG_MONO_CHANGED=1` を追加し、pass ごとに変化した binding/root を見られるようにした。`examples/07_junction.yu` の role-specialization 2 周目では、`std::junction::junction::{all__ddmono9, any__ddmono10}`、`std::list::&impl#187::fold__ddmono11`、`std::list::fold_impl__ddmono13` の 4 binding だけが変わる。新規 specialization はない。次は、この 4 binding で demand-specialize がどの call path を変え、refine-types がどの type slot を更新するかを見る。

方針をさらに補正した。今必要なのは、既存 demand-driven monomorphize の不動点を少し減らすことではなく、主型に対する型代入だけで specialization を作る新しい 1-pass 経路を用意すること。単一化できる場所は素直に同じ型として扱い、単一化できない境界にだけ `Coerce` / `Thunk` / `BindHere` を入れる。既存の demand / refine / role fixpoint は fallback として残し、新パスで処理できる範囲が増えるほど後半 pass が使われなくなることを測る。

`tasks/current.md` に `SubstitutionSpecialize` の方向性を追加した。最小実装は、`ApplyEvidence.principal_callee + substitutions` がある direct generic call だけを対象にし、`target + normalized type substitutions` を key にした specialization table と、型代入を body / scheme / evidence / runtime annotation に適用する clone emitter から始める。
