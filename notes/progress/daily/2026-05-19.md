# 2026-05-19

- Added an MMTk allocation profile for functional-language GC tuning probes.
  `MmtkHeap` now records total object count, trace slots, raw payload bytes,
  total object bytes, max trace slots, max raw payload bytes, object-kind
  buckets, and compact storage-shape buckets as allocations happen.
- Generated MMTk executable harnesses can dump the profile with
  `YULANG_MMTK_CPS_HEAP_STATS=1`. The lines are prefixed with
  `[JIT-MMTK-STATS]` and are no-ops unless the env var is set.

    899 objects and 4,872 trace slots.
  - `.list.say`: heap objects=12,459, trace_slots=58,165; string has 5,984

- Started the CPS optimization iteration loop with an implementation note at
  `notes/design/cps-optimization-pass-plan.md`.
- Added unused continuation environment slot trimming. The pass accounts for
  continuation entries referenced by closure/thunk creation, handler envs, and
  resume targets before removing environment slots, then reindexes remaining
  slots. The 20x20 traces show `removed_unused_environment_slots=1` for
  `.list.len` and `=2` for `.list.say`; generated-executable hyperfine does not
  show a clear wall-clock win from this small pass alone.
- Added `\case:` / `\catch:` shorthand parsing and lowering. The syntax is
  restricted to no-space `\case` / `\catch`, then lowers to a one-argument
  lambda whose receiver becomes the existing `case` scrutinee or `catch`
  computation. Parser and VM regressions cover both forms.
- Added labeled recursive `case` / `catch` shorthand:
  `case 'label x:`, `catch 'label action:`, `\case 'label:`, and
  `\catch 'label:`. The labeled expression forms lower to a recursive
  one-argument function applied to the target expression; the lambda forms
  return that recursive function. Catch labels are tested with shallow-handler
  resume behavior by explicitly calling the label again after `k`.
- Added the `...` YadaYada operator as a nullfix expression. It lowers to a
  `never`-typed primitive, type-checks in unreachable positions, and raises a
  VM error when evaluation reaches it.
- Started the parser-combinator stdlib surface for `rule` / `~"..."` desugar.
  `std::parse` now defines the thin `[parse]` effect kernel (`item`,
  `position`, `fail`, `commit`) plus first-order combinators for `choice`,
  `read`, `satisfy`, optional parsing, greedy repetition, and lazy repetition.
- Added the first `std::parse::run` / `run_full` handler slice. It handles
  `item`, `position`, `fail`, and `commit`, returning
  `result (value, position) parse_error`. `choice` is an ordinary combinator
  built with a local shallow handler that tracks whether the current parser
  branch committed or consumed input. It retries the right branch only when the
  left branch fails without consuming input and without commit; committed
  failures propagate to outer choices.
- Reported and fixed a recursive local handler specialization bug. A program
  with two local variable handlers and a recursive function could specialize
  the same `run : init -> result -> result` handler with a stale contextual
  output (`bool`) while the input continuation result forced `result = int`.
  Principal-unify now refines contextual specializations from input shapes
  before accepting output hints, ignores conflicting output hints when inputs
  determine the result, carries those shapes through active recursive rewrites,
  and refreshes rebuilt apply arguments before using them as specialization
  evidence. Added a VM regression for the two-handler recursive case.
- Added a first-class `char` surface while keeping the VM representation as a
  one-grapheme `StringTree`. `str[index]` now has type `std::char::char`;
  `std::char` exposes equality, `to_string`, whitespace, punctuation, and broad
  word-token predicates.
- Switched that `char` surface from Unicode scalar values to extended grapheme
  clusters. `StringTree` now counts/indexes/slices by grapheme clusters while
  retaining byte length metadata. MMTk compact string helpers use the same
  grapheme boundary rules.
- Extended `std::parse` with `peek`, `token`, and `word`. `item` now returns
  `char`; `read` remains as a `token` alias. `word` uses `peek` so it can stop
  before punctuation or whitespace without consuming the delimiter. VM
  regressions cover char `item`, non-consuming `peek`, `token`, and broad
  Unicode word parsing.
- Added the first parser-sugar lowering slice. Plain `~"abc"` and a single-token
  `rule { "abc" }` lower to parser values equivalent to
  `\() -> std::parse::token "abc"`. Unsupported rule-literal capture /
  interpolation and richer rule bodies intentionally do not become successful
  parsers yet; they lower to the existing `...` runtime-error primitive until
  the full combinator desugar is implemented.
- Extended the parser-sugar slice with capture-only `~":name"`. It lowers to a
  parser value that runs `std::parse::word()`, binds the result, and then returns
  `{ name: value }`, so effectful parsing stays outside record-field lowering.
- Generalized rule-literal capture lowering to mixed token/capture sequences
  like `~":category/:article/index.html"`. The desugar runs each token or word
  parser in source order, then returns a record containing the captured fields.
- Split `std::parse::run` from positional prefix parsing. `run_with_pos` keeps
  the old `(value, pos)` result, while `run` now requires full consumption and
  returns only the parser value. `run_full` remains as a compatibility alias.
- Extended `rule { ... }` lowering so string literals are direct parser items
  and `name = parser` captures the parser's returned value into the final
  record.
- Added the first `rule { ... }` PEG operators. Alternation lowers to
  `std::parse::choice`, and `+` / `*` plus lazy and optional variants lower to
  the stdlib repetition combinators. Literal `token` now checks with `peek`
  before consuming, so failed first-character alternatives can backtrack.
- Collected the CPS-optimization paper recommendations from ChatGPT Pro into
  `papers/`. Downloaded 10 PDFs (Appel's *Compiling with Continuations* book
  was skipped as it is commercially published) and ran marker to produce
  markdown copies under `papers/md/`. `papers/README.md` indexes the papers
  with priority and the implementation-pass plan: CPS simplifier ->
  administrative continuation elimination -> continuation parameter trimming
  -> shot-aware inlining -> handler/delimiter elimination via effect evidence
  -> known-closure reification -> direct-style/SSA island extraction. First
  reading order: Flanagan + Kelsey/Appel-SSA + Bruggeman as the core axis,
  then Plotkin/Pretnar + Koka + Retrofitting OCaml for the handler-elimination
  work.
- Added a CPS deep-force chain collapse pass. Since `ForceThunk` already peels
  nested thunks to a non-thunk value, consecutive `ForceThunk` statements can
  be represented by one force when the intermediate result has no other use.
- Added a second force simplifier for structurally known non-thunk values whose
  force result uses stay in the same continuation tail. On
  `(each 1..20 + each 1..20).list.say`, the two force simplifiers remove 55
  force statements, reduce generated native `force_thunk` helper calls from
  37368 to 27349, and measured no-collapse 679.0ms, collapse-only 672.0ms, and
  final 637.6ms with generated-executable `hyperfine --warmup 2 --runs 10`.
  Added `YULANG_CPS_DISABLE_FORCE_CHAIN_COLLAPSE=1` for isolated before/after
  measurements.
- Extended redundant non-thunk force removal with a primitive-result whitelist.
  Scalar/string/bytes/path/list-construction and range primitives are treated as
  known non-thunk results, while `ListIndex` is kept force-required because it
  can surface a stored thunk handle. The 20x20 trace now reports
  `optimized_statements=283`, `collapsed_force_thunk_chains=28`, and
  `removed_redundant_non_thunk_forces=42`; generated native `force_thunk`
  helper calls are down to 23266. The latest generated-executable comparison
  measured no-collapse 662.4ms and primitive-whitelist final 652.4ms, with
  visible run-to-run noise.
- Added interprocedural direct-return non-thunk evidence. A direct call result
  can now feed redundant-force removal when all direct return continuations in
  the callee return known non-thunk values; thunk/closure entry continuations
  are excluded from that proof because they run under a separate call protocol.
  On the 20x20 case this moves `removed_redundant_non_thunk_forces` to 56,
  `optimized_statements` to 269, and generated native `force_thunk` calls to
  20382. It also changes all three ready finite thunk rewrite seeds from
  `post_call_force_chain=1` to `=0`, leaving handler/effect-boundary blockers
  as the next real defunctionalization target. The unsafe ready-finite inline
  gate now only body-clones seeds classified as `CalleeBodyClone`, so
  `DefunctionalizeFiniteHandler` seeds no longer trigger the known invalid
  callee clone experiment.
- Added dynamic effect thunk rewrite plans. The 20x20 ready finite seeds now
  classify as `DefunctionalizeFiniteHandler`, with body cloning blocked by
  callee handlers, effectful boundaries, and recursive callee shape for the
  recursive seeds.
- Extended those finite rewrite plans with concrete schedule coordinates. The
  analysis now carries callee effect boundaries, handler arm entries, dynamic
  `Perform` continuations, resume continuations, and locally visible resume
  argument literals from thunk argument plans through specialization seeds into
  rewrite plans. The 20x20 trace now shows the three ready finite plans with
  four finite arms, nine performs, fourteen resumes, two resume actions, and
  `Bool(true),Bool(false)` action literals, while still classifying the real
  rewrite as `DefunctionalizeFiniteHandler` rather than a body clone.
- Added an opt-in finite-handler thunk contification experiment behind
  `YULANG_CPS_ENABLE_FINITE_HANDLER_DEFUN=1`. The safe slice only rewrites a
  ready finite plan when the known local thunk body has no effectful
  terminators: the callee and thunk region are cloned into the caller and the
  cloned `EffectfulForce` boundary becomes a direct continuation edge. Trying
  the same shape on the 20x20 effectful thunk body was incorrect: both a direct
  edge and an extracted ordinary `EffectfulCall` lost the force boundary's
  blocked-force / return-frame contribution, causing the branch resume boolean
  to reach list code as a list pointer. The pass now rejects effectful thunk
  bodies; the next 20x20 rewrite needs explicit IR support for that force-frame
  evidence.
- Followed up with an uncommitted `EffectfulKnownThunkCall` experiment and
  reverted it after testing. Calling an extracted thunk-body continuation with
  the force-frame push/eval protocol was still insufficient. Adding the
  boundary thunk's active-blocked context preserved the `branch` blocked-effect
  selection, but the run still leaked the branch resume boolean into
  `list_len`. The missing piece is not only active-blocked evidence; it is the
  full `force_native_i64_thunk_step` restoration behavior around handler /
  guard snapshots, return-frame routing, eval context, abort/scope-return
  state, and nested thunk forcing. The next attempt should factor that protocol
  into a reusable runtime/codegen primitive before the IR rewrite uses it.
- Tried the next known-thunk runtime-primitive shape and found a stricter
  blocker: even reusing `force_native_i64_thunk_step` does not make it safe to
  clone the effectful callee that owns handler installs and return-frame owner
  metadata. The 2x2/20x20 case still leaked the branch resume bool into list
  code while `defunctionalized_finite_handler_thunk_calls=1`. The destructive
  pass is therefore back to the conservative effectful-thunk rejection; the
  design note now records that the next real finite-handler defunctionalization
  must consume the finite schedule in place, or first reify handler-owner /
  return-frame ownership explicitly in the IR.
- Added a small runtime-side display optimization for `.list.say`: `ListTree`
  now exposes `for_each_ref`, and native CPS list display/debug rendering walks
  the tree by reference instead of cloning all elements with `to_vec()` and
  building an intermediate `Vec<String>` before `join`. On the generated
  20x20 `.list.say` executable, a same-machine `hyperfine --warmup 2 --runs 10`
  comparison measured baseline 633.3ms and streaming render 632.8ms, i.e. no
  material wall-clock win but a little less system time.
- Broadened `fold_structural_projections` into a larger local structural
  simplifier. It now tracks exact tuple / record / variant values inside one
  continuation and folds tuple gets, record selects / defaults / has-field, and
  variant tag / payload projections without looking at std paths. Projection
  aliases are skipped when the destination is captured by another
  continuation environment, because that would need environment-slot rebasing.
  Added a direct regression for record + variant folds. The 20x20 hot path does
  not currently expose local structural projection opportunities after earlier
  passes, so this improves the generic optimizer surface rather than that
  benchmark's profile counters.
- Added a native CPS runtime fast path for thunk checks / force calls: words
  that cannot be aligned control pointers (`0` or low bits set) return before
  touching the thunk registry or MMTk control thunk path. This targets the
  many scalar `force_thunk` calls left in nondeterministic list code. The
  current generated 20x20 `.list.say` executable measured 625.4ms ± 18.1ms
  over 10 hyperfine runs after the fast path.
- Added Koka-style read-only effect row evidence. Boundary rows now preserve
  duplicate handler layers by selecting the nearest active matching handler arm
  and reporting unmatched dynamic performs as open effects. Because a bare
  `EffectfulForce` boundary has an unknown value target, the boundary row for
  20x20 remains conservatively open. The callsite thunk-row evidence uses the
  existing thunk argument / finite schedule seeds to recover the useful row:
  all three 20x20 `list(thunk)` sites are now traced as `ClosedFinite`, with
  `branch` as the finite two-resume effect, `reject` as no-resume, and no
  blocked effect. This is still a proof/instrumentation pass, not the
  destructive handler defunctionalization itself. The generated 20x20
  executable still sits in the same noisy range after this read-only pass:
  `hyperfine --warmup 2 --runs 10 target/yulang/bin/bench_20x20_list_say-effects`
  measured 632.8ms ± 12.0ms.
- Reduced one runtime control-stack cost in dynamic `Perform` handling.
  `yulang_cps_select_handler_i64` now searches a non-empty active handler stack
  in place and truncates it directly after saving the full pre-arm stack,
  instead of materializing a cloned stack and then replacing the active stack
  with a cloned prefix. Resumption creation also stores the active handler
  snapshot directly. The 20x20 stats show handler-stack replace calls dropping
  from 8048 to 7187 and replace items from 166542 to 148692; the generated
  executable measured 628.7ms ± 12.1ms over 20 runs, still within the noisy
  band but with less handler-stack copying.
- Added explicit handler-owner / return-frame ownership metadata to CPS IR.
  `EffectfulForce` and direct thunk-producing callsites can now carry the owner
  function, return-frame resume, forced thunk value, finite handler layer
  schedule, no-resume effects, blocked effects, and open effects. This is the
  IR-side version of the Koka duplicate-effect lesson: the optimizer records
  which nearest handler layer owns the boundary and which effects remain open,
  instead of treating `branch` or `reject` as globally special names.
- On the 20x20 `.list.say` trace, the generic `std::undet::undet::list__mono3`
  force boundary still stays open because an unknown forced thunk can write to
  `std::out::out::write`, but the three concrete `list(thunk)` callsites now

  `annotated_effectful_force_ownerships=2` and
  `annotated_direct_call_thunk_ownerships=3`. This is still a proof-carrying IR
  change, not the destructive handler lowering yet.
- Added the first ownership-consuming destructive slice. The optimizer now
  creates an owned clone of the dynamic perform function when a forced thunk
  callsite has closed finite ownership evidence, and rewrites matching
  `EffectfulCall`s inside the known thunk region to that clone. For 20x20 this
  specializes `std::undet::undet::each__mono1` once and rewrites the root
  thunk's two `each` calls. The clone's `branch` `Perform` carries the concrete
  `std::undet::undet::list__mono3` owner / handler / arm entry in IR.
- Cranelift consumes that perform ownership by calling an exact owned-handler
  selector and then lowering directly to the known handler candidate instead of
  running the generic selected-handler candidate dispatch. The owned selector
  has a top-handler restore path so the common case avoids cloning the whole
  active handler stack just to restore it after the arm returns.
- This pass is correct on the 20x20 `.list.say` native run and exposes
  `specialized_owned_perform_functions=1` /
  `rewritten_owned_perform_call_sites=2` in the optimizer trace. A direct
  generated-executable comparison is currently neutral rather than a win:
  with `YULANG_CPS_DISABLE_OWNED_PERFORM_SPECIALIZATION=1`, 10-run hyperfine
  measured 636.0ms ± 11.3ms; with the pass enabled it measured
  635.6ms ± 8.4ms. The path is now real and ownership-driven, but the next
  speed step needs to remove the runtime resumption/handler-arm protocol for
  finite schedules, not only the candidate dispatch.
- Added the first destructive handler-defunctionalization kernel. The pass
  rewrites same-function finite handler regions when a local `Perform` targets
  a handler arm whose resumption pointer is only used by terminal `Resume`
  statements: the `Perform` becomes a direct `Continue` to the arm, and the
  terminal `Resume` statements become direct `Continue`s to the original resume
  continuation. Uninstalled static handler arms are removed when their matching
  local performs have disappeared, which keeps Cranelift handler-entry lowering
  consistent. A unit regression fixes the direct `Perform`/`Resume` shape.
  The 20x20 `.list.say` trace still reports
  `defunctionalized_local_finite_handler_regions=0`: its useful finite row is
  interprocedural (`each__mono1` owns the dynamic `Perform`; `list__mono3` owns
  the handler arm and `EffectfulForce` boundary), so the real benchmark pass
  still needs forced-thunk/callee specialization or explicit handler-owner /
  return-frame evidence in the IR.
- Added a transactional owned-perform defunctionalization attempt for the
  interprocedural clone path. It can delete an owned `Perform` only when the
  cloned owner arm region contains the terminal `Resume` uses directly in the
  reachable arm continuations; failed attempts now leave the owned perform
  clone untouched and fall back to the previous owned-selection lowering. The
  20x20 `.list.say` case still does not qualify: its two `Resume` uses are in
  local thunks created by the `list` handler arm, so a naive rewrite to
  `each__mono1`'s raw perform resume drops the `EffectfulForce` return-frame
  contribution and lets scalar values reach list merge. This confirms that the
  next real pass needs to reify or inline the return-frame-owned resumption
  composition, not just clone the handler arm into the callee.

  `CpsEffectPerformOwnership` carries the owning `.list` force return-frame

- Split owned-perform resumption construction from generic resumption
  construction in Cranelift/runtime. Owned performs now call
  `yulang_cps_make_owned_resumption_i64_*`, carrying owner function id, owner

  runtime allocation point. The runtime shape is still the same
  `NativeCpsI64Resumption`, so this is a safe composition hook rather than the
  final deletion of the resumption object.
- Removed a duplicate handler-stack clone in the perform path by storing the
  selected outer handler stack as a persistent handler snapshot instead of a
  fresh `Vec`. Handler selection can now share the snapshot that
  `make_resumption` just produced; restore uses `replace_snapshot` to preserve
