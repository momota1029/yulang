# 2026-06-20

## Infer stack overflow

- `cargo test -p infer` を止めていた stack overflow は、constraint bounds replay の var-to-var direct replay が同期再帰し、同じ `SubtractId` の裸 `pop` を cycle ごとに増やすことが原因だった。
- replay action の適用を明示 stack に移し、Rust stack を消費しないようにした。
- var-to-var alias edge では、同じ `SubtractId` の裸 pop-only count を 1 に正規化するようにした。
  - 実際の nested annotation は別 `SubtractId` を使う。
  - 同一 id の `pop(n)` は alias cycle が同じ predicate boundary を再消費した結果なので、新情報として扱わない。
- `infer` full suite は `447 passed` まで通るようになった。

## LSP safety guard

- LSP 全体が落ちて見える直接原因は、`didOpen` / `didChange` が full std diagnostics を await し、full std lowering が 90 秒でも返らないため request loop 全体を塞いでいたこと。
- `YULANG_INFER_TIMING=files` では full std route が `lower binding bodies` 中で長時間返らない。短い timeout では `std::data::list` や `std::text::str` 付近に見えるが、少なくとも `target/debug/yulang --std-root lib check <small file>` は `timeout 90s` に到達する。
- server 側は diagnostics を background task 化し、同時実行 slot と timeout guard を入れた。hover / definition にも request slot と timeout を入れた。
- busy / timeout は source diagnostic として publish しない。busy は黙って既存 diagnostics を残し、timeout / worker failure は LSP log にだけ出す。
- hover / definition は std route を優先し、std route が `Err` の時だけ local/no-prelude fallback を使う。
  local/no-prelude を先に採用すると、std/prelude の role 解決が必要な `+` / `*` を含む定義で `next_area: never` のような壊れた hover が出るため。
  debug server では full std hover が 2 秒を超える例があるため、request timeout は 5 秒へ広げた。

## LSP std file route fix

- Zed で `lib/std/core/ops.yu` を開き、`label.next()` から definition へ飛ぶと `pub prefix(next)` 付近へ行くことがあった。
- 直接原因は、std root 配下のファイルを `--std-root lib` 付きで開いたとき、同じ file が root module と `std::...` module の両方として collect されて `DuplicateModuleFile` になり、server が local fallback へ落ちていたこと。
- std root 配下の opened file は root source として追加せず、std module tree 内の該当 module の source を置き換えるようにした。
- `check_loaded_files` は root module を要求するため、推論器用に空の synthetic root だけ足す。cursor/hover/definition/references の focus module は開いている `std::...` module のままにする。
- std file route は implicit prelude を opened source に足さないため、LSP range 変換でも root offset subtraction を無効化した。local fallback も raw range として扱う。
- 追加 regression:
  - `cargo test -q -p yulang source::tests::case_02::definition_entry_source_with_std_reports_std_method_target -- --test-threads=1`
  - `cargo test -q -p yulang server::tests::definition_for_source_reports_std_file_method_target_location -- --test-threads=1`
  - `target/debug/yulang --std-root lib server` を JSON-RPC で起動し、`ops.yu` の `label.next()` が `std/control/flow.yu` の role member `next` へ飛ぶ smoke を通した。

## LSP rename slice

- `textDocument/prepareRename` と `textDocument/rename` を有効化した。
- rename は source/check 層で cursor target を `DefId` に解決し、references と同じ target 集合から declaration + use-site の `SourceTextEdit` を作る。
- LSP 側は `SourceRename` を `WorkspaceEdit.changes` に変換する。root source は implicit prelude offset の有無に応じて range を変換し、他 file は disk source から range を計算する。
- 初期対象は value reference、lambda/local arg、method selection。record field selection はまだ target `DefId` がないため対象外。
- 新しい名前は Unicode XID identifier + optional `?` / `!` suffix のみ受け付け、keyword は拒否する。
- 追加 regression:
  - `cargo test -q -p yulang source::tests::case_02::rename_entry_source -- --test-threads=1`
  - `cargo test -q -p yulang source::tests::case_02::prepare_rename_entry_source_reports_target_range -- --test-threads=1`
  - `cargo test -q -p yulang server::tests::rename_for_source -- --test-threads=1`
  - `cargo test -q -p yulang server::tests::prepare_rename_for_source_reports_reference_range -- --test-threads=1`
  - `target/debug/yulang --std-root <temp-lib> server` を JSON-RPC で起動し、`prepareRename` と `rename` が期待 range / 2 edits を返す smoke を通した。

## Effect stack replay hang

- full std check の停止は、`e37e375d Fix predicate stack pop composition` で入った裸 `pop(n)` の保持が原因だった。
- `c6ec4923` / `861c6fdb` は release `--std-root lib check` が 0.4 秒台で完了するが、`e37e375d` 以降は `std::data::list` lowering 中に同一 subtype cycle が `pop(3555)` のような有限 count を増やし続け、`SubtypeConstraint` の `seen` が毎回別物として扱っていた。
- `f7fd1238` の iterative replay は Rust stack overflow を消したが、同じ重み増殖は残っていたため timeout になっていた。
- bounds replay の正規化では、同じ `SubtractId` の有限な裸 `pop(n)` を `pop(1)` へ丸める。`pop(1)` は surface に残る predicate boundary として保ち、`u32::MAX` sentinel や stack/floor 付き未対応 pop の扱いは変えない。
- release `target/release/yulang --std-root lib check <small file>` は `total: 494.8ms` に戻った。
- `timeout 240s cargo test -q -p yulang -- --test-threads=1` は 145 tests + 16 tests で通る。
