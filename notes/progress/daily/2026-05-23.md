## yulang-runtime-finalize crate skeleton

`crates/yulang-runtime-finalize` を追加し、workspace member に入れた。

初手として、旧 `yulang-runtime` の monomorphize pipeline にはまだ接続せず、runtime
finalization 専用 crate の最小核だけを置いた。

- public entrypoint: `finalize_module`
- output/report: `FinalizeOutput` / `FinalizeReport`
- top-level demand surface: `TopLevelDemand`
- monomorphic instance identity: `InstanceKey`
- principal type call-site solver: `PrincipalGraph::from_binding` /
  `PrincipalGraph::solve_call`
- structured diagnostic: `FinalizeDiagnostic`

現時点の solver は、1 引数分の declared principal function type を call-site の argument
lower で解き、closed `InstanceKey` を作るところまでに限定している。body graph、nested
polymorphic call queue、role associated type、effect lane、cast insertion は未接続。

### Runtime finalize root rewrite + executable smoke

新 `yulang-runtime-finalize` の type graph path で、root 式の単相化結果を実際の runtime IR へ
反映するところまで進めた。

- root 式中の `Apply(Var(poly), arg)` を use-site ごとに解き、callee を `poly.monoN` へ差し替える。
- 同じ binding / type substitutions を持つ use-site は同じ alias を再利用する。
  例: `(id 1, id 2)` は `id.mono0` を一つだけ emit し、両方の call が同じ alias を呼ぶ。
- emit 済み alias の body / scheme / runtime type は、式全体、pattern、join evidence、
  handler evidence、coerce、thunk、block まで type substitutions を通す。
- root rewrite 後に参照されなくなった specialized original polymorphic binding は active module から外す。
  これにより、少なくとも今回扱う root-only の単相化結果は VM の residual polymorphism check を通る。
- `my id x = x; id 1` は新 finalizer の出力をそのまま `yulang-vm::compile_vm_module` へ渡し、
  VM 実行結果 `1` まで確認した。
- emit した alias body も worklist に入れるようにした。
  `my f y = id y; f 1` では、まず `f.mono0` を作り、その body 中の `id y` を次の型グラフで解いて
  `id.mono1` を emit / rewrite する。
  `y` の runtime type が hole の場合は、単相化済み binding scheme / lambda scope から得た param 型を
  local lower bound として使う。
  この finalized module も VM 実行結果 `1` まで確認した。
- apply の単相化を、`param` と実引数型の直接対応から `callee <: arg -> result_hole` 制約へ寄せ始めた。
  最小核として `my first x y = x; first 1 true` を追加し、`a -> b -> a <: int -> r0` と
  `r0 <: bool -> r1` を型グラフで分解して `a = int`, `b = bool`, `r1 = int` へ落とす。
  source から finalized module を作り、そのまま VM 実行結果 `1` まで確認した。

### Runtime finalize higher-order value arguments

apply spine の引数に多相 binding が値として現れるケースを、新 type graph path に乗せた。

- `twice id 1` では、`twice` の graph とは別に、引数 `id` をその use-site の expected function type
  で解き、`id.mono0` へ差し替える。
- `flip pair 1 2` では、`pair` を値引数として単相化し、root 側は `flip.mono1 pair.mono0 1 2`
  相当へ rewrite する。
- 引数型は、closed expression/local type がある場合はそれを優先し、`Unknown` / type variable を含む場合だけ
  `ApplyEvidence` の `expected_arg` / `arg` bounds を使う。
  これにより `my f y = id y; f 1` のような単相化済み binding body では local `y : int` を優先し、
  `pair` のような root value argument では evidence から関数型を拾う。
- `Never` lower を constraint collection の入口で捨てないようにした。
  これは blind default ではなく、`pair <: f` のような実制約から来た `⊥ <: effect_var` を保持するためで、
  `flip` の effect variable が `Never` へ解ける。
- apply root の result type は、graph の core result だけで上書きせず、元の runtime type shape が closed
  ならそれを保持する。これにより `BindHere` 内側の `Thunk` shape を VM erase 前に壊さない。

追加 regression:

- `my id x = x; my twice f x = f (f x); twice id 1` が VM 実行結果 `1` になる。
- `my pair x y = (x, y); my flip f x y = f y x; flip pair 1 2` が VM 実行結果 `(2, 1)` になる。


### Runtime finalize rewrite archive

`yulang-runtime-finalize` は、旧 `BodyGraph` / `InstancePlanner` / emit / validator ベースの
実装を active `src/` から外し、`archive/2026-05-23-pre-rewrite` に保存した。

active `src/` は型グラフ rewrite の最小核へ戻した。

- `graph`: `TypeGraph` / `PrincipalInstance` / `RuntimeBounds` / `GraphSolution`
- `solver`: root から `RootGraphInput` を集める最小 entrypoint
- `output`: `FinalizeOutput` / `FinalizeReport`

新しい前提では、`Any` は Top、`Never` は Bottom、`Unknown` は hole として扱う。
`Any` を曖昧な未解決型として使わない。vacuous な `Never` lower / `Any` upper は slot を解かず、
実情報になる `Any` lower / `Never` upper は極値として保持する。

型グラフは、主型を fresh 化して graph に入れ、式や引数から得た lower / upper bounds を集める。
solve は下界を優先し、solve 後に各型変数の lower / upper と単相化済み型を同時に得る。

最初の具体例として、runtime IR 手組みの `id 1` と、source lowering から来る
`my id x = x\nid 1\n` が root graph solution まで通るようにした。
現時点では emitted monomorphic binding や root rewrite はまだ行わない。
root `Apply(Var(id), 1)` から `id` の主型を fresh 化し、argument lower を型グラフへ入れ、
`GraphSolution` が complete になり、root result が concrete runtime core type になるところまで確認している。
この regression は `std_root = None` / `implicit_prelude = false` で source lowering するため、
旧 helper のような large-stack thread は不要である。
さらに、同じ root tuple 内で `id` を 2 回使う runtime IR と、source lowering から来る
`my id x = x\nid 1\nid true\n` も通る。各 `id` use は別の `TypeGraph` で主型を fresh 化するため、
片方が int、もう片方が bool でも別々に complete な `GraphSolution` になる。
`GraphSolution` から元の type parameter への substitution も復元し、最小 emit として
`id.mono0` / `id.mono1` のような type parameter なしの binding を module に append する。
まだ root expression rewrite と dedupe はしていない。

### Type graph structural lower profiling

`YULANG_DEBUG_TYPE_GRAPH_SOLVER_PROFILE=1` を追加し、type graph の type-var lower solver が
structural decomposition の round ごとに scoped var 数、候補数、multi-candidate var 数、
最大 type node 数、最大 depth、addition 数、normalize 時間を出せるようにした。

`YULANG_DEBUG_TYPE_GRAPH_PIPELINE=1 cargo test -q -p yulang-vm vm_runs_source_int_float_add_example -- --nocapture`
で発生していた stack overflow は、structural decomposition の追加ではなく、scope 内
substitution の structural cycle を候補 normalize に渡していたことが原因だった。
profile では addition が 0 のまま normalize だけが毎 round 変化し、`t2579` 系候補が
3, 5, 9, ... と指数的に深くなることを確認した。

対処として `retain_acyclic_type_substitutions` を `types/substitution` に追加し、recursive
lower solution と structural normalize の substitution map から自分へ戻る structural cycle
を除外するようにした。cycle は未解決 lower として残し、閉じた代入としては使わない。

さらに structural lower decomposition の検証を、`substitute_type(template) == actual` の
浅い全体比較から、visited pair を持って型の中身を辿る比較へ変えた。これにより
`Union(list<a>, int)` と `list<bool>` のように、choice の構造枝が actual と一致する場合は
`a = bool` を取り出せる。一方で `Union(a, int)` と `bool` のような bare var branch は
曖昧な lower として扱い、代入を作らない。

effect row についても structural comparison を effect head 単位にした。異なる effect item は
無視し、同じ effect head の item だけ中身を比較して lower を取り出す。row tail の residual から
異なる effect item 全体を代入することは避ける。

handler expression は、`HandleEffect.residual_after` を出力側の lower evidence として
type graph に追加するようにした。`residual_after` は lowering が body effect から handled
effect を引き、arm 側 effect を足した型の引き算結果なので、type graph では body の value
type と組み合わせて handler result slot の runtime lower candidate にする。
`RuntimeLower` projection を追加し、core `TypeBounds` では表せない `Thunk[effect, value]`
も下界候補として扱えるようにした。

下界 snapshot の後に upper supplement preview を追加した。これは下界を上書きしない逆側
pass として扱う。下界候補がある slot では upper requirement を満たすか検査するだけにし、
下界候補がない upper-only slot だけ、補足候補として数える。複数の異なる upper 候補がある場合は
補足を確定せず ambiguous として報告する。

最後の穴埋め preview として、下界候補も upper supplement 候補も持たない slot だけを対象に、
value position の `Unknown` / `Any` / `Var` は `()`、effect position の
`Unknown` / `Any` / `Var` は `Never` として数える pass を追加した。

module へ書き戻す実適用 pass も追加した。ただし、裸の value `Var` や関数 param / ret を
即 `unit` に落とすと、後続の apply / local constraint と衝突して `show` / `debug`
composite が壊れることが分かったため、実適用は安全側へ絞った。現時点で実際に書くのは、
構文形状から作れる tuple / record / variant の value 型、apply bounds や let value から
直接取れる value 型に限る。value `Var` は type graph の実代入側に残す。

effect 型の `Never` default は実適用から外し、preview 専用に戻した。legacy runtime surface
上で handler / thunk / AddId の effect hole を先に `Never` へ閉じると、後段の全代入前に
std request や synthetic request を handler から隠してしまう。実際に `Never` を書くのは、
type graph の total substitution が lower / upper 候補を module surface へ反映した後の
本当の最終段に置く。

`vm_runs_source_int_float_add_example` の post-monomorphize preview では
`filled_value_holes=0`、`filled_effect_holes=4` まで閉じた。`show` / `debug` の

effect default を実適用した状態で `cargo test -q -p yulang-vm` を走らせると、

### Runtime finalize if forced effects

`yulang-runtime-finalize` の `If` 処理を effect-aware にした。

source lowering 側では effectful condition に `junction::junction` を噛ませるが、runtime IR の
`If` 自体は残る。そのため finalizer は `junction` を value join として扱わず、`If` の
condition / branch が thunk result を持つ場合に `BindHere` で値へ戻し、そこで発生する forced
effect を `If` 全体の `Thunk` として持ち上げる。

`JoinEvidence.result` は if 全体の value result として materialize する。branch は、その value
result へ流れることだけを見る。`Never` branch は値を返さない branch として受け入れる。
finalizer は branch cast を新規に発明しない。numeric widening や user-defined `Cast` は、lowering /
adapter evidence 側で `Coerce` または通常の cast call として既に入っている前提で扱う。
明示 cast 済みの branch は、その cast 後の型を通常どおり見る。
validator も同じ規則に寄せ、effectful condition が `BindHere` 済みで if 全体が thunk に包まれた形を
受け入れるようにした。finalized output では branch value は result 型へ適合済みであることを見る。

追加テスト:

- `BodyGraph` が effectful condition の `If` を `Thunk(effect, int)` へ持ち上げ、condition を
  `BindHere` にする。
- `BodyGraph` が `Coerce(int -> float)` 済み branch を持つ `If` を `float` result として閉じる。
- validator が同じ effectful if shape を受け入れる。

確認:

```bash

RUSTC_WRAPPER= cargo check -q -p yulang-runtime-finalize
```

はいずれも通る。
