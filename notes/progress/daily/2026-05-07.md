# 2026-05-07

## Compiled Runtime Surface

- Started Phase 4 of the compiled-unit cache path with a conservative
  `CompiledRuntimeSurface`.
- `CompiledUnitArtifact` now bundles syntax, namespace, typed, and runtime
  surfaces under the same unit manifest.
- The runtime surface currently stores a unit-local `CoreProgram` exported from
  the namespace value symbols that belong to the unit.
- `import_compiled_unit_artifact` returns the runtime surface alongside restored
  syntax and typed-ref coverage. This is still a read-only artifact checkpoint;
  it does not yet merge dependency runtime surfaces into a downstream user core
  export.
- Wasm now exposes `std_compiled_unit_artifacts()` so playground build-time
  tooling can inspect the unit-oriented artifact shape instead of the older
  std-only core snapshot shape.
- The wasm artifact export is intentionally not on the normal playground run
  path. It is a persistence/bundling checkpoint for moving std work out of the
  first interactive compile.
- Added `CompiledRuntimeBundle::from_surfaces` as the first runtime/core merge
  checkpoint:
  - dependency core bindings are deduplicated by canonical path;
  - conflicting same-path bindings or runtime symbols are rejected;
  - expected-edge evidence ids are remapped across merged surfaces, including
    references inside binding bodies and principal plans.
- Added `CompiledRuntimeBundle::merge_with_user_program` so cached dependency
  runtime surfaces can be placed before a freshly exported user program.
- Added a wasm regression that removes a dependency binding from the fresh user
  core export, restores it from a compiled runtime surface, then lowers,
  monomorphizes, compiles, and evaluates the merged runtime program.
- Runtime surfaces are now seeded from all binding paths owned by the unit's
  module paths, not just namespace value exports. This keeps generated same-unit
  runtime helpers such as variant constructors while avoiding reexport-owned
  duplicates from other units.
- Added a std-facing wasm regression that bundles source-std runtime surfaces,
  removes those bundled bindings from the fresh program export, merges the
  bundle back in, and evaluates `1 + 2`.
- Added a `crates/yulang-wasm` build script that builds std compiled-unit
  artifacts at package build time and embeds the serialized JSON from `OUT_DIR`.
- Wasm now exposes `embedded_std_compiled_unit_artifacts()` and
  `embedded_std_compiled_unit_artifact_status()`.
- `warm_std_cache()` includes embedded artifact counts, runtime binding counts,
  and artifact byte size in its console-visible warmup output.
- The std runtime-surface merge regression now uses the embedded artifacts
  rather than rebuilding artifacts inside the test body.
- Embedded artifact loading now validates manifest format versions plus
  namespace/typed surface completeness. Status reporting returns `valid=false`
  and a fallback reason instead of panicking, so the normal source compile path
  can remain the conservative fallback.
- The playground logs embedded std artifact status immediately after wasm init,
  and warmup logs include the same cache-artifact status fields.
- Typed surfaces now serialize `CompactTypeScheme` payloads for exported unit
  value schemes instead of only carrying rendered debug strings.
- Typed import installs imported compact schemes into the restored `Infer`
  state and rebuilds frozen schemes from those compact payloads, so downstream
  refs can instantiate cached dependency schemes without re-lowering the cached
  source body.
- Added a multi-unit compiled artifact import path that merges namespace/typed
  surfaces, imports them into one `LowerState`, and carries a merged runtime
  bundle alongside the lowered user state.
- The playground run path now tries embedded std compiled-unit artifacts before
  the process-local source std cache. On a hit it skips std source lowering and
  merges embedded std runtime surfaces into the fresh user core program.
- Artifact execution keeps a conservative fallback: if import, diagnostics,
  runtime merge, VM lowering, monomorphize, compile, or eval fails on the
  embedded path, the run retries through the existing source std cache and
  records the fallback reason in timings.
- The fallback also triggers if embedded runtime execution leaves a root request
  unresolved. This keeps current console/effect/role gaps from changing user
  visible playground results while typed-surface import is still incomplete.
- Namespace surfaces now include same-unit companion modules, fixing imported
  paths such as `std::prelude::Add::add` without pulling unrelated root children
  into entry artifacts.
- Runtime bundle merge now prunes user-exported duplicate dependency runtime
  bindings/symbols before merging cached dependency surfaces.
- Typed surface import now restores role arg info, role method lookup, role impl
  candidates, effect arities, and effect method lookup tables. This lets the
  embedded std artifact path infer downstream code against cached role/effect
  metadata instead of falling back to source std lowering for those tables.
- Namespace surface import now reuses already-installed builtin value/type defs
  when a cached symbol path resolves to a primitive. This keeps imported std
  surfaces aligned with builtin runtime bodies such as `std::int::add`.
- The wasm embedded std path now builds a small builtin runtime surface from
  primitive principal bodies and merges it with embedded std runtime surfaces.
  Bindings already present in the embedded artifact bundle are pruned from this
  builtin surface before merge, avoiding duplicate source-std runtime paths such
  as `std::list::empty`.
- `reports_compile_timings` now requires an embedded compiled-std cache hit with
  no fallback reason for `1 + 2`, so the basic playground run path is covered by
  compiled-unit artifacts end-to-end.
- Merged namespace surfaces now preserve value/type vector order because typed
  surfaces refer to symbols by positional unit ids. Sorting merged symbols by
  path invalidated cross-artifact typed refs.
- Namespace/runtime surfaces now include hidden `&impl#...` helper modules, so
  role impl member defs can be represented in typed surfaces and their runtime
  bindings can travel with the unit artifact.
- `std::undet` list execution no longer fails during embedded runtime lowering:
  cached runtime surfaces had exported both `std::list::&impl#322::fold` and a
  reexported `std::prelude::&impl#322::fold`, making principal role selection
  ambiguous. Runtime alias generation now suppresses aliases for hidden
  `&impl#...` helper paths, leaving the canonical impl candidate only.
- Effect operation runtime symbols are now canonicalized during core export, and
  compiled runtime alias generation skips effect operation aliases. Operation
  paths encode their parent effect path in runtime lowering, so aliases such as
  `std::undet::branch` are not valid runtime duplicates of
  `std::undet::undet::branch`.
- Fixed the `std::undet` embedded fallback. The cached path had specialized the
  reexported `.list` effect-method helper as a value receiver, which let one
  `std::undet::undet::branch` request escape the runtime bundle. Namespace
  symbol selection now
  canonicalizes hidden effect-method paths back under their effect module, and
  imported computation-receiver effect methods use the direct runtime handler
  target when a same-name effect function exists. Runtime aliases for hidden
  effect-method reexports are emitted as direct aliases instead of cloned helper
  bodies, so downstream export no longer specializes the reexport helper as a
  value receiver.
- `runs_undet_list_example` now requires an embedded compiled-std cache hit with
  no fallback reason.
- `each 1..` no longer falls back from the embedded compiled-std path. Core
  graph/runtime modules now carry role impl associated-type projections, and
  runtime principal-plan requirement projection uses those impl facts to close
  cached `std::undet::each` without hardcoding `Fold range -> int`.
- `runs_undet_once_range_example` now requires an embedded compiled-std cache
  hit for a small `.once` range program.
- Added `bench/static_analysis_bench.sh`, a representative CLI benchmark that
  runs examples with infer/runtime phase timings and prints one row per case.
  The first baseline shows static analysis is still dominated by source
  lower/infer (`lower`) plus principal elaboration/monomorphize for larger
  examples:
  - `examples/showcase.yu`: static about 592ms on repeat 1, about 602ms on
    repeat 3; `lower` about 391ms, `mono` about 184ms.
  - `examples/06_undet_once.yu`: static about 414ms on repeat 1, about 404ms on
    repeat 3; VM eval dominates total runtime, but static analysis still spends
    about 348ms in `lower`.
  - smaller examples are roughly 198-223ms static, mostly `lower`.
- Extended the static-analysis benchmark with detail columns for std/entry
  lower, path resolve, suffix lowering, act body bindings, block lowering,
  principal root rewrite, and intern body rewrite. Repeat-3 detail baseline:
  - `examples/showcase.yu`: static about 576ms; `lower` about 382ms with
    `std_low` about 289ms and `ent_low` about 59ms; `mono` about 171ms with
    `root_rw` about 152ms and `intern` about 72ms.
  - `examples/06_undet_once.yu`: static about 405ms; `lower` about 343ms with
    `std_low` about 289ms and `ent_low` about 18ms; `mono` about 41ms.
  - smaller examples are still mostly std source lowering in the CLI path:
    `std_low` about 163-164ms, with entry lower below 9ms.
- `bench/infer_simple.yu` still overflows the stack even with `--infer-only`
  and `RUST_MIN_STACK=67108864`, so it is not a usable speed benchmark yet.
  Treat it as a separate deep-recursion regression candidate.
- Principal unification now returns from cached incomplete-plan hits before
  cloning the generic binding body. This is correct but not a major speedup for
  `showcase`: `root_rw` stays around 153-155ms. The useful next target is the
  repeated principal rewrite of root/specialized bodies, not binding lookup.
- Root-level static-analysis speedup direction:
  1. Add observability for principal rewrite duplication: split apply counts by
     root/body context, target, specialization depth, and repeated visit
     patterns. The goal is to prove which work should become one-shot graph
     work instead of recursive tree rewrite.
  2. Move principal specialization toward a request graph/queue: collect
     `target + substitutions + handler_plan` requests, intern each node once,
     and rewrite callsites after the node exists instead of recursively
     rewriting the same body shape from root traversal.
  3. Start a `NameResolvePass` boundary for lowering: resolve path/operator
     identity before expression lowering so `lower` can consume `DefId`s
     instead of repeatedly searching locals/modules/use paths. This is the
     broader entry/user-source optimization after runtime rewrite duplication is
     measured.
- Added principal rewrite target observability to runtime phase timings. The
  `showcase` hot target is `std::list::index_raw`: about 1948 visits, only 2
  rewrites, about 970 cached incomplete hits, almost all from `root` context.
  Other high-count targets are already specialized or ref helper calls, which
  confirms that root traversal is repeatedly walking large already-known apply
  spines.
- Tried a small specialization-queue prototype by suppressing partial callee
  principal rewrite attempts when the outer apply spine can make the full
  request. It skipped about 1113 partial attempts on `showcase`, but did not
  materially reduce `root_rw`. This points away from partial-plan execution and
  toward splitting principal rewrite into collect/apply phases or memoizing
  expression rewrite/projection by expression identity.
- Moved that partial-callee suppression before target visit accounting and
  generic binding clone. On `showcase`, `std::list::index_raw` visits dropped
  from about 1948 to about 974 and `principal-unify-clone-generic-binding`
  dropped from about 1176 to about 63, but `root_rw` still stayed around
  159-161ms. The remaining cost is not plan lookup/clone; it is the recursive
  expression walk/rebuild itself.
- Added optional `YULANG_PRINCIPAL_REWRITE_KIND_PROFILE=1` profiling for
  principal rewrite expression kinds. In `showcase`, inclusive time is dominated
  by nested `block` / `apply` / `bind-here` / `thunk` / `lambda` rewrites. An
  apply-chain flatten prototype reduced `principal-rewrite-expr-apply` counts
  but made profiled `root_rw` worse, so it was not kept. This reinforces that a
  useful rewrite needs a real collect/apply or expression-identity memo design,
  not just a different recursive traversal shape.
- Added the first real specialization request queue. `intern_specialization`
  now allocates/dedupes the specialized path immediately, but queues binding
  body rewrite into `pending_specializations`; root rewrite processes those
  requests after root expressions instead of recursively rewriting bodies as a
  side effect. On `showcase`, `root-rewrite` moved from roughly 154ms to roughly
  109ms in runtime phase timings, and the representative bench showed `root_rw`
  about 95ms with `intern` about 50ms. Total mono time is not yet materially
  lower because the work moved into `process-root-specialization-requests`
  (about 67ms), but the responsibility boundary is now clear enough to optimize
  request processing separately.
- Added target phase timing for queued specialization processing and printed a
  `rewrite_phase_target` list sorted by total queued work. In `showcase`, the
  queue is dominated by `std::flow::loop::for_in`: about 28ms total, almost all
  in `intern-rewrite-body`. The next queue optimization should target the loop
  helper body rewrite path, or more generally avoid recursively rewriting large
  std helper bodies when their request graph can be represented explicitly.
- Added target-scoped expression-kind profiling behind
  `YULANG_PRINCIPAL_REWRITE_KIND_PROFILE=1`. This keeps normal playground runs
  clean while showing which queued specialization body owns nested rewrite
  cost. For `showcase`, `std::flow::loop::for_in` was dominated by nested
  `lambda` / `apply` / `bind-here` / `thunk` work.
- Removed the first generic child rewrite pass from principal-unify `Block`
  handling. Blocks now enter the block-local context pass directly, then keep
  the existing local-use refinement pass for let initializers. This removes a
  scope-poor pre-walk and keeps the block-specific responsibility in one place.
  On warm `showcase` bench, `root_rw` dropped to about 26.8ms and
  `monomorphize` to about 88.7ms. `std::flow::loop::for_in` queued body rewrite
  dropped from about 28ms to about 16.8ms.
- Added the first host-backed `std::fs` surface:
  `read_text: str -> opt str`, `write_text: (str, str) -> bool`,
  `exists/is_file/is_dir: str -> bool`. The CLI/native basic host handles these
  requests with `std::fs`; wasm keeps fs requests unresolved rather than
  pretending browser filesystem access exists. `read_text` returns `nil` on
  read failure and `just text` on success. The runtime regression now checks
  writing, path predicates, reading, and `case read_text path` unwrapping.
- Added `notes/design/static-analysis-speed-plan.md` with the root-level plan:
  principal rewrite observability, specialization request graph/queue, and the
  NameResolvePass boundary.
- Started the NameResolvePass boundary by splitting value path resolution in
  `lower::expr::path` into `ResolvedValueUse` / `UnresolvedValueUse`. This is a
  behavioral no-op for now, but gives a concrete place to lift name resolution
  out of expression lowering.
