## Elaborated IR boundary

- `yulang-elaborated-ir` crate を追加した。
  - `yulang-elaborate` が `InferExport` から組む、単相化済み・型付き output IR の data crate。
  - algorithm は持たせず、runtime-lower が読む型 surface だけを置く。
- concrete type wrapper を追加した。
  - `ConcreteType::try_from_type` は `Unknown` と free type variable を拒否する。
  - `Any` / `Never` は Top / Bottom として concrete type に残せる。
  - serde deserialize も検証経路を通す。
- `EffectiveThunkType` と `ForceThunk` node を追加した。
  - effect を持つ thunk は `MonoType::EffectiveThunk` として表す。
  - thunk open は旧 `BindHere` ではなく、`source effective-thunk -> target computation`
    の boundary として `ForceThunk` に残す。
  - `Cast` と同じく、producer-consumer の型差が確定した場所で elaborated IR に明示する。
- `yulang-elaborate` から `yulang_elaborated_ir` を `elaborated_ir` として re-export した。

確認:

- `cargo test -q -p yulang-elaborated-ir`

## Tuple computation solving

- constraint seed を `ConstraintSet::solve` へ進めた。
  - root の concrete tuple type を分解し、tuple child expr の computation slot へ要素型を入れる。
  - tuple child effect は、親 tuple が pure effect (`Never`) の場合だけ `Never` として seed する。
  - 非 pure compound effect はまだ落とし、effect 分配を捏造しない。
- `build_program` は solver の結果から recursive に `ElaboratedExpr` を組むようにした。
  - concrete tuple root は elaborated tuple に落ちる。
  - apply は function constraints が未実装なのでまだ `UnsupportedExpr::Apply`。
- public `ConstraintError` から internal draft slot 型が漏れないように、公開用
  `ConstraintComputationSlot` を使う形にした。

## Lambda / immediate apply solving

- computation bounds から一意の concrete candidate を選ぶ規則を追加した。
  - `Union(var, int)` のように concrete 候補が一つだけなら、その concrete 型を採用する。
  - 候補が 0 個または複数なら、型を捏造せず `NonExactComputationBounds` のまま落とす。
- pure literal / pure tuple / lambda など、構文上 pure な expression は effect bounds に
  concrete 候補が出ない場合でも `Never` effect として seed する。
  - `Any` / `Unknown` fallback ではなく、構文規則による pure effect。
- lambda computation solving を追加した。
  - exact function type から body computation を復元する。
- immediate lambda apply の solving を追加した。
  - literal arg から arg computation を作る。
  - root result computation から inline lambda callee の function computation を作る。
  - direct ref callee は、ref scheme constraints がまだないため明示的に未対応で落とす。
- actual infer export canary:
  - `"1\n"` を `infer -> erased export -> elaborate build_program` まで通した。

## Direct ref apply solving

- `InferredBinding` に erased `DefId` を持たせた。
  - `RefExportTable::direct` の `RefId -> DefId` から binding scheme を引けるようにした。
  - `yulang-infer` の erased export でも binding def を出力する。
- erased scheme export で、body / role requirements に出ない unused quantified type var を落とすようにした。
  - `my f(x: int) = x` のような単相 binding が、unused quantified var のせいで
    generic direct ref と誤判定されないようにするため。
  - body / requirements に残る free var は quantified に戻し、generic scheme の surface は保つ。
- `yulang-elaborate` の constraint environment を追加した。
  - direct ref の単相 function scheme から callee / arg / result computation を復元する。
  - `quantified_*` / typeclass obligation / role requirement が残る scheme は、まだ
    instantiation と evidence solving が必要なので `GenericDirectRefScheme` として落とす。
- actual infer export canary:
  - `"my f(x: int) = x\nf 1\n"` を `infer -> erased export -> elaborate build_program`
    まで通した。
  - apply root の effect bounds に残差変数が残る場合でも、単相 direct ref の function
    scheme から `ret_effect` を復元する。`Any` / `Unknown` fallback ではない。
  - infer export には unused quantified type var が残ることがあるため、scheme body に
    実際に量化変数が出る場合だけ instantiation 待ちとして扱う。

## Function adapter hygiene plan

- 高階関数をさらに高階関数へ渡す場合、call site だけで `LocalPushId` / `AddId` を後挿入すると
  必要な boundary が first-class function value と一緒に移動できない。
- 方針:
  - `LocalPushId` / `AddId` は cast そのものではなく、effect request / effective thunk の
    runtime control boundary として扱う。
  - 挿入 phase は cast / `ForceThunk` と同じ producer-consumer boundary。
  - 関数値が境界を越える場合は `FunctionAdapter` に hygiene plan を持たせ、adapter が値として
    高階関数間を移動できるようにする。
- `yulang-elaborated-ir` に次を追加した。
  - expression node: `LocalPushId`, `PeekId`, `FindId`, `AddId`
  - `EffectIdVar` / `EffectIdRef`
  - `FunctionAdapterCall`
  - `FunctionResultBoundary::{AddId, FunctionAdapter}`
- `spec/2026-06-07-principal-monomorphization.md` に、runtime lower が関数型や apply spine から
  hygiene boundary を推測してはならないことを追記した。

## Elaboration constraint bounds graph

- `yulang-elaborate` の `ConstraintSolver` を、`DraftComputationId -> MonoComputation` の
  代入表から `TypeNode -> { lower, upper }` の制約グラフへ変更した。
  - computation value / effect と、scheme instantiate で fresh 化した型変数を同じ graph node として扱う。
  - `Type::Var(a) <: Type::Var(b)` は subtype edge、`T <: a` は lower bound、`a <: T` は upper bound。
  - lower 追加時は既存 upper へ、upper 追加時は既存 lower へ `constrain_types` を流す。
  - edge は lower を上流から下流へ、upper を下流から上流へ伝播する。
- generic direct ref apply は、量化変数を use-site fresh node に置き換えてから、
  actual arg / expected result の上下界で解くようにした。
  - `id : α -> α` / `id 1 : int` は `int <: α` と `α <: int` から `α = int` を読む。
  - まだ concrete 候補が一意に得られない場合は `GenericDirectRefScheme` で止め、`Any` / `Unknown`
    へ逃がさない。
- first cut の構造分解は function / tuple / 同一 named constructor の type args まで。
  record / variant / row / role obligation の本格制約化と、cast boundary 生成は次の作業。

## Structural expression elaboration

- `yulang-elaborate` の solver で record / variant / select / pack を扱うようにした。
  - record literal は期待 record type から field / spread child computation へ pure value 型を配る。
  - variant literal は期待 variant case の payload 型を child computation へ配る。
  - field select は selected result と base record field 型の一致を確認し、base record computation を解く。
  - pack は外側 computation を body へ伝播する。
- concrete candidate selection を structural type の中へ再帰するようにした。
  - `Union(Var, int)` のような bounds が record field 内に残る actual infer export でも、
    field ごとに一意な concrete candidate を拾って record 全体を concrete 化する。
  - `Any` / `Unknown` / `Never` を fallback として使わず、一意な concrete candidate が取れない場合は
    これまで通り bounds selection error で止める。

## ForceThunk elaboration

- `yulang-elaborate` の solver に explicit computation assignment table を追加した。
  - 通常の value computation は従来通り type bounds graph にも入れる。
  - `EffectiveThunk` は erased `Type` に戻せないため、graph に押し込まず exact computation として保持する。
- `BindHere` を `ForceThunk` boundary として解くようにした。
  - site の target computation から `EffectiveThunkType { value, effect }` を作る。
  - child expression は `MonoType::EffectiveThunk(source)` / pure effect の computation として解く。
  - elaborated IR では旧 `BindHere` node ではなく、`source effective-thunk -> target computation`
    の `ForceThunk` node を出す。

## Function value runtime shape

- `spec/2026-06-07-principal-monomorphization.md` に、concrete function type を
  runtime shape へ読む規則を明記した。
  - `a [b] -> [c] d` は、`b` / `c` が effectful な場合だけ
    `effective-thunk` boundary として扱う。
  - `Never` は empty effect row なので、`thunk[Never, a]` は作らず bare value shape に残す。

  - `EffectiveThunk` と同じく erased `Type` graph に戻せない shape として、
    `yulang-elaborate` の explicit computation assignment table へ載せられるようにした。
- まだ solver output の全 computation をこの shape へ自動変換する段階には進めていない。

## Runtime-shaped apply boundaries

- solver 内部は erased `Type::Fun` のまま解き、elaborated IR 出力時に
  `MonoComputation::into_runtime_shape` で concrete function 型を `FunctionBoundary` へ変換するようにした。
  - `ProgramBuilder` の specialization key は raw solver signature のまま保持する。
  - materialized `MonoInstance.signature` は、実際に出力される runtime-shaped computation を使う。
- apply elaboration で callee の `FunctionBoundary` を読むようにした。
  - parameter shape が `EffectiveThunk` の場合だけ、argument computation を `MakeThunk` で閉じる。
  - return shape が `EffectiveThunk` の場合だけ、inner `Apply` の pure 結果を `ForceThunk` で開く。
  - `Never` effect boundary は thunk 化せず、pure `int -> int` のような apply は bare `Apply` のまま残す。
- lambda parameter type も raw function type から同じ runtime shape 変換を通して決めるようにした。
  - pure parameter は bare value type。
  - effectful parameter は `EffectiveThunk`。
- `spec/2026-06-07-principal-monomorphization.md` の function runtime shape も、
  `thunk[Never, a]` を作らない規則へ明示した。
