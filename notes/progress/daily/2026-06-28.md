# 2026-06-28

## Evidence VM direct resumptive route

`DirectTailResumptive` を provider-env 経由の operation に広げるため、まず
request 経路との差分を確認した。

分かったこと:

- `DirectTailResumptive` を `MayEscapeYield` 全体へ無条件に広げると、
  `std::control::flow::sub::return` を扱う callback / returned thunk 系の
  hygiene regression が出る。
- `YieldFallback` は provider grant がまだ有効かを確認してから routed yield にするが、
  `DirectTailResumptive` は同じ gate を持っていなかった。
- `case` scrutinee や ref-set の force 後に direct resumptive / routed yield が出た場合、
  request 経路と同じ continuation frame を積む必要がある。
- `bench/nondet_20_discard.yu` の `nondet.list` は、`k(true).list` が
  `make-thunk((k true)).list` として下がるため、handler arm classifier からは
  `MayEscapeYield` に見える。
- provider grant の strict gate を path-aware に緩めても、nondet では対象 path の
  guard が実際に増えているため、request-free direct-tail には落ちなかった。
- provider route を gate 失敗時に `RoutedYield` へ早めに固定する実験は、
  returned thunk 系の `sub::return` regression を再発させたため採用しない。
- `direct_tail_gate_fail_add_id_foreign_path` の内訳を一時 trace で確認したところ、
  `bench/nondet_20_discard.yu` の 420 件はすべて
  `std::control::nondet::nondet::branch` に対する
  `std::control::junction::junction` marker だった。
  `junction` は `or` / `and` / `ret` を `my` operation として持ち、foreign path を守る
  add-id marker を積む。したがって、provider route lookup が失敗しているのではなく、
  別 handler 境界の衛生性により request-free direct-tail が止められている。

採用した変更:

- direct-tail force は `request_free_yield` または provider grant の path-aware gate を
  通った場合だけ request-free にする。
- `MayEscapeYield` は direct-tail execution plan へ分類できるようにしたが、
  provider-env route では上記 gate により unsafe な returned thunk は request fallback に残る。
- force 後の `case` scrutinee / ref-set continuation を request 経路と揃えた。

現時点の性能:

- `bench/nondet_20_discard.yu` はまだ direct-tail に乗らず、`direct_tail_continuation_appends`
  は 0 のまま。
- 次に大きく速くするには、guard / handler-boundary evidence を持った direct resumptive
  表現を入れるか、`MakeThunk` 内の direct continuation use を安全に分類する必要がある。
- 今回の trace から、少なくとも nondet/list 系では単に gate を緩めるのは危険だと分かった。
  次の本命は、`junction` の foreign-path guard を request object ではなく lightweight
  hygiene state として direct-tail 側に保持し、catch 境界で同じ可視性判定を行うこと。

## Evidence VM guarded direct-tail / forced effect fuse

`AddIdShadowed` で落ちていた provider direct-tail route を、request fallback ではなく
guarded direct-tail として表すようにした。request と同じ `guard_ids` / `carried_guards`
を集め、そのまま `EvidenceDirectTailResumptive` に載せる。`ScopeMissing` と
`HandlerShadowed` は従来通り fallback に残す。

あわせて `ForceThunk(Apply(EffectOp, arg))` の exact pattern だけを fuse した。ここで
重要なのは、provider route / grant は元の EffectOp apply と同じく marker frame の外で
作り、force だけを marked thunk force 相当の marker frame 内で行うこと。route lookup
まで marker frame 内へ入れると、grant baseline がずれて direct-tail guard が消えるので
採用しない。

採用しなかった案:

- `apply_scoped_value_result` を飛ばして EffectOp を直接 force する案。
  一時的には速く見えたが、function-call marker を省くため衛生性の根拠がなくなる。
- release build でも `ActiveAddIdIndex` を使う案。
  scan count は減ったが、path bucket の維持と candidate sort が重く、実時間は悪化した。

確認した数値:

- `bench/nondet_20_discard.yu`: `runtime_evidence_execute` は release repeat で
  33.6ms / 34.5ms / 37.6ms。`forced_effect_call_fusions: 861`。
- `examples/showcase.yu`: `runtime_evidence_execute` は release repeat で
  60.6ms / 61.7ms / 62.9ms。`forced_effect_call_fusions: 2457`。
- どちらも `compare.control: match`。

## Evidence VM AddId marker sharing

`DirectTailResumptive` と forced effect fuse 後も evidence VM が ControlVM に勝てない
主因は、provider lookup ではなく marker scope の再構築に残っていた。特に
`EvidenceAddIdMarker` は path を持つため、continuation resume / marker frame のたびに
deep clone すると、nondet / showcase のような workload で clone cost が支配的になる。

そこで `EvidenceValueMarker::AddId` と active add-id marker を `Rc<EvidenceAddIdMarker>`
へ変更し、通常の marker frame push では marker 本体を共有するようにした。関数呼び出しや
continuation resume で depth / guard flag を変える場合だけ、内側の marker を clone して
新しい `Rc` に包む。

採用しなかった案:

- release build で `ActiveAddIdIndex` を有効にする案。scan count は下がったが、HashMap
  bucket の維持と candidate sort が重く、`bench/nondet_20_discard.yu` は約 55ms、
  `examples/showcase.yu` は約 90ms まで悪化した。

確認した数値:

- `bench/nondet_20_discard.yu`: `runtime_evidence_execute` は release repeat で
  31.2ms / 31.9ms / 32.2ms。`compare.control: match`。
- `examples/showcase.yu`: `runtime_evidence_execute` は release repeat で
  49.1ms / 50.2ms / 51.5ms。`compare.control: match`。
- 同じ release binary の ControlVM は、`--runtime-phase-timings` で
  `bench/nondet_20_discard.yu` が `runtime_execute: 19.2ms`、
  `examples/showcase.yu` が `runtime_execute: 42.5ms`。

現時点の読み:

- showcase では ControlVM にかなり近づいたが、nondet はまだ大きく負けている。
- 残る差は、request-free direct route の不足だけでなく、
  `continuation_resume_marker_steps` / `marker_frame_entries` / `active_add_id_scans`
  が ControlVM と同程度に残っているのに、evidence VM 側では別表現への積み直しも
  払っている点が大きい。
- 次に成果が出る可能性があるのは、AddId marker の path 自体を intern すること、
  provider direct-tail が通った operation で marker frame を閉じ直さない fast path を
  作ること、または direct resumptive 専用 continuation を request fallback と完全に
  分けること。

## Evidence VM marked force cleanup

Koka 型の evidence passing に寄せるため、`Marked(callable)` を callable 側の hygiene
carrier へ吸わせる実験を行った。結果として、単に carrier を持たせるだけでは
`apply` / `force` 時に従来通り marker frame を開くため、速くならなかった。
さらに `Thunk::Value` へ hygiene を持たせないと marker を落としうるため、この形は採用しない。

採用した小さい fast path:

- `Marked(Thunk::Value(v), markers)` の force は request を出さない。
  現行意味論では marker frame の中で `v` を返し、frame close で `v` に marker を付けるだけなので、
  `mark_runtime_value_shared(v, markers)` へ直接置き換えた。

採用しなかった案:

- callable hygiene carrier 全体。`bench/nondet_20_discard.yu` は 30.3ms / 31.1ms 程度まで
  出ることもあったが、`examples/showcase.yu` は 51ms 台へ悪化し、
  `marker_frame_entries` も増えた。
- `Marked(Thunk::Effect)` の direct-abortive fast path。forced effect call fuse 後は
  対象が 0 件だった。
- identity continuation thunk の fast path。対象が 0 件だった。
- trivial expr thunk の fast path。対象が 0 件だった。

確認した数値:

- `bench/nondet_20_discard.yu`: `marked_force_value_fast_paths: 420`、
  `marker_frame_entries: 60566 -> 60146`、
  `marker_frame_marked_force_entries: 3444 -> 3024`。
- `examples/showcase.yu`: `marked_force_value_fast_paths: 815`、
  `marker_frame_entries: 98804 -> 97989`、
  `marker_frame_marked_force_entries: 8500 -> 7685`。

実時間の改善は小さく、支配項はまだ `continuation_resume_marker_steps` と
request/continuation fallback 側に残っている。ここから先は、値 wrapper の置き換えではなく、
operation 発生時に closed marker を request/direct route へ直接解釈する設計が必要。

## Evidence VM closed marker interpretation attempt

`MarkedForce` の残りを、marker frame ではなく closed marker evidence として解釈できるか
調べた。

まず `Marked(Thunk::Effect)` だけを直接 request/direct route へ解釈する fast path を
検討したが、現在の workload では対象がほぼ無かった。`bench/nondet_20_discard.yu` は
effect thunk fallback が 0、`examples/showcase.yu` は 7 件だけで、いずれも marker path の
条件により fast path には乗らなかった。

次に、active AddId を持たない marker set なら active frame/add-id を積まず、
active marker plan だけを積む `plan-only` closed marker として扱えるか試した。
これは sound そうだったが、実際の fallback marker set はすべて `depth == 0` の AddId を
持っており、対象は 0 件だった。

さらに攻めて、`carry_after_frame == false` の active AddId なら外へ持ち越さないので
plan-only にできるのではないかと試した。これは失敗した。

- `debug_runtime_evidence_run_resumes_carried_callback_markers` が
  `runtime-evidence-run escaped effect request: tick::ping` で落ちた。
- `carry_after_frame == false` でも、callback marker の resume 中には active AddId frame が
  必要になる。
- したがって「外へ carry しない marker」でも、continuation resume の局所衛生性には
  active frame として解釈する必要がある。

現在残したもの:

- `Marked(Thunk::Value)` の fast path。
- `MarkedForce` fallback の分類 counter。

現在の分類:

- `bench/nondet_20_discard.yu`
  - value fast path: 420
  - expr thunk fallback: 1323
  - continuation thunk fallback: 840
  - active AddId markers: 4306
  - carry-after-frame markers: 0
- `examples/showcase.yu`
  - value fast path: 815
  - expr thunk fallback: 3566
  - effect thunk fallback: 7
  - continuation thunk fallback: 1655
  - active AddId markers: 9632
  - carry-after-frame markers: 0

結論:

- 残りの本丸は `Marked(Thunk::Expr)` と `Marked(Thunk::Continuation)`。
- 単純な plan-only 化はできない。
- 次にやるなら、active frame を丸ごと消すのではなく、AddId frame の意味を
  continuation resume 専用の軽い hygiene state として表現し直す必要がある。

## Evidence VM local AddId frame entry fast path

continuation resume 専用の marker plan を試したが、これは採用しなかった。
`EvidenceContinuationFrame::MarkerFrame` に `frame_ids` / `add_ids` を持つ plan を保存し、
resume 時の marker slice 走査を避ける案だったが、plan 構築と追加 allocation の方が重く、
`bench/nondet_20_discard.yu` は 35ms 台、`examples/showcase.yu` は 55ms 台へ悪化した。

代わりに、通常の `push_marker_frame` に小さい fast path だけを残した。
`EvidenceValueMarker::AddId` は多くの場合、同じ marker slice の直前にある
`EvidenceValueMarker::Frame` と同じ guard id を指す。この場合は
`active_frames` 全体を `rposition` で探さず、直前の local frame entry をそのまま
`entry_frame_len` に使う。直前 frame と一致しない場合は従来どおり
`entry_frame_len_for_marker` に fallback する。

確認した数値:

- `bench/nondet_20_discard.yu`: `runtime_evidence_execute` は release 5 回で
  30.9ms / 31.8ms / 31.5ms / 31.9ms / 31.2ms。`compare.control: match`。
- `examples/showcase.yu`: `runtime_evidence_execute` は release 5 回で
  50.0ms / 50.2ms / 51.1ms / 53.4ms / 49.5ms。`compare.control: match`。

読み:

- marker の意味を変えずに入れられる改善としては安全だが、構造を変えるほどの効果はない。
- plan を別 object として保存する案は、この粒度では allocation が勝ってしまう。
- 本当に大きく削るには、marker slice の再解釈ではなく、operation / continuation
  表現そのものを request path から分ける必要がある。

## Evidence VM follow-up optimization sweep

追加で候補を一通り試した。今回残すコード変更はない。

試した案:

- DirectTailResumptive の no-continuation arm fast path。
  handler の該当 operation arm が continuation を実値として使わない場合だけ
  DirectAbortive として扱う実験を入れた。`bench/nondet_20_discard.yu` と
  `examples/showcase.yu` では `direct_tail_no_continuation_fast_paths` がどちらも 0。
  現在の代表 workload には刺さらないため戻した。
- `Marked(Thunk::Expr)` / `Marked(Thunk::Continuation)` の二重 dispatch 削減。
  `Marked` の内側が `Thunk` だと分かっているので、marker frame 内で
  `force_thunk_result(value.clone())` へ戻らず直接 thunk body を force する実験を入れた。
  回帰テストは通ったが、実時間は改善せず、showcase はむしろやや悪化したため戻した。
- active AddId marker に frame position を保存する案。
  request に guard を追加した後、`EvidenceRequestEntryExceptIndex` が
  `active_frames` を再検索する部分を避けるため、active marker に frame position を持たせた。
  回帰テストは通ったが、active marker のサイズ増加が勝ったようで実時間は改善せず戻した。
- continuation flattening。
  現在の `Then` の多くは `MarkerFrame` / `ProviderEnv` の scope boundary を保つために残っている。
  ここへ tail を押し込むと dynamic scope が変わるため、小さい局所変更としては触らない判断にした。
  本当にやるなら control VM の frame segment / snapshot に近い表現変更が必要。
- nondet/list。
  `ListTree` は persistent concat で、代表 workload の `list_values_copied` は 0。
  `list_merge_calls` も nondet 420、showcase 832 程度なので支配項ではない。
- allocator pressure。
  `/usr/bin/time` で全体は nondet が約 0.07s / 20MB、showcase が約 0.12s / 28MB。
  allocator 差し替えを試すには依存追加が必要で、構造改善なしに入れる段階ではない。

読み:

- いまの小技系はかなり頭打ち。
- 残る本命は、direct operation / continuation / marker scope を現行の
  request-passing 表現から切り離す大きめの runtime 表現変更。
- 特に `continuation_resume_marker_steps` と `direct_tail_continuation_then_appends` は、
  marker の意味を保ったまま単発 fast path で消すのは難しい。

## Evidence VM runtime redesign first slice

`notes/design/2026-06-28-evidence-vm-runtime-redesign-pro-review.md` の方針を受けて、
最初の小さい slice として `EvidenceResumeMarkerPlan` を入れた。

目的は速度改善ではなく、payload marking と resume dynamic scope をコード上で分けること。
従来は `close_marker_request` / `close_marker_direct_tail` / `close_marker_routed_yield`
がそれぞれ `markers_for_continuation_resume(...)` と
`EvidenceContinuation::marker_frame(...)` を直接扱っていた。今回から、まず
`resume_marker_plan(...)` で `EvidenceResumeMarkerPlan` を作り、最後に
`plan.attach(continuation)` で旧 marker-frame continuation へ戻す。

現時点では挙動は変えない。`ResumeMarkerPlan` はまだ旧 `MarkerFrame` への bridge であり、
push/pop 削減はしていない。次に `PureValueMark` / `DynamicScope` へ分ける足場として扱う。

確認:

- `cargo fmt --all --check`
- `cargo check -q -p evidence-vm -p yulang`
- `cargo test -q -p evidence-vm`
- `cargo test -q -p yulang runtime_evidence -- --test-threads=1`
- release compare-control:
  - `bench/nondet_20_discard.yu`: 31.1ms / 31.5ms / 31.3ms、`compare.control: match`
  - `examples/showcase.yu`: 51.4ms / 51.6ms / 50.0ms、`compare.control: match`

次の候補:

- `RouteCert` を first-class にして `EffectThunkEvidence` の `route_origin` 依存を薄める。
- `EvidenceRoutedYield` から `request` を剥がす。ただし最初は `to_request()` bridge で
  旧 path へ戻す。
- `ResumeMarkerPlan` を `PureValueMark` / `DynamicScope` に分け、値 mark と resume scope を
  実際に別経路へ逃がす。
