## `std::text::parse.word` の char specialization

- `word = label(some(satisfy(is_word)), "word")` は、主型としては
  `satisfy(char -> bool)` の反変引数から parse item が `char` の subtype として残る。
  `ParseError.unexpected` も associated `item` / `pos` に実 item / pos の supertype を要求するため、
  `item = char|'d` や `pos = 'b|'e` の残差は principal な情報として説明できる。
- 推論 core でこれを `char` に同一化すると、関数引数の反変性を弱めて principal 性を落とすので避けた。
- `word` 自体は標準の char parser として出したい値なので、alias ではなく
  `pub word(): [parse char 'err 'pos 'snap] list char = ...()` にした。
  direct alias annotation (`pub word: () -> ... = ...`) は temp 実験で `check-poly-std` が 35 秒 timeout したため採らない。
- surface は
  `() -> [std::text::parse::parse(char, 'a, 'b&'e, 'c)] list char where 'a: ParseError(item = char|'d, pos = 'b|'e)`
  まで戻った。associated 側の supertype 残差は仕様寄りで残す。

## string interpolation formatting protocol

- `spec/2026-06-06-syntax-design.md` に Rust 風の `%format{expr}` mini grammar を追記した。
  - kind は空/`?`/`x`/`X` を `Display` / `Debug` / `LowerHex` / `UpperHex` へ対応させる。
  - align/fill/sign/alternate/zero_pad/width/precision は `format_spec` に構造化して std protocol へ渡す。
- lowering 側で `StringInterpFormatText` を `FormatSpec` へ parse するようにした。
  - layout 指定なしの `%?{x}` / `%x{x}` / `%X{x}` は従来の `%{x}` と同じく direct selection
    (`.debug` / `.lower_hex` / `.upper_hex`) へ下げる。
  - layout 指定ありの `%04{x}` / `%#08x{x}` は
    `std::core::fmt::format_* (format_spec {...}) x` へ下げる。
  - invalid format text は引き続き `UnsupportedSyntax(StringInterpFormatText)` とする。
- std 側へ `format_kind` / `format_align` / `format_sign` / `format_spec` と
  `format_display` / `format_debug` / `format_lower_hex` / `format_upper_hex` を追加した。
  - `format_with_padding` の zero padding 分岐は、dump 上の式形が意図通り読めるよう途中値に分けた。
  - `+` sign は format 済み body が `-` で始まらない場合に sign prefix として反映する。
  - `prefix` は operator declaration keyword と衝突しやすいため、std 実装の内部名には使わず `head` にした。

## specialize root case の result slot join / direct cast

- 裸の root expression `case true: true -> 1, false -> 2.0` が、poly では
  `int | float` として通る一方で、mono では `conflicting type candidates for slot 0: int vs float`
  で落ちていた。
- 原因は specialize の expression type solver が、未知 result slot に集まった複数 lower bound を
  互いに衝突する候補として扱っていたこと。
- lower bounds / upper bounds は、それぞれ代表 concrete 型 1 個へ畳めるときだけ解決するようにした。
  - `case` 専用分岐ではなく、slot solver 側の一般規則。
  - 複数候補が subtype edge で一つにまとまらない場合は、union / intersection を合成せずエラーにする。
  - 両側候補がある場合は `lower <: upper` を満たすときだけ lower 側を採る。
  - 境界から concrete 候補が一つも出ない場合も、default へ落とさずエラーにする。
- direct cast 宣言を `poly::Arena` に残し、specialize の subtype 判定で direct edge として読む。
  - primitive 名の直書きではなく、読み込まれた `cast` 宣言だけを根拠にする。
  - cast graph の推移閉包は作らない。
- regression:
  - `specialize::solve` で direct cast のない root case が候補衝突で落ちること。
  - local `cast(x: int): float` がある場合だけ、root case の actual が `float` になり、
    `yulang2 dump-mono` で `coerce[int => float](1)` が挿入されること。
  - `cast(x: int): bool` と `cast(x: bool): float` だけでは `int <: float` とみなさず、
    候補衝突で落ちること。
- spec 同期:
  - `spec/2026-06-07-principal-monomorphization.md` に concrete selection は一つの型へ
    畳める場合だけ成功し、それ以外はエラーにする規則を追記。

## specialize handler arm identity / continuation shape

- `poly::expr::CatchArm` に `CatchOperation { path, def }` を追加し、effect arm が扱う
  operation の exact path と、解決済み operation def を erased IR に残すようにした。
  - mono IR / dump でも `out::say payload, k -> ...` のように handler arm の相手が読める。
  - runtime handler の path 照合に必要な identity を、型制約だけに閉じ込めて捨てない。
- `specialize::solve` の apply を、callee の actual runtime function shape から読むようにした。
  - operation call の `int -> thunk[[out], unit]` を `int -> unit` adapter で潰さず、
    call 結果の thunk/value boundary として `ForceThunk` を入れられる。
- catch solver は scrutinee body の runtime shape から value/effect を分け、
  body と continuation resume に必要な thunk force 境界を作る。
  - operation def の scheme から payload 型と continuation 入力型を読み、
    `out::say: int -> [out] unit` なら `msg: int`、`k: unit -> thunk[[out], unit]` とする。
  - generic operation では operation の return effect を scrutinee effect row の同じ family item に
    結び直す。`var::get: unit -> [var 'a] 'a` は `k(1)` から `[var(int)]` / `int` へ解ける。
- act operation def の参照を mono で `EffectOp { path }` に下げるようにした。
  - これまで `out::say(1)` は `(<local d1> 1)` に見えており、runtime lower が effect request の
    exact path を読めなかった。
  - lowering が `DefId -> EffectOperation(path)` を poly に残し、specialize の var 変換が
    `<effect-op out::say>` を作る。
  - handler payload の constructor pattern は、`eff::send(opt::some x), k -> ...` のように
    payload を括弧で囲めば mono まで下がることを yulang2 regression に追加した。
- root expression の actual が `Type::Thunk` なら runtime root の出口で `ForceThunk` を入れる。
  - `out::say(1)` だけの program も `force-thunk[thunk[[out], unit] ...]((<effect-op out::say> 1))`
    として実行対象になる。
  - root 以外の first-class thunk には触れない。
- `case` arm pattern を scrutinee の mono 型で bind するようにした。
  - constructor pattern は constructor def の scheme を instantiate し、return 型を scrutinee 型へ
    結んで payload 型を読む。
  - mono pattern の constructor は bodyless def なので、`InstanceId` ではなく `DefId` を保持する。
  - record expression は `Record` actual 型を作り、record pattern は field 名から局所変数へ型を流す。
    `case { width: 1 }: { width } -> id(width)` で `id: int -> int` まで単一化できる。
  - record field selection は base の `Record` 型から field 型を読み、`row.width` を `int` として
    body 側の generic call に渡せるようにした。
  - method selection は `SelectResolution::Method { def }` の method scheme を読み、receiver を base
    expected 型へ、method return を selection actual 型へ流すようにした。
    method instance も `receiver_actual -> selection_actual` の concrete 型で作るため、generic method
    が default 型で単一化されない。
  - poly variant expression / pattern と list pattern も、scrutinee の構造型から payload / item 型を
    渡す入口を specialize 側に足した。
- `my run = out::say(1); catch run: ...` は handler の内側で RHS を再評価しない。
  `run` は source order の computed runtime root として先に force され、effect は外へ逃げる。
  mono dump では `m0 = run` が `force-thunk[...]`、`handled` は `catch m0` になる。

## mono-runtime effect request / shallow catch

- `mono-runtime` の内部評価結果を `Value | Request` に分けた。
- `EffectOp` apply は exact path と payload、resume continuation を持つ request を作る。
- `Catch` は `operation_path` exact match の arm で request を捕まえ、continuation pattern には
  runtime continuation value を渡す。
- 対象外 request は外側へ pass-through するが、resume 後に同じ `catch` へ戻る continuation で包む。
  これにより shallow handler は最初の request を見るだけで止まらず、外側 handler が対象外 request を
  resume した後、body の奥へ進んで対象 request を捕まえられる。
- 対象 request を捕まえたときに arm へ渡す continuation は shallow なので、同じ handler は自動では
  再設置しない。
- tuple / record / poly variant の構築途中で発生した request は、resume 後に残りの構築へ戻る。
- case guard / catch arm guard の評価中に発生した request も、resume 後に guard の bool 判定へ戻る。
- `spec/2026-06-13-mono-vm-contract.md` に shallow handler の pass-through / resume 規則を追記した。

## mono-runtime FunctionAdapterHygiene

- `mono-runtime` に runtime guard marker の oracle semantics を追加した。
  - effect request は `guard_ids` を持つ。
  - runtime は dynamic `GuardIdList` と active value marker を持つ。
  - `FunctionAdapterHygiene` は adapter call ごとに fresh `GuardId` へ展開し、call body 全体を
    marker frame で囲む。
  - marker frame は request が外へ抜ける前に pop し、continuation resume で同じ id を re-push する。
  - `CatchArm.operation_path` が一致しても、request の `guard_ids` と current `GuardIdList` が交差する場合は
    handler が読まずに pass-through する。
  - `Value::Marked` を追加し、thunk force / function call / record projection / pattern destructuring で
    shape-directed に marker を解釈する。
  - scalar value は marker を保持しても発火点がないため、runtime value では落とす。
- adapter argument/result adaptation は marker frame の内側で実行する。
  - target argument は adaptation 前後で marker を保持し、source result は target adaptation 前に marker を付ける。
  - これにより adaptation が thunk force / thunk wrap を含む場合も marker が落ちない。
- regression として、foreign thunk argument、allowed path、depth=1 callback、record projection で返った thunk の
  marker propagation、outer handler resume 後の marker frame 復元を追加した。
