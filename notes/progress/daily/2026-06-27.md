## Playground install / LS request cache

- Playground front page に installer command strip を追加した。
  Unix は `curl -fsSL https://yulang.momota.pw/install.sh | sh`、Windows は
  `irm https://yulang.momota.pw/install.ps1 | iex` を出す。
- web editor は diagnostics debounce を持つようにし、入力ごとの即時 recheck を避ける。
- language server は hover / definition / references / prepareRename / rename の
  request-result cache を持つようにした。
  `PolyCheckOutput` は `RefCell` / `rowan::SyntaxNode` を含むため `Send + Sync` ではなく、
  `LanguageServer` state に full analysis cache として置けない。
  そのため cache するのは LSP の返却値だけに限定した。

## Control VM request guard set compaction

`Request.guard_ids` と `CarriedGuard.exposed_guard_ids` を `Vec<GuardId>` から
`GuardIds { SmallVec<[GuardId; 4]> }` に変えた。
request に載る guard set は小さいことが多く、ここで heap allocation と重複 push を避ける。

確認:

- `cargo fmt --check`
- `cargo check -q -p yulang`
- `cargo test -p control-vm`
- `cargo test -p yulang server`
- `cargo build -q -p yulang --release`

`bench/nondet_20_discard.yu` の cache hit 計測では、
`runtime_execute` は 25ms 台に寄った。直前の観測はおおむね 27〜29ms 台だった。
ただし `run.active_add_scans` と `run.marker_scope_frame_touches` は変わらないため、
これは request-local の小さい allocation / membership cost を削っただけで、
ScopeState 問題そのものを解いた変更ではない。

Commit:

- `4c6c1bbe perf: compact VM request guard sets`

## Rejected active guard index experiment

`active_frames[..entry_frame_len]` の prefix scan を避けるため、
active guard に first frame index を持たせる実験をした。
`request_excepted_at_marker_entry` と exposed guard collection は、
request の guard set から first frame index を引けば同じ意味で判定できる。

結果:

- `cargo fmt --check`
- `cargo check -q -p yulang`
- `cargo test -p control-vm`
- `bench/nondet_20_discard.yu` cache hit は 25〜26ms 台でほぼ横ばい
- `marker_scope_frame_touches` と `active_add_scans` は変わらない

差分は約 76 行増えたが、実時間の改善が薄く、本丸の scope touch に届かない。
そのため未コミットのまま戻した。次に VM を触るなら、active frame prefix scan ではなく
`ScopeState` shadow か continuation marker scope の表現側から入る。
