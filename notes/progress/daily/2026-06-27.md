## Control VM request guard set compaction

`Request.guard_ids` と `CarriedGuard.exposed_guard_ids` を `Vec<GuardId>` から
`GuardIds { SmallVec<[GuardId; 4]> }` に変えた。
request に載る guard set は小さいことが多く、ここで heap allocation と重複 push を避ける。

確認:

- `cargo check -q -p yulang`
- `cargo test -p control-vm`
- `cargo test -p yulang server`

これは request-local の小さい allocation / membership cost を削っただけで、
ScopeState 問題そのものを解いた変更ではない。

## Control VM ScopeState shadow

runtime の本命である `ScopeState` production path へ入る前段として、
`YULANG_VM_SCOPE_STATE_SHADOW=1` で有効になる read-only shadow state を追加した。

現行 implementation は active add-id marker に `entry_frame_len` を保存し、
request marking 時に `active_frames[..entry_frame_len]` を再走査して
entry-except guard set を復元している。
shadow state はこれを、marker activation 時点の `entry_except: GuardIds` として明示的に保存する。

現時点では production path は変えない。request が発生したら、
まず shadow state が expected marking を計算し、その後で既存 linear scan が request を更新する。
最後に `Request.guard_ids` と `CarriedGuard.exposed_guard_ids` を比較し、ずれたら panic する。

確認:

- `cargo fmt --check`
- `cargo check -q -p control-vm`
- `cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p yulang run_control_with_std_collects_nondet_each_list -- --test-threads=1`
- `cargo build -q -p yulang --release`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 target/release/yulang --std-root lib --runtime-phase-timings run --print-roots bench/nondet_20_discard.yu`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 target/release/yulang --std-root lib --runtime-phase-timings run --print-roots examples/showcase.yu`

shadow 有効時は当然 request marking を二重計算するため遅くなる。
`bench/nondet_20_discard.yu` の cache hit では `runtime_execute` が約 51.7ms。
env 無しの通常 path は 25〜27ms 台で、前段の guard set compaction 後から大きく崩れていない。

次の候補:

- shadow state を使って `request_excepted_at_marker_entry` と
  `request_guard_ids_at_marker_entry` を production path へ切り替える。
- その後、path-index / handler-index を入れる前に、foreign-path marker と
  entry-time excepted request の canary を増やす。
- `active_add_scans` と `marker_scope_frame_touches` はまだ構造的に残るため、
  ScopeState query の production 化だけで終わりではない。

## Persistent marker scope list

`Continuation.marker_scopes` を `Rc<[ContinuationMarkerScope]>` から
persistent cons list (`Rc<MarkerScopeList>`) へ変更した。

これまで `prepend_marker_scope` は request が marker boundary を外へ出るたびに
`[new scope, existing scopes...]` の新しい slice を作り、既存 scope を全 clone していた。
今回の変更では、先頭に `MarkerScopeList` node を一つ足すだけにし、既存 tail は `Rc` 共有する。

この変更で `enter_continuation_marker_scopes` の走査、
`consume_marker_frame` の scope touch、capture/invoke 時の shallow clone 数は変わらない。
効く箇所は、request continuation に marker scope を蓄積する時の O(depth) コピーだけ。
そのため `marker_scope_frame_touches` などの counter は同じままだが、実時間は少し下がった。

## Remove obsolete runtime guard stack

`Runtime::guard_ids` を削除した。

これは `Request.guard_ids` とは別物で、request marking が読む guard 集合ではない。
古い runtime 側の stack は marker frame push 時に `contains` で重複確認され、
frame pop 時に truncate されていたが、request の可視性判定には使われていなかった。

- `Runtime` が持っていた `guard_ids: Vec<GuardId>`
- `MarkerCheckpoint::guard_len`

- `pop_marker_frame` の runtime guard truncate

`Request.guard_ids` と `GuardIds` は残している。
`ScopeState` shadow も request 側の guard ids を読むため、意味論上の marker coloring には触れていない。

## Incremental ScopeState index truncate

`ScopeState::truncate` が add-marker index を毎回 full rebuild していたため、
shadow path の push/pop が重くなっていた。

add-marker index は `add_markers` と同じ LIFO 順で積まれるので、truncate では末尾から
pop しながら対応する path index だけを戻せばよい。
これにより、shadow state の truncate は「残っている marker 全体」ではなく
「今回 pop される marker 数」に比例する。

## VM marker scope absolute close frame

continuation resume 中の active marker scope は、これまで frame を 1 つ消費するたびに
全 active scope の `frames_remaining` を decrement していた。
marker-heavy workload ではこの per-depth touch が非常に太く、
100x nondet workload で `marker_scope_frame_touches` が 56,676,200 まで増えていた。

今回、各 scope を「resume 開始時から何 frame 消費した地点で閉じるか」という
`close_at_frame` で持ち、resume 全体の `consumed_marker_frames` から残り frame を計算する形にした。
これにより frame 消費時の更新は active scope 数に比例せず、resume 全体の単一 counter を進めるだけになる。

注意点として、`marker_scope_frame_touches` / `marker_scope_consume_touches` は
「論理的に影響を受ける scope 数」ではなく「consume 時に実際に触った state 数」を表す counter へ意味が変わった。
scope close / request-close 側の counter は引き続き別に見ている。

確認:

- `cargo fmt`
- `cargo check -q -p control-vm`
- `cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `cargo test -p yulang --test cli`
- `cargo test -p yulang source::tests -- --test-threads=1`
- `cargo build -q -p yulang --release`

## Active add path condition fast path

## Continuation frame storage abstraction

## Stored continuation snapshot experiment

`Runtime.continuations` の value を `Continuation` から `StoredContinuation` へ分けた。
first-class continuation の保存側は immutable snapshot、
invoke 側は `ContinuationFrames` の cursor として resume する。

実装内容:

- `StoredContinuation { frames, marker_scopes }` を追加した。
- `StoredContinuationFrames` は `Rc<[FrameSnapshotSegment]>` を持つ。
- active な `ContinuationFrames` は `Vec<FrameSegment>` を持つ。
  当初 `SmallVec<[FrameSegment; 2]>` を試したが、
  再帰 VM の stack frame が太り、100 回程度の ref-update loop で
  CLI stack overflow が再発したため採用しなかった。
- snapshot segment の partial `split_back` は suffix を materialize する。
  suffix view が元の `Rc<[SharedFrame]>` 全体を保持すると、
  request continuation が不要になった prefix frame まで保持してしまうため。
- `prepend` で active continuation 同士を連結する場合は owned に materialize する。
  segment chain をそのまま伸ばすと ref-update 系の request/marker close で悪化するため。
- `continuation_frames_cloned` / `continuation_marker_scopes_cloned` は actual copy 寄りにし、
  旧来の shape count は `continuation_frames_observed` /
  `continuation_marker_scopes_observed` として分けた。

## Hygiene marker filtering for pure values

値に付ける hygiene marker は、後で effectful computation を発生させうる値にだけ必要。
純粋な record / tuple / data constructor / list に marker を付けても、
あとから捕捉される effect は発生しないため、`Value::Marked` を作らないようにした。

実装:

- `is_scalar_value` を廃止し、`value_needs_hygiene_mark` を追加した。
- `Closure`, `RecursiveClosure`, `Thunk`, `FunctionAdapter`, `EffectOp`,
  `Continuation` は marker が必要。
- `Tuple`, `Record`, `PolyVariant`, `DataConstructor`, `List` は中身を見て、
  marker が必要な値を含む場合だけ marker を付ける。
- `ConstructorFunction` と `PrimitiveOp` は callable ではあるが、
  runtime hygiene 上は pure wrapper として扱い、保持している引数に
  marker が必要な値がある場合だけ marker を付ける。
- `markers.is_empty()` の場合は完全 no-op にして、既存 `Value::Marked` を剥がさない。

追加テスト:

- pure record payload は marker frame を抜けても `Value::Marked` にならない。
- closure を含む record payload は従来通り `Value::Marked` になる。

## Handler-passing runtime strategy note

Koka-like な handler/evidence passing 方向の設計メモを追加した。

追加:

- `notes/design/2026-06-27-handler-passing-runtime.md`

要点:

- Yulang では `for` / `ref` / parser / control flow が普通に effect を背負うため、
  pure island だけを切っても本命の速度改善にはなりにくい。
- 長期的には、generic control VM の上で handler を探索するのではなく、
  effectful computation へ handler evidence object を渡し、
  operation がその evidence を直接呼ぶ形へ寄せる。
- hidden stack evidence と runtime evidence routing は同じ handler visibility relation を
  表さなければならない。
- 最初の実装単位は execution change ではなく Stage 0 metrics:
  effect family / handler family / static handler candidate / fallback reason を数える。

これは runtime の局所最適化ではなく、control lowering 戦略として扱う。

## Handler-passing Stage 0.5 scoped candidate profile

`crates/control-vm/src/effect_profile.rs` に scoped traversal を追加した。

これは実行意味論を変えない profile で、top-level instance entry と expression root から
control IR をたどり、次を静的に数える。

- exact nearest handler candidate が見える effect operation visit
- handler が見えない effect operation visit
- callback hygiene boundary に遮られる candidate
- delayed computation boundary に遮られる candidate
- direct handler-passing に落とせそうな candidate
- resumptive handler candidate / abortive handler candidate
- direct-known effect call site
- dynamic call site
- known thunk force site / dynamic force site
- callback boundary / delayed boundary
- traversal の missing expr refs / cycle refs

Traversal rule:

- `catch` body は handler arms を積んで見る。
- `catch` の handler arm guard / body は、同じ handler の内側として見ない。
- non-empty hygiene の `FunctionAdapter` は callback boundary と見る。
- `lambda` / `make-thunk` body は delayed boundary と見る。

この分類はまだ typed hidden stack evidence を使っていない。
したがって、direct candidate は保守的に「lexical catch の内側にあり、callback / delayed boundary を跨がない」
ものだけにしている。

さらに実測で、単純な tree traversal だけでは実 workload の handler 候補が見えないことが分かった。
`catch action` のような形では、handler の内側に直接 `EffectOp` があるのではなく、
parameter value の thunk force がある。

## Handler-passing profile stop decision

Stage 0.5 の後、さらに `ForceThunk.source.effect` などを読んで typed force profile を増やす案を一旦止めた。

理由:

- 現行 control VM の profile は、現行 VM の request / catch / force 構造を測るもの。
- handler/evidence passing runtime は、handler evidence の位置と thunk/function の受け取り方を変える可能性が高い。
- そのため、現行 VM 上の「dynamic force under handler」を細かく測っても、新 runtime の意味論設計には直結しない。
- `crates/control-vm` の中で profile を増やし続けると、現行 VM の都合と新 runtime strategy の意味論が混ざる。

決定:

- Stage 0 / Stage 0.5 profile は、現行 VM の比較 baseline として残す。
- これ以上 typed-force profile を足さない。
- 現行 request/catch VM を少しずつ handler-passing 化しない。
- handler-passing は別 runtime strategy として、まず evidence IR / evidence lowering / evidence runtime の設計を書く。
- 現行 control VM は behavior oracle と fallback runtime として残す。

次に詰めるもの:

- effectful thunk / function が受け取る evidence parameter の形
- handler evidence object の operation / resume interface
- forwarding / yielding と hidden stack evidence の対応
- callback hygiene を evidence routing として表す方法
- generic VM fallback node の形
- current VM との oracle 比較 fixture

## Runtime evidence cache boundary

`notes/design/2026-06-27-handler-passing-runtime.md` に runtime evidence の cache staging を追加した。

結論:

- evidence の正本は `specialize` が `TypeGraph` / `SolvedTask` を持っている間に作る。
- `infer::CompiledRuntimeSurface` は pre-specialize の poly surface なので、最初の置き場所ではない。
- control cache は `control_vm::Program + runtime evidence sidecar` として拡張するのが自然。
- control cache hit なら evidence も再利用できる。
- mono cache hit だけでは solver-local stack evidence を復元できない。
- そのため evidence-passing build では、mono cache を bypass するか、mono cache にも同じ sidecar を足す必要がある。

次の実装候補:

1. `specialize` に `SpecializeOutput { program, runtime_evidence }` を足す。
2. 互換用に既存 `specialize(arena) -> Program` は残す。
3. `CachedControlArtifact` / `ControlCacheEnvelope` に sidecar を足し、`CONTROL_CACHE_FORMAT` を bump する。
4. 実 evidence は、まず root / instance / expr の actual / consumer computation type と、stack-weighted relation の canary から載せる。

## Runtime evidence continuation slice

`debug runtime-evidence-run` に、direct request を再開するための最小 continuation frame を追加した。
前段の direct handler route は、catch body 自体が operation call の thunk になるケースだけを扱えた。
今回の slice では、operation が subexpression から request として上がる場合に、
残りの評価文脈を `EvidenceContinuation` として request に積む。

追加した frame:

- `ForceThunk` — thunk 式を評価中に request が出た後、再開値を force する。
- `ApplyCallee` — callee 評価中に request が出た後、arg を評価して apply する。
- `ApplyArg` — arg 評価中に request が出た後、保存済み callee に apply する。
- `ApplyForcedCallee` — thunk callee の force 中に request が出た後、再開値を関数として apply する。
- `CaseScrutinee` — case scrutinee 評価中に request が出た後、arm 選択へ戻る。

continuation value は即値を返すのではなく、
`RuntimeEvidenceThunk::Continuation` として保存し、handler body の force 経路で再開する。
これにより、次の canary が現 control VM と一致する。

  v -> v
```

```

まだ残る境界:

- tuple / record / polyvariant / select / block の中で出る request は、まだ値専用 `eval_expr` 経路に落ちる。
- `all_paths` は list / std combinator / typeclass method select と高階 thunk の route 対応が必要。
- 実行はまだ `ControlEvidenceProgram` の expr id に依存しており、source/runtime evidence surface 自体を実行していない。
- blocked/delayed route は握りつぶさず `Unhandled` として外へ出す。ここを変えると hygiene の意味論が変わる。

- `cargo fmt`
- `cargo check -p yulang`
- `cargo test -p yulang debug_runtime_evidence_run -- --nocapture`
- `cargo build -q -p yulang --release`
- `cargo build -q -p yulang --release`

## Direct evidence-passing handler route

`debug runtime-evidence-run` が、control evidence を実行に使うようになった。
前段の subset runner は runtime evidence の件数を表示するだけだったが、
この変更では `ControlEvidenceProgram` から operation call route を index 化し、
operation call が `Effect` thunk を作る時点で次を埋める。

- `apply` expr id
- operation callee expr id
- direct handler expr id
- resumptive flag

`catch` は body から request が返ったとき、request に入っている direct handler id が
自分自身の catch expr id と一致する場合だけ operation arm を実行する。
一致しない request や blocked/unhandled route は握りつぶさず、外へ流す。

これにより、次の direct handler canary が現 control VM と一致する。

```yu
act flip:
  our coin: () -> bool

catch flip::coin():
  flip::coin(), k -> k true
  v -> v
```

出力:

```text
runtime evidence run:
  evidence.tasks: 1
  evidence.nodes: 3
  evidence.node_evidence_refs: 0
  control_evidence.effect_calls: 1
  control_evidence.direct_effect_calls: 1
  compare.control: match
run roots [true]
```

現時点でまだ未解決:

- `all_paths` のような高階 thunk 経由の evidence passing
- source/runtime evidence node id と control `ExprId` の対応
- request continuation に `case` / `apply` / `block` などの周辺文脈を積む処理
- list / std combinator / typeclass method select を含む handler body

この境界を曖昧にすると、ただの direct catch evaluator を evidence passing VM と誤認する。
現段階で入ったのは「direct operation route を evidence として渡して handler arm を呼ぶ」
最初の実行経路である。

- Ningning Xie et al., “Effect Handlers, Evidently”
