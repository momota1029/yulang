## Control VM request guard set compaction

`Request.guard_ids` と `CarriedGuard.exposed_guard_ids` を `Vec<GuardId>` から
`GuardIds { SmallVec<[GuardId; 4]> }` に変えた。
request に載る guard set は小さいことが多く、ここで heap allocation と重複 push を避ける。

確認:

- `cargo fmt --check`
- `cargo check -q -p yulang`
- `cargo test -p control-vm`
- `cargo test -p yulang server`
- `cargo build -q -p yulang --release`

`bench/nondet_20_discard.yu` の cache hit 計測では、
`runtime_execute` は 25ms 台に寄った。直前の観測はおおむね 27〜29ms 台だった。
ただし `run.active_add_scans` と `run.marker_scope_frame_touches` は変わらないため、
これは request-local の小さい allocation / membership cost を削っただけで、
ScopeState 問題そのものを解いた変更ではない。

Commit:

- `4c6c1bbe perf: compact VM request guard sets`

## Control VM ScopeState shadow

runtime の本命である `ScopeState` production path へ入る前段として、
`YULANG_VM_SCOPE_STATE_SHADOW=1` で有効になる read-only shadow state を追加した。

現行 implementation は active add-id marker に `entry_frame_len` を保存し、
request marking 時に `active_frames[..entry_frame_len]` を再走査して
entry-except guard set を復元している。
shadow state はこれを、marker activation 時点の `entry_except: GuardIds` として明示的に保存する。

現時点では production path は変えない。request が発生したら、
まず shadow state が expected marking を計算し、その後で既存 linear scan が request を更新する。
最後に `Request.guard_ids` と `CarriedGuard.exposed_guard_ids` を比較し、ずれたら panic する。

確認:

- `cargo fmt --check`
- `cargo check -q -p control-vm`
- `cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p yulang run_control_with_std_collects_nondet_each_list -- --test-threads=1`
- `cargo build -q -p yulang --release`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 target/release/yulang --std-root lib --runtime-phase-timings run --print-roots bench/nondet_20_discard.yu`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 target/release/yulang --std-root lib --runtime-phase-timings run --print-roots examples/showcase.yu`

shadow 有効時は当然 request marking を二重計算するため遅くなる。
`bench/nondet_20_discard.yu` の cache hit では `runtime_execute` が約 51.7ms。
env 無しの通常 path は 25〜27ms 台で、前段の guard set compaction 後から大きく崩れていない。

次の候補:

- shadow state を使って `request_excepted_at_marker_entry` と
  `request_guard_ids_at_marker_entry` を production path へ切り替える。
- その後、path-index / handler-index を入れる前に、foreign-path marker と
  entry-time excepted request の canary を増やす。
- `active_add_scans` と `marker_scope_frame_touches` はまだ構造的に残るため、
  ScopeState query の production 化だけで終わりではない。

## Rejected ActiveAddId entry-except snapshot

ScopeState shadow が通ったため、production path の小さい切替として
`ActiveAddIdMarker` 本体に `entry_except: GuardIds` を持たせる実験をした。
狙いは `request_excepted_at_marker_entry` と `request_guard_ids_at_marker_entry` の
`active_frames[..entry_frame_len]` 再走査を消すことだった。

結果:

- `cargo fmt --check`
- `cargo check -q -p control-vm`
- `cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `cargo build -q -p yulang --release`

`bench/nondet_20_discard.yu` の cache hit では、通常 path の
`runtime_execute` が 58〜60ms 台へ悪化した。
直前の request guard set compaction 後は 25〜27ms 台だったため、この形は棄却した。

見立て:

- entry snapshot を `ActiveAddIdMarker` 本体に載せると、marker push / dedup /
  active marker clone が太くなる。
- `active_add_scans` と `marker_scope_frame_touches` は変わらないため、
  この変更は本丸の構造コストを減らせない。
- entry-except は production marker 本体へ eager に持たせるのではなく、
  ScopeState の query/cache 側へ寄せる方がよい。

## ScopeState path candidate counters

`ScopeState` shadow に path candidate counter を追加した。
`YULANG_VM_SCOPE_STATE_SHADOW=1` の時だけ、全 active add marker 数と、
path index がある場合に request ごとに見る必要がある candidate 数を集計する。
通常 path では counter は 0 のままで、追加 query は走らない。

追加した timing labels:

- `run.scope_state_shadow_checks`
- `run.scope_state_shadow_active_add_markers`
- `run.scope_state_shadow_path_candidates`
- `run.scope_state_shadow_all_path_candidates`
- `run.scope_state_shadow_own_path_candidates`
- `run.scope_state_shadow_foreign_path_candidates`

代表値:

```text
bench/nondet_20_discard.yu:
  active add markers: 32634
  path candidates:    17703
  all-path:            2562
  own-path:               0
  foreign-path:       15141

examples/showcase.yu:
  active add markers: 54648
  path candidates:    24696
  all-path:            5896
  own-path:               0
  foreign-path:       18800
```

own-only marker はこの二つの workload では 0 だった。
したがって、`request path -> own marker` だけの prefix index はほとんど効かない。
効かせるなら、foreign-only marker を「全体 - matching prefix」として扱い、
all-path / foreign-path / matching-excluded を順序つきに merge する必要がある。

確認:

- `cargo fmt --check`
- `cargo check -q -p yulang`
- `cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `cargo test -p yulang --test cli`
- `cargo build -q -p yulang --release`

## Rejected ScopeState indexed marking

`YULANG_VM_SCOPE_STATE_INDEX=1` で、ScopeState の path candidate index から
request marking を行う実験をした。
`YULANG_VM_SCOPE_STATE_SHADOW=1` も併用し、indexed marking と linear marking が
同じ `Request.guard_ids` / `CarriedGuard` になることを確認した。

確認:

- `YULANG_VM_SCOPE_STATE_INDEX=1 YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p control-vm`
- `YULANG_VM_SCOPE_STATE_INDEX=1 YULANG_VM_SCOPE_STATE_SHADOW=1 cargo test -p yulang run_control_with_std_collects_nondet_each_list -- --test-threads=1`
- `cargo check -q -p yulang`
- `cargo build -q -p yulang --release`

同値性は通ったが、実時間は悪化した。

```text
bench/nondet_20_discard.yu cache hit:
  normal runtime_execute: 26〜28ms
  index runtime_execute:  71〜75ms
  active_add_scans:       32634 -> 17703
```

scan 数は減ったが、ScopeState の HashMap 更新、candidate index の sort/dedup、
foreign-only exclusion の merge が重すぎた。
この形は未コミットで戻した。

次に production 化を狙うなら、request ごとの `Vec<usize>` + sort ではなく、
scope push/pop 時点で LIFO に削除できる軽い indexed frontier を作る必要がある。
ただし `marker_scope_frame_touches` はまだ別の支配項なので、
path candidate index だけを急いで production 化しない。

## Persistent marker scope list

`Continuation.marker_scopes` を `Rc<[ContinuationMarkerScope]>` から
persistent cons list (`Rc<MarkerScopeList>`) へ変更した。

これまで `prepend_marker_scope` は request が marker boundary を外へ出るたびに
`[new scope, existing scopes...]` の新しい slice を作り、既存 scope を全 clone していた。
今回の変更では、先頭に `MarkerScopeList` node を一つ足すだけにし、既存 tail は `Rc` 共有する。

この変更で `enter_continuation_marker_scopes` の走査、
`consume_marker_frame` の scope touch、capture/invoke 時の shallow clone 数は変わらない。
効く箇所は、request continuation に marker scope を蓄積する時の O(depth) コピーだけ。
そのため `marker_scope_frame_touches` などの counter は同じままだが、実時間は少し下がった。

確認:

- `cargo fmt --check`
- `cargo check -q -p control-vm`
- `cargo test -p control-vm`
- `cargo test -p yulang --test cli`
- `cargo build -q -p yulang --release`
