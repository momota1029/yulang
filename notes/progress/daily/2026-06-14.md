# 2026-06-14

## yulang2 specialize: `[_]` handler annotation

- `specialize` が `Scheme.stack_quantifiers` を unsupported として拒否していた制限を外した。
- scheme matching では `Stack(inner, weight)` の `inner` を expected type に直接合わせないようにした。
  - expected は stack 適用後の型であり、inner に束縛すると handler argument effect と return effect が衝突する。
- `mono::Type::Stack` を effect row に materialize / resolve するとき、stack weight の floor/stack subtractability を effect row item に適用して簡約するようにした。
  - `all_except(signal)` が `[signal]` を消し、root 側に余計な `force-thunk[thunk[stack(...), _]]` が残らないことを確認した。
- regression:
  - `specialize::tests::string_input_specializes_stack_quantified_handler_annotation`
  - `yulang2::source::tests::dump_mono_without_std_lowers_wildcard_stack_handler_annotation`

## yulang2 lowering: `ApplyColon` block argument

- `infer::lowering` の `ApplyColon` 引数収集で、inline `Expr` が無い場合に `IndentBlock` / record literal ではない `BraceGroup` を 1 つの引数として拾うようにした。
  - `sub::sub:` が `App(sub.sub, Unit)` ではなく `App(sub.sub, <block>)` へ落ちる。
- `specialize` の effect row subtype で、末尾 effect slot を row tail として扱うようにした。
  - `[sub] <: [sub; β]` のような形で β が他に拘束されなければ、未拘束 effect slot を空 residual `[]` として解く。
  - `β & [signal; β]` のように β が全体 effect としても出る場合は、row tail 側だけを見て `β = []` と決め打たない。
- `Neg::Row` materialize 時に tail が `EffectRow` なら item として push せず flatten するようにした。
- regression:
  - `infer::lowering::tests::apply_colon_indent_block_lowers_block_as_argument`
  - `yulang2::source::tests::dump_mono_without_std_lowers_apply_colon_indent_block_argument`
  - `yulang2::source::tests::dump_mono_without_std_lowers_apply_colon_polymorphic_stack_handler_call`

## yulang2 specialize: polymorphic stack handler call

- `pub act sub 'a` の `sub::sub: sub::return 0` が `dump-mono` まで通るようにした。
- `Neu::Bounds(lower, upper)` の materialize で lower だけを採用して upper 側の関係を捨てていたため、effect family argument 経由で `'a = int` が伝わっていなかった。
  - lower/upper の両方がある場合は intersection として保持し、`T & T` を `T` に簡約する。
- effect を含まない具体値にかかった `stack(T, weight)` は `T` に簡約する。
  - `stack(int, pop(...)) | stack(int, pop(...))` が `int` へ落ちる。
- `sub::sub:` を使った nested handler source は `dump-mono` まで通ることを手元確認した。

## yulang2 execution: mono-runtime bridge and stack handler hygiene

- `yulang2 run-mono` / `run-mono-std` を追加し、`dump-mono` と同じ source -> infer -> specialize の `mono::Program` を `mono-runtime::run_program` へ渡す入口を作った。
- `thunk -> thunk` の境界を `coerce` ではなく lazy な `make-thunk { force source; value boundary }` として mono へ落とすようにした。
  - `Coerce` は内側式を先に評価するため、handler 引数の thunk を `coerce[thunk=>thunk]` にすると handler に渡る前に effect が漏れる。
- stack handler の mono instance 本体に `marker[path](...)` frame を入れ、handler の thunk body 内で読んだ local function/thunk へ innermost marker を染み込ませるようにした。
  - direct `sub::return` は depth 1 の frame では marker 付与されず、handler 自身が捕まえる。
  - callback 経由の `sub::return` は function call で depth が 0 になり、inner handler を素通りして outer handler が捕まえる。
- regression:
  - `yulang2::source::tests::run_mono_without_std_runs_root_expression`
  - `yulang2::source::tests::run_mono_without_std_runs_apply_colon_block_argument`
  - `yulang2::source::tests::run_mono_without_std_runs_polymorphic_stack_handler_call`
  - `yulang2::source::tests::run_mono_without_std_keeps_stack_handler_hygiene`

## control-vm: first mono lowering and runner

- `control-vm` を placeholder から `ExprId` arena 型の軽量 control 表現へ差し替えた。
  - `mono::Program` を `control_vm::Program` へ機械的に lowering し、runtime は control expr arena を読む。
  - oracle と同じ handler / continuation / marker / thunk adapter の意味を持たせた。
- `yulang2 run-control` / `run-control-std` を追加し、`dump-mono` と同じ mono program を control VM で実行できるようにした。
- mono VM contract に `thunk -> thunk` lazy boundary と `MarkerFrame` を追記し、runtime guard marker spec に stack handler marker frame の読み方を追記した。
- regression:
  - `control_vm::tests::runs_specialized_generic_call_like_oracle`
  - `control_vm::tests::runs_stack_handler_hygiene_to_outer_handler`
  - `yulang2::source::tests::run_control_without_std_runs_apply_colon_block_argument`
  - `yulang2::source::tests::run_control_without_std_keeps_stack_handler_hygiene`

## control-vm: module split

- 初期実装で大きくなった `control-vm/src/lib.rs` を `ir` / `lower` / `runtime` / `format` / `tests` に分けた。
  - `lib.rs` は crate の入口と re-export だけを持つ。
  - `lower` は `mono` tree から control IR arena への変換だけを持つ。
  - `runtime` は control IR interpreter と runtime value / marker / continuation を持つ。

## control-vm: VM-ready validation and oracle parity

- control VM 実行入口に `validate` を挟み、runtime が読む前に VM-ready でない control IR を止めるようにした。
  - unsupported primitive / ref set / list pattern / constructor pattern / unresolved select / unresolved typeclass method / `OpenVar` / unresolved stack・union・intersection 型を validator 側で構造化エラーにする。
  - boundary support 判定を `boundary` module に分け、validation と runtime の thunk/value boundary 判定を共有した。
- control VM tests に source -> specialize -> mono-runtime oracle / control VM の parity helper を追加した。
  - generic call、apply-colon block argument、stack handler hygiene を同じ helper で比較する。

## mono/control runtime: list primitive and constructors

- 旧 `yulang-vm` の persistent list 実装を `crates/yulang-list-tree` として共有 crate 化し、`mono-runtime` / `control-vm` の両方から使うようにした。
- `mono` / `control-vm` の IR で primitive op を文字列ではなく `PrimitiveOp` enum + `PrimitiveContext` として持つようにした。
  - `PrimitiveOp::arity()` を `mono` 側へ寄せ、runtime / formatter は同じ定義を読む。
  - `ListViewRaw` には `list_view::{empty, leaf, node}` の constructor def id を context として渡す。
- `mono-runtime` / `control-vm` に list 値、constructor 値、constructor partial、primitive partial、list pattern、constructor patternを実装した。
  - 実装済み primitive は list empty/singleton/merge/len/index/raw range/splice/view、主要 int/float/bool/string/char helper。
  - bytes/path と range wrapper 系の一部 primitive はまだ unsupported のまま。
- `specialize` で list literal の primitive spine に expected result を流し、`std::data::list::view_raw [1, 2]` のような std list path を持つ文脈で bare list fallback にならないようにした。
