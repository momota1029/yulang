## Evidence VM 独力高速化メモ

目標:

- `bench/nondet_20_discard.yu` の `runtime_evidence_execute` を一桁 ms 台へ寄せる。
- Pro に頼る前に、手元で説明できる VM hot path を順に削る。

採用した変更:

- primitive partial application の引数を `SmallVec<[SharedValue; 4]>` にした。
- effect call route / operation visibility / provider lookup を `(ExprId, ExprId)` `HashMap`
  から apply expr indexed table に寄せた。
- provider env / slot / small candidate tables を `SmallVec` 化した。
- provider env active refs/views の一時 `Vec` allocation を `SmallVec` 化した。
- `with_provider_env` で active frame 用に provider env を clone せず、pop した frame から
  close 用 env を回収する形にした。
- shared continuation frame は、scope boundary でなければ clone した spine に tail append
  する既存方針を維持した。

試して不採用にした案:

- release default から主要 runtime stats counter を外す案。
  - 速度差がほぼなく、通常 diagnostics が薄くなるので戻した。
- `Bool` / `Unit` の thread-local shared `Rc` 化。
  - `thread_local` 入口コストが勝ち、nondet / showcase とも悪化。
- `MarkedApply` の frame-less close 拡張。
  - 代表 workload では該当せず、marker frame 数が減らなかった。
- duplicate marker scope skip。
  - `marker_frame_entries` は nondet `13526 -> 7293` まで減ったが、cover 判定と
    close 形状のコストで実時間は悪化した。
- runtime value payload (`Tuple` / `DataConstructor`) の `SmallVec` 化。
  - 代表 workload ではほぼノイズで、diff が大きいので戻した。
- fat LTO + `target-cpu=native`。
  - たまに `9.7ms` は出るが平均は 10ms 台。build が重く、通常解決ではない。

最終測定:

- release `bench/nondet_20_discard.yu --compare-control`
  - `runtime_evidence_execute`: おおむね `10.5ms`〜`12.1ms`
  - `compare.control: match`
- release `examples/showcase.yu --compare-control`
  - `runtime_evidence_execute`: おおむね `24ms`〜`26ms`
  - `compare.control: match`

読み:

- 小さい allocation 削減や counter 削減では一桁台に安定して入らない。
- 残りの主戦場は `MarkerFrame` / `ProviderEnv` / `Then` を含む continuation 表現。
- duplicate marker scope は数としては太いが、単純 skip は速くならない。
- 次に進むなら、profiler が使える環境を用意するか、ProviderEnv/marker scope の
  意味論を保った continuation representation を設計し直す必要がある。

- `cargo check -q -p evidence-vm -p yulang`
