## control VM shared marker function-call path

- request に guard を付ける `active_add_ids` は、request 時点で効く `depth == 0` の
  `AddIdMarker` だけを積むようにした。
  `depth > 0` の marker は `active_marker_plans` 側に残り、call / resume 変換で
  depth が落ちた後に request-visible stack へ入る。
- request がすでに live guard を持つ場合と、scan 中に live guard を見つけた場合は、
  以降の add-id scan を打ち切るようにした。
  `active_add_id_scans` は repeat 計測で
  `bench/nondet_20_discard.yu: 21273 -> 1302`、
  `examples/showcase.yu: 41150 -> 4106`。

  identity になる場合は `Rc<[ValueMarker]>` をそのまま共有するようにした。
  これに合わせて shared marker frame の enter / close 経路を追加し、
  identity path では `Vec<ValueMarker>` の clone と `markers.to_vec() -> Rc` を避ける。
- handler path ごとの `HashMap` index も試したが、path prefix check は減る一方で
  marker push 側の map 更新が重く、repeat 5 の `runtime_execute` が悪化したため採用しない。

## control VM structural expr payload IDs

- tuple / record / poly variant の静的 payload を `EvalExpr` や `Frame` へ clone せず、
  frame 側には元の `ExprId` と進行中の index だけを持たせる形にした。
  item / field / tag は `Program` の immutable IR から index で読み直す。
- `Program` への raw pointer slice 参照も検討したが、crate は `#![forbid(unsafe_code)]` のため不採用。
  safety を崩さずに同じ方向へ寄せるため、`ExprId` 経由の静的 payload lookup にした。
- `HashMap` cache と dense `Vec<Option<...>>` cache も試したが、lookup / allocation の固定費が増えて
  `examples/showcase.yu` が悪化したため採用しない。

## control VM cheap-clone control values

- `Value::Closure` / `Value::RecursiveClosure` / `Value::Thunk` /
  `Value::FunctionAdapter` が大きな runtime payload を直に所有しないようにし、
  `Rc` handle を持つ形へ変えた。
  env lookup や continuation / frame 内の `Value` clone では、closure の `Pat`、
  thunk payload、function adapter の `Type` と inner function を複製しない。
- thunk は `Rc::try_unwrap` で一意なら従来どおり payload を move し、
  共有中だけ clone に落とす。`Rc` 化による force hot path の余分な clone を避けるため。
- repeat 5 の `bench/nondet_20_discard.yu` release / no-cache:
  `runtime_execute` 41.9ms / 42.5ms / 42.3ms / 44.5ms / 41.7ms。
  親ポインタ env 直後は 45.1ms / 44.7ms / 45.0ms / 44.7ms / 46.4ms。

## control VM marker scope frame-delta experiment

- active marker scopes の `frames_remaining` を各 frame consume / extend ごとに全 scope へ反映せず、
  `frame_delta` を一つ持つ案を試した。
  completed 判定だけ `frames_remaining + frame_delta` を見て、request で innermost marker を閉じる時だけ
  outer scope の remaining を materialize する形。
- `cargo check -q -p control-vm -p yulang` と
  `timeout 240s cargo test -q -p control-vm -p mono-runtime -- --test-threads=1` は通ったが、
  repeat 5 の `runtime_execute` は改善しなかった。
  `bench/nondet_20_discard.yu` は 41.6ms / 41.5ms / 40.7ms / 42.9ms / 43.8ms、
  `examples/showcase.yu` は 91.5ms / 96.3ms / 96.6ms / 96.7ms / 93.9ms。
  cheap-clone control values 後の揺れ幅と同程度で、複雑性に見合う差ではない。
- この方向は現状では採用しない。
  marker scope は単なる counter cursor 化より、active marker plan / handler state 自体を
  `ScopeId` 風の pointer state に畳む方を次に見る。

## control VM owned frame slot experiment

- 通常 request continuation では `Frame` を owned のまま積み、catch で multi-shot continuation として
  捕捉する時だけ `Rc<Frame>` snapshot へ変換する `FrameSlot::{Owned, Shared}` 案を試した。
  狙いは通常 path の frame ごとの `Rc::new(Frame)` を外すこと。
- `cargo fmt --all -- --check` と
  `timeout 240s cargo test -q -p control-vm -p mono-runtime -- --test-threads=1` は通った。
  repeat 5 では `frame_allocs` が `bench/nondet_20_discard.yu` で 13,968 -> 4,768、
  `examples/showcase.yu` で 42,773 -> 12,760 まで落ちた。
- ただし `runtime_execute` は悪化した。
  `bench/nondet_20_discard.yu` は 53.6ms / 57.3ms / 55.9ms / 54.8ms / 58.5ms、
  `examples/showcase.yu` は 116.5ms / 115.6ms / 120.5ms / 123.9ms / 111.9ms。
  `FrameSlot` 分岐、owned frame の大きさ、capture 時 snapshot 化の定数が、
  allocator 削減より重かったと見る。
- この方向は採用しない。
  current stack / captured snapshot の分離をやるなら、`Frame` enum 自体の payload を
  先にさらに小さくするか、chunked snapshot のように branch を増やさない表現へ寄せる必要がある。
