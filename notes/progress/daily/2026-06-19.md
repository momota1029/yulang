## control VM arm payload lookup

- 残っている `shared_frame_unwrap_clones` のうち `case` / `catch` bucket を見た。
  `examples/showcase.yu` の単発 debug timing では
  `case=1655`、`catch=3285`、`block=966`、`apply=831`。
- `CaseGuard` / `CatchValueGuard` / `CatchRequestGuard` だけを borrowed execution に載せる案を試したが、
  `shared_frame_unwrap_clones` が動かず、該当 clone は guard frame 由来ではなかったため戻した。
- `BindThen::CaseArm` / `CatchValue` / `CatchRequestPayload` が共有済み arm table の要素を
  さらに所有していたため、`arm` payload を外した。
  bind 後は `arms + index` から `guard` / `body` / `continuation` を読む。
  `eval_case_arm` / `handle_catch_*_arm` も `arms[index].clone()` せず、
  `bind_pat` に必要な pattern だけ clone する。
- repeat 5 の `bench/nondet_20_discard.yu` release / no-cache:
  `runtime_execute` 56.9ms / 56.3ms / 59.9ms / 56.3ms / 54.8ms。
- repeat 5 の `examples/showcase.yu` release / no-cache:
  `runtime_execute` 126.9ms / 122.5ms / 128.7ms / 122.7ms / 122.9ms。
  `shared_frame_unwrap_clones` は変わらないため、これは arm payload clone 削減として見る。

  result frame を borrowed execution で処理するようにした。
  `CatchResult` / `RefSetHandleResult` / `RefSetHandleValueResult` は、
  継続再開に必要な payload だけ clone する。

  `ApplyAdapterArg` / `CaseScrutineeForce` / `CaseScrutinee` /
  `BlockLetValue` / `BlockExprValue` を borrowed execution に載せた。

  `env` / shared arm table / statement table / marker table だけを clone する。

  `bench/nondet_20_discard.yu=881`、`examples/showcase.yu=1797` まで低下。
  apply frame を載せた後は `bench/nondet_20_discard.yu=21`、`examples/showcase.yu=966`。
  block frame まで載せると、repeat 5 の両対象で `shared_frame_unwrap_clones=0`。

## control VM shared marker function-call path

- request に guard を付ける `active_add_ids` は、request 時点で効く `depth == 0` の
  `AddIdMarker` だけを積むようにした。
  `depth > 0` の marker は `active_marker_plans` 側に残り、call / resume 変換で
  depth が落ちた後に request-visible stack へ入る。
- request がすでに live guard を持つ場合と、scan 中に live guard を見つけた場合は、
  以降の add-id scan を打ち切るようにした。
  `active_add_id_scans` は repeat 計測で
  `bench/nondet_20_discard.yu: 21273 -> 1302`、
  `examples/showcase.yu: 41150 -> 4106`。
- `apply_scoped_value` と direct-known callee apply で、function-call marker 変換が
  identity になる場合は `Rc<[ValueMarker]>` をそのまま共有するようにした。
  これに合わせて shared marker frame の enter / close 経路を追加し、
  identity path では `Vec<ValueMarker>` の clone と `markers.to_vec() -> Rc` を避ける。
- handler path ごとの `HashMap` index も試したが、path prefix check は減る一方で
  marker push 側の map 更新が重く、repeat 5 の `runtime_execute` が悪化したため採用しない。
- repeat 5 の `bench/nondet_20_discard.yu` release / no-cache:

- repeat 5 の `examples/showcase.yu` release / no-cache:

## control VM structural expr payload IDs

- tuple / record / poly variant の静的 payload を `EvalExpr` や `Frame` へ clone せず、
  frame 側には元の `ExprId` と進行中の index だけを持たせる形にした。
  item / field / tag は `Program` の immutable IR から index で読み直す。
- `Program` への raw pointer slice 参照も検討したが、crate は `#![forbid(unsafe_code)]` のため不採用。
  safety を崩さずに同じ方向へ寄せるため、`ExprId` 経由の静的 payload lookup にした。
- `HashMap` cache と dense `Vec<Option<...>>` cache も試したが、lookup / allocation の固定費が増えて
  `examples/showcase.yu` が悪化したため採用しない。
- repeat 5 の `bench/nondet_20_discard.yu` release / no-cache:
  `runtime_execute` 49.7ms / 48.2ms / 50.9ms / 50.5ms / 48.9ms。
- repeat 5 の `examples/showcase.yu` release / no-cache:
  `runtime_execute` 108.4ms / 109.5ms / 110.9ms / 109.9ms / 112.1ms。

## control VM static frame payload IDs

  `FunctionAdapterHygiene` を所有しないようにし、元の `ExprId` だけを持つ形にした。
  resume 時は `Program` の static payload から必要な型や hygiene を読み直す。
- `EvalExpr::from_expr` でも、上記三種の `source` / `target` / `target.value` /
  `hygiene` clone をやめた。型 clone は実際に boundary adaptation や
  `Value::FunctionAdapter` を作る時点まで遅延する。
- closure の `Pat` も `ExprId` 化する案を試したが、apply ごとの `Program` lookup が増え、
  `examples/showcase.yu` が悪化したため不採用。closure は従来どおり `Pat` / `body` を持つ。
