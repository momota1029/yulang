# 2026-06-26

## CLI source-unit prefix cache widening

Continued the source-unit compiled cache integration from 2026-06-25.

The CLI still uses a single compiled source-unit prefix. It does not merge
multiple `.yuunit` surfaces yet, because `BodyLoweringPrefix` currently imports
one namespace/lowering/runtime surface at a time. Instead, when multiple cached
dependency-free source units are available, the CLI now chooses one unit as the
prefix and lowers every other file from source.

This keeps the implementation inside the existing single-prefix invariant while
avoiding a full compile when independent cached leaf modules exist.

Added a CLI regression with two independent leaf modules:

- first run writes the full-source compiled unit plus both leaf source-unit
  artifacts;
- the root file is changed;
- the second run reports `run.cache: source-unit-prefix-hit`;
- the compiled-unit cache count does not grow, proving the run did not fall
  back to a new full-source `.yuunit`.

The next step after this is true multi-prefix surface merge. That should not be
implemented by concatenating artifacts in the CLI: `BodyLoweringPrefix` imports
one namespace/lowering/runtime surface at a time, and `ModuleTable` rebuilds
from one `CompiledRuntimeImport`. A real merge needs explicit remapping for
module IDs, namespace symbols, and runtime defs.

Wrote the implementation plan in:

- `notes/design/2026-06-26-compiled-unit-surface-merge.md`

Added `yulang cache stats` as a small observability command. It prints the user
cache root and artifact counts for:

- `control-vm`
- `poly`
- `compiled-unit`

Started the multi-prefix merge implementation with the syntax-only layer:

- `CompiledSyntaxSurface::merge_prefixes` merges prefix syntax exports by module
  path;
- duplicate synthetic parent module records are combined by unioning
  `module_loads`;
- exact duplicate exported operators are accepted;
- conflicting exported operators for the same module path/name/visibility reject
  the merge.

This is intentionally below inference. It only prepares the parser surface used
by suffix loading; namespace/runtime/typed surface remapping is still separate.

Added the next pure layer:

- `CompiledNamespaceSurface::merge_prefixes` merges modules by namespace path;
- every module/value/type symbol gets a fresh merged ID;
- root/synthetic parent module children from independent units are unioned;
- same-module value/type/module name collisions reject the merge.

This still does not build a usable multi-prefix lowering prefix. Runtime defs,
typed schemes, and lowering metadata still need matching remaps before the CLI
can use more than one prefix artifact at once.

Extended the namespace merge API with `merge_prefixes_with_remap`. It returns
the merged namespace surface plus per-prefix maps for:

- module IDs;
- value symbols;
- type symbols.

The plain `merge_prefixes` helper remains for callers that only need the
surface. The remap output is the bridge needed by typed/runtime/lowering
surface merge.

Added typed-surface merge on top of the namespace remap:

- each prefix imports its compiled type arena into one fresh merged type arena;
- value schemes are attached to the remapped namespace value symbols;
- type variables and subtract IDs are reissued per merged target, so independent
  units do not share IDs by accident;
- missing or duplicate value-symbol remaps reject the merge.

This still does not make the CLI use multiple compiled-unit prefixes. The next
surface boundary is runtime/lowering remap, then the cache layer can synthesize a
single merged compiled-unit artifact for the existing single-prefix path.

Added runtime-surface merge for independent prefix units:

- each runtime surface imports into one fresh `poly::Arena` and shared
  `DumpLabels`;
- module/value runtime defs are attached to namespace-remapped IDs;
- `merge_prefixes_with_remap` exposes per-prefix `DefId` remaps for lowering
  metadata that stores source defs;
- missing namespace remaps reject the merge;
- duplicate merged value defs reject for now.

Runtime merge now coalesces duplicate merged module defs when both sides are
`Def::Mod`: child defs are unioned and the duplicate imported module root is
removed. This covers shared synthetic parent modules such as cached `a/b` and
`a/c` leaf units. Conflicting non-module defs still reject.

Added lowering-surface merge:

- type/value symbols are remapped through `CompiledNamespaceMergeOutput`;
- source `DefId`s are remapped through `CompiledRuntimeMergeOutput`;
- stored signature payloads remain structural and are not rewritten by path
  strings during merge;
- missing namespace/runtime remaps reject the merge.

This completes the pure syntax/namespace/typed/runtime/lowering surface merge
building blocks for independent leaf units and shared synthetic parent modules.
The cache layer still needs a merged compiled-unit artifact wrapper before the
CLI can use multiple prefixes.

## Source-Unit Prefix Materializes Full Compiled Unit

Continued the compiled-unit cache normal-path work.

Previously, a `source-unit-prefix-hit` or `merged-source-unit-prefix-hit` wrote
only the hydrated `.yuir` poly artifact for the changed full source set. If the
poly/control cache was later missing for that same source set, the CLI had to
repeat the prefix import and suffix lowering.

Now the prefix-hit path returns both:

- `BuildPolyOutput`;
- a full-source `CachedCompiledUnitArtifact` built from the same prefix +
  suffix lowering result.

The CLI writes that `.yuunit` under the full source-set key before writing the
poly artifact. The next run for the same source set can therefore use
`compiled-unit-hit` even if `.yuir` is removed.

The new artifact path builds the syntax surface by merging the cached prefix
syntax surface with the freshly parsed suffix syntax surface. Namespace,
lowering, typed, and runtime surfaces are still extracted from the single
combined `BodyLowering`, so this does not add a second inference pass.

Strengthened cache canaries:

- `compatible_run_uses_single_source_unit_prefix_cache`;
- `compatible_run_uses_merged_source_unit_prefix_when_many_are_cached`;
- `scripts/source-unit-cache-smoke.sh`.

Both single-prefix and merged-prefix smokes now verify this route progression:

```text
source-unit-prefix-hit / merged-source-unit-prefix-hit
  -> compiled-unit-hit after removing poly/control cache
```

Ran a source-metrics-only performance gate:

```text
target/performance-gate/source-unit-materialize-check/summary.txt
```

The `source-unit-cache-smoke` row now reports:

```text

```

## Merged Source-Unit Prefix Artifact Cache

Added a cache key for merged compiled-unit prefix artifacts. The merged prefix
route still selects individual source-unit candidates, but after that it can
reuse an already-merged `.yuunit` instead of rebuilding the merged syntax /
namespace / typed / runtime / lowering surfaces every time.

On a cache miss, the CLI writes the merged prefix artifact under its merged
source-hash key before using it as the prefix. This keeps the route name
unchanged (`merged-source-unit-prefix-hit`) while making repeated entry edits
avoid repeated surface merge work.

Strengthened the independent leaf module CLI canary to account for the extra
merged-prefix `.yuunit`, and added a cache-level assertion that
`merged_compiled_unit_artifact_key` matches the merged artifact's manifest key.

## Prefix Runtime Provenance

Started the external-reference table preparation without changing cache

`BodyLoweringPrefix` now records runtime defs imported from a compiled-unit
prefix:

- the imported runtime `DefId` set;
- namespace-keyed module defs;
- namespace-keyed value defs.

`lower_loaded_files_with_prefix` and `lower_root_loaded_file_with_prefix` carry

## Std Compiled-Unit Prefix Cache

Added a native std prefix cache route.

For CLI runs that include a filesystem std root, the cache path now separates
the std source-unit closure from the entry/local suffix:

- first cached run can build the std prefix artifact and report
  `std-prefix-build`;
- later runs for a different entry with the same std root can report
  `std-prefix-hit`;
- the prefix route still materializes the full-source `.yuunit`, so same-source
  reruns can continue to use `compiled-unit-hit` / `poly-hit` / `control-hit`.

This avoids the earlier `source_set_contains_std` dead end where every new
entry had to infer std again unless the exact full source-set cache was already
warm.

Also added persistent native cache reads for embedded full std and compact
playground std source-text prefixes. These routes still build an in-memory
`BodyLoweringPrefix`, but they can now hydrate the compiled-unit surfaces from
the user cache across processes. On wasm the filesystem cache branch is
disabled, so build-time artifact embedding remains separate work.

Quick measurement with real `lib` std and two separate entry files:

```text
--no-cache:        run.build_poly 2.914s, run.total 2.956s
std-prefix-build: run.build_poly 3.216s, run.total 3.258s
std-prefix-hit:   run.build_poly 215.2ms, run.total 256.0ms
```

The first cached run is allowed to be slower than `--no-cache`, because it
materializes both the std prefix artifact and the full-source artifact. The
payoff is across entry/source-set changes that keep the same std root: the
next entry can import the std compiled-unit prefix instead of inferring std.

## Embedded Std Build-Time Artifacts

Added wasm build-time `.yuunit` artifacts for both embedded std profiles.

The `wasm` crate now runs a build script that uses the host `yulang` crate to
compile both the compact playground std source set and the full embedded std
source set into compiled-unit artifacts. It serializes them with the same cache
envelope used by the CLI and embeds the bytes with `include_bytes!`.

At runtime, the playground run path decodes the compact artifact first and feeds
it into the same embedded playground prefix-lowering route used by the source
fallback. Parser/text-heavy examples that miss the compact profile then use the
full embedded std artifact instead of rebuilding full std from source. If either
embedded artifact fails validation, the corresponding old source route remains
as a fallback.

## External Runtime Reference Table Preparation

Started the external-reference table path after the compiled-unit cache speedup
was pushed.

`BodyLoweringPrefixRuntime` now stores path-bearing module/value entries, not
only raw imported `DefId`s. When suffix lowering runs against a compiled prefix,
the resulting `BodyLowering` still knows which runtime defs came from the
prefix and which were created by the suffix.

Compiled-unit artifacts now serialize that provenance as
`CompiledUnitExternalRuntimeRefs`:

- `imported_def_count` records the size of the prefix-owned runtime def set;
- `modules` records namespace-keyed imported module defs with module paths;
- `values` records namespace-keyed imported value defs with value paths;
- the manifest includes `external_runtime_hash`;
- the cache format was bumped to 13 so older `.yuunit` artifacts are ignored;
- compiled-unit merge remaps external refs through the namespace/runtime remap.

The table now also serializes the full prefix-owned `DefId` set. On the runtime
side, `CompiledRuntimeSurface::import_into_with_external_defs` can map those
source defs to already-imported target defs instead of allocating fresh copies,
and skips external roots / metadata while importing the local surface.

Added the first cache-layer resolver for those refs:

- `compiled_unit_external_runtime_def_pairs` resolves serialized external
  module/value refs against an already-imported prefix runtime by stable path;
- `BodyLoweringPrefix::extend_with_compiled_unit_surfaces_and_external_defs`
  imports a dependent suffix surface on top of an existing prefix arena, so the
  dependency value def is reused instead of fresh-copied;
- `compiled_unit_complete_external_runtime_def_pairs` is the rejection gate for
  artifacts whose reachable runtime graph still needs external `DefId`s not
  covered by path-keyed external refs;
- `CompiledRuntimeSurface::import_reachable_into_with_external_defs` imports
  only the non-external runtime graph reachable from local roots and metadata,
  so private child defs inside skipped dependency bodies do not need stable
  paths;
- the canary lowers `pub y = id 1` against a cached `deps::id` prefix, records
  `deps::id` as an external ref, rebuilds the suffix artifact on top of the
  prefix, and checks that `deps::id` keeps the prefix `DefId` while `y` stays
  local.

That canary showed the previous all-external-def gate was too strong: a
dependency body may contain private child defs that the suffix graph never
imports. The gate now asks for keyability only for external defs reachable from
the local runtime graph. Dependent units still keep using dependency-closure
artifacts until broader canaries cover constructors, effect operations, role
impls, casts, and root expressions.

This does not change cache selection yet. Dependency-bearing source units are
still written as dependency-closure artifacts. The new table is the next
foundation for trimming prefix-owned defs out of a dependent unit's runtime
surface and resolving them against an already-imported dependency prefix.

## Local realm/band import route slice

Started the local realm/band implementation from the design notes.

- `UseImport` now preserves the import route (`relative`, `band::...`,
  `realm/...::...`, slash-qualified) and optional version suffix.
- Source collection discovers current-realm band roots referenced by
  `use realm/path/to/band::...` and loads `<realm root>/path/to/band.yu`.
- The collector rejects the same file being claimed by a normal `mod` path and
  a separate current-realm band owner.
- Module import lookup resolves `band::...` and current-realm band paths from
  the root namespace for this first local slice.
- Cache hashing includes import route/version metadata, and
  `CACHE_SCHEMA_VERSION` was bumped so old artifacts are not reused under the
  old import interpretation.

This is still not the full realm cache model. `realm.toml` package policy,
snapshot layout, exact per-file resolution artifacts, and true per-band module
roots remain separate slices.

## Realm Manifest Package-Premise Removal

Removed the package-manager-shaped fields from the executable realm manifest
model.

- `SourceRealmManifest` now carries only the local realm identity.
- `SourceRealmDependency` was removed.
- Human-written `realm.version` is rejected. Freeze versions live in
  `.yulang/versions/<version>/snapshot.json`, not in `realm.toml`.
- Human-written `[dependencies]` is rejected. Dependency requests are source
  imports plus the future per-file resolution cache / lock projection.
- Frozen snapshots still record the selected version in `snapshot.json`, but
  the copied `realm.toml` no longer injects a version.
- Frozen snapshot source hashes now use a source-hash salt and do not include
  the version string as source content.

## Source-Site Resolution Metadata

Added the first executable slice of the per-file resolution-cache path.

- `UseImport` now preserves `with` anchors as structured `UseAnchor` metadata.
- `CollectedSource` carries `resolution_imports`, currently the source-site
  imports whose route/version/anchor can affect realm resolution.
- Source collection extracts module loads and resolution metadata from one
  parsed source header instead of separately reparsing for `mod` and
  current-realm band discovery.
- Current-realm band loading still uses the local `<realm root>/<band>.yu`
  rule, but the collected source now also keeps the original request metadata
  for the future exact resolution cache / lock projection.
- Source and source-unit cache keys include the structured resolution imports,
  and `CACHE_SCHEMA_VERSION` was bumped to `3`.

## Persistent Realm Resolution Cache API

Added the first persistent cache envelope for exact realm resolution entries.

- Resolution artifacts are stored under
  `realms/<source-realm-cache-component>/resolution/<request-key>.yures`.
- The source realm controls the cache namespace, but artifacts still live in
  the persistent user cache rather than mutating the source realm tree.
- A resolution entry stores the source realm, original structured request, and
  resolved target realm / band / module / source fingerprint.
- `yulang cache stats` now reports `realm-resolution` entries.

This is only the storage API. Source collection still resolves local realm
bands directly, and the next slice should write/read these entries while
validating the target source fingerprint.

Cached CLI builds now populate the current-realm part of this cache after source
collection:

- `realm/...::...` imports are mapped to the already loaded band-root source
  file;
- the stored target records the local realm identity, band path, module path,
  source path, source length, and source hash;
- identical cached entries are left untouched, while changed fingerprints are
  overwritten.

The cache is still not used to decide which source files to load. That requires
an explicit cache policy in source collection plus stale-entry fallback for
future global providers.

Added the explicit source collection cache policy for this local slice.

- `SourceCollectionCache` wraps the persistent `ArtifactCache` and is only
  passed by CLI routes when `--no-cache` is not set.
- Current-realm resolution entries are used as checked lookups, not authority:
  the cached target must still point at the deterministic local
  `<realm>/<band>.yu` candidate and match the target source fingerprint.
- If the entry is stale, corrupted, or redirected to another path, collection
  falls back to the existing filesystem resolver.

This keeps cache state from changing local source selection while laying the API
shape needed for future slash-qualified/global realm providers.

## Realm Imports In Source-Unit Planning

Source-unit dependency planning now treats route-aware resolution imports as
semantic dependencies.

- `realm/...::...` and `band::...` imports look for the longest already-loaded
  module prefix and add a source-unit dependency edge.
- Ordinary relative imports still do not create a new edge here; existing `mod`
  containment remains the local module-tree edge.
- This prevents a cached source-unit from being selected without the imported
  band/module prefix that its source-level request needs.

This is deliberately conservative. Slash-qualified cross-realm imports that
have not been loaded yet do not create a dependency edge; the future resolver
must load or reject them before source-unit cache selection.

## Exact Mono Cache Scope

Reviewed the realm-owned mono cache idea against the current cache pipeline.

The safe near-term slice is an exact source/resolution-keyed mono artifact:

- the source key already includes structured realm resolution request metadata;
- the mono artifact can sit between poly and control artifacts;
- it helps `dump-mono`, `run-mono`, and control-cache misses without sharing
  specialization results between different programs.

Realm-level mono sharing is still deferred. It needs a typed/runtime surface key
and a specialization demand key before mono can be indexed by realm/band without
accidentally reusing the wrong specialization set.

Implemented the exact mono cache slice:

- `ArtifactCache` now reads/writes `.yumo` mono artifacts under
  `artifacts/mono`;
- `dump-mono`, `run-mono`, and the cached control build path share the same
  mono specialization helper;
- cache stats now include the `mono` stage.

## Release `.yucu` Artifact Boundary

Recorded the future release/freeze cache boundary:

- fixed realm/band releases should materialize `.yucu` artifacts for published
  band roots;
- `.yucu` is a dependency-prefix cache artifact, not the release source of
  truth;
- source snapshots plus resolution metadata remain authoritative, and stale or
  missing `.yucu` entries can be rebuilt from them;
- `.yuir`, `.yumo`, and `.yuvm` stay exact-program caches and should not be
  included in default release snapshots.

## Current-Realm Band Visibility Slice

Started tightening the implemented module system around current-realm bands.

- `CollectedSource` and `sources::LoadedFile` now preserve `band_path`.
- `ModuleTable` stores a band path per module.
- Local `realm/...::...` imports can materialize a synthetic band-root module
  without requiring `mod <band>;` in the current band.
- Ordinary bare paths do not cross into those synthetic current-realm band
  modules.
- Cross-band imports require `pub`; `our` remains same-band only.
- Compiled namespace artifacts serialize and hash module `band_path`, so `.yucu`
  prefix imports do not erase this boundary.

`std::...` remains on the existing route for now. It needs a structured
prebound alias route before std can be isolated as its own band without
breaking implicit prelude imports.

## Source-Unit Band Path Cache Fix

Tightened the source-unit `.yucu` creation path after the current-realm band
slice:

- standalone and closure source-unit artifact builders now lower through
  `LoadedFile`s that preserve per-file `band_path`;
- synthetic parent modules keep the band path needed by current-band imports
  below a current-realm band root;
- synthetic roots expose current-realm band roots as `pub mod <band>;` only
  when the child is a band root, while ordinary synthetic module parents still
  require a real module-load visibility;
- cache regression tests now cover current-realm band paths in closure
  compiled-unit artifacts.

Also tightened current-realm resolution artifact creation so a cached target is
written only when the target file is both the requested band root module and in
that same band. This keeps manually assembled source sets from producing
resolution metadata that the collector path would never produce.

Cache keys now include `band_path` in the shared collected-source hash and in
the source-file resolution context. This keeps full-source, source-unit, and
realm-resolution cache keys aligned with the `.yucu` manifest file hash, which
already included `band_path`.

## Local Realm Boundary Spec

Recorded the release target for local realm boundaries:

- every source is inside a realm;
- a nearest `realm.toml` ancestor marks an editable multi-band local realm;
- a file outside such a manifest is a singleton implicit realm, and the file
  itself is the only band root (`realm = band`);
- same-directory band discovery through `realm/...::...` should require an
  explicit local realm boundary;
- the current collector still falls back to the entry file's parent directory,
  so tightening that behavior remains a local realm/band release blocker.
