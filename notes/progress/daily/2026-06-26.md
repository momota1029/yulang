# 2026-06-26

## CLI source-unit prefix cache widening

Continued the source-unit compiled cache integration from 2026-06-25.

The CLI still uses a single compiled source-unit prefix. It does not merge
multiple `.yuunit` surfaces yet, because `BodyLoweringPrefix` currently imports
one namespace/lowering/runtime surface at a time. Instead, when multiple cached
dependency-free source units are available, the CLI now chooses one unit as the
prefix and lowers every other file from source.

This keeps the implementation inside the existing single-prefix invariant while
avoiding a full compile when independent cached leaf modules exist.

Added a CLI regression with two independent leaf modules:

- first run writes the full-source compiled unit plus both leaf source-unit
  artifacts;
- the root file is changed;
- the second run reports `run.cache: source-unit-prefix-hit`;
- the compiled-unit cache count does not grow, proving the run did not fall
  back to a new full-source `.yuunit`.

The next step after this is true multi-prefix surface merge. That should not be
implemented by concatenating artifacts in the CLI: `BodyLoweringPrefix` imports
one namespace/lowering/runtime surface at a time, and `ModuleTable` rebuilds
from one `CompiledRuntimeImport`. A real merge needs explicit remapping for
module IDs, namespace symbols, and runtime defs.

Wrote the implementation plan in:

- `notes/design/2026-06-26-compiled-unit-surface-merge.md`

Added `yulang cache stats` as a small observability command. It prints the user
cache root and artifact counts for:

- `control-vm`
- `poly`
- `compiled-unit`

Started the multi-prefix merge implementation with the syntax-only layer:

- `CompiledSyntaxSurface::merge_prefixes` merges prefix syntax exports by module
  path;
- duplicate synthetic parent module records are combined by unioning
  `module_loads`;
- exact duplicate exported operators are accepted;
- conflicting exported operators for the same module path/name/visibility reject
  the merge.

This is intentionally below inference. It only prepares the parser surface used
by suffix loading; namespace/runtime/typed surface remapping is still separate.

Added the next pure layer:

- `CompiledNamespaceSurface::merge_prefixes` merges modules by namespace path;
- every module/value/type symbol gets a fresh merged ID;
- root/synthetic parent module children from independent units are unioned;
- same-module value/type/module name collisions reject the merge.

This still does not build a usable multi-prefix lowering prefix. Runtime defs,
typed schemes, and lowering metadata still need matching remaps before the CLI
can use more than one prefix artifact at once.

Extended the namespace merge API with `merge_prefixes_with_remap`. It returns
the merged namespace surface plus per-prefix maps for:

- module IDs;
- value symbols;
- type symbols.

The plain `merge_prefixes` helper remains for callers that only need the
surface. The remap output is the bridge needed by typed/runtime/lowering
surface merge.

Added typed-surface merge on top of the namespace remap:

- each prefix imports its compiled type arena into one fresh merged type arena;
- value schemes are attached to the remapped namespace value symbols;
- type variables and subtract IDs are reissued per merged target, so independent
  units do not share IDs by accident;
- missing or duplicate value-symbol remaps reject the merge.

This still does not make the CLI use multiple compiled-unit prefixes. The next
surface boundary is runtime/lowering remap, then the cache layer can synthesize a
single merged compiled-unit artifact for the existing single-prefix path.

Added runtime-surface merge for independent prefix units:

- each runtime surface imports into one fresh `poly::Arena` and shared
  `DumpLabels`;
- module/value runtime defs are attached to namespace-remapped IDs;
- missing namespace remaps reject the merge;
- duplicate merged module/value defs reject for now.

The duplicate rejection is intentional. Shared synthetic parent modules need a
real module-def coalescing rule, so runtime merge currently accepts the
independent leaf case and leaves shared-parent coverage for a later slice.
